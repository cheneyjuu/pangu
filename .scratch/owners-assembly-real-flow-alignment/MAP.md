# 对齐江湾真实业主大会办理流程

Status: open
Label: wayfinder:map

## Destination

形成一份可直接进入实施规划的业主大会业务规格：以江湾国际公寓真实纸质表决流程为当前基线，明确会前筹备、业委会召开决定、方案公示、意见处理、正式会议通知、选票用印与逐户送达、回收、开箱计票、结果公告，以及未来可信线上表决的启用门槛。

## Notes

- 领域：上海住宅小区业主大会，重点是全小区公共事项；楼栋或单元维修接龙属于另一条流程。
- 事实优先级：本小区有效备案议事规则和真实办理材料 > 业委会、居委会等实际办理人员确认 > 上海现行规则和示范文本 > 当前代码行为。
- 现有六篇公众号材料已形成“方案公示 -> 正式会议/投票公告 -> 盖章选票送达与回收 -> 开箱计票 -> 结果公告”的历史证据链。
- 当前系统已经支持一次会议包含多个事项、归档方案附件与纸质选票模板、逐户送达和纸票录入、人数与面积双维度计票；这些能力可以保留。
- 当前系统只能视为简化的纸质表决录入流程，不能视为已对齐：缺少业委会召开决定，合并了方案公示与正式会议通知，允许在公示期内送票，缺少逐事项方案版本、完整纸票台账、开箱计票和结果公告。
- 管理端现有“线下会议”只是存储值，后续仍走纸质表决流程；在完整现场集体讨论流程确定前，不应把它宣传为可用会议形式。
- C 端现有真实附件、正式会议通知、纸票送达回收、结果公告和可信历史记录均未闭环；硬编码附件、房屋、阈值、司法链摘要或历史结果不能作为目标方案的一部分。
- 新办理入口当前固定为纸质表决是合理的风险控制；旧技术接口仍能创建线上或混合表决，需在后续决策中统一边界。
- 本地图使用 local-markdown tracker；本轮只制图和建立决策票，不实施代码、不迁移数据、不部署。
- 每个后续会话应先查阅 `docs/报修与业主大会表决流程重构方案.md`，并在涉及术语和聚合边界时使用 `domain-modeling`。

## Decisions so far

- 议事规则采用“原件约束的治理配置”：规则原件必须上传，结构化配置由人工填写或系统辅助解析，主任/副主任逐项核对后启用；自动解析不是必经业务步骤，也不能直接使规则生效。
- 规则启用版本不可修改。业委会形成召开决定并进入正式办理、开始方案公示前，系统生成不可变会议规则快照；无有效规则版本时只允许内部筹备，不得进入正式公示、通知、投票和计票。
- 上述跨端约束已固化为 `docs/adr/0007-governed-owners-assembly-rule-registry.md`。

## Not yet specified

- 仍须取得江湾国际公寓实际备案且当前有效的《业主大会议事规则》原件，逐项确认其具体选择和生效期间；产品架构决策不能代替该外部业务事实。
- 待关键业务节点确定后，再拆解在办会议与历史表决数据的迁移、冻结或只读兼容策略。
- 待角色、材料和状态机确定后，再形成跨 Pangu、Yaochi、Shennong 的端到端验收矩阵和审计导出要求。
- 待纸质票生命周期明确后，再确定批量打印、扫码录入、异常补录和大规模房屋台账的具体交互。
- 待结果公告边界明确后，再确定业主异议、勘误、撤销和重新表决的后续流程。

## Out of scope

- 楼栋或单元维修的微信接龙、审价、用印和楼栋验收；它属于维修工程治理流程，不并入本次业主大会地图。
- 在本轮 Wayfinder 制图中编写或部署业务代码。
- 用平台默认天数、默认未表态处理或历史公众号时间替代江湾当时有效的备案议事规则。
- 在可信身份、房屋、表决权、规则快照和纸电去重闭环前直接开放线上表决。
