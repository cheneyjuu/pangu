在多租户自治 SaaS 平台中，权限控制（RBAC + ABAC 数据隔离）是保障底层法权数据安全的核心屏障。由于社区设置涉及行政归属、法定计票分母（总专有面积）等可能引发民事诉讼的关键元数据，必须实施极其严苛的非对称权限划分。

基于我们已闭环优化的 `t_tenant_community`、`t_governance_policy` 库表设计及前端四 Tab 页面框架，以下将为您深度拆解 **G端（监管层）**、**B端（自治层、物业层）** 各细分角色在社区设置页面中的可见性（查看）与可操作性（编辑/提交）权限矩阵。

---

### 一、 核心角色定义与系统上下文

在系统权限底座中，登录账户（`account_id`）在进入特定租户小区（`tenant_id`）后，会被激活对应的局部角色标识（`role_code`）：

1. **G端（政府监管层）**：`GOV_SUPER_ADMIN`（街道办事处管理员）、`COMMUNITY_ADMIN`（居委会监管员）
2. **B端-自治组织层（业主自治）**：`COMMITTEE_DIRECTOR`（业委会主任）、`COMMITTEE_MEMBER`（业委会成员/秘书）
3. **B端-物业服务层（受托执行）**：`PROPERTY_MANAGER`（物业经理）、`PROPERTY_STAFF`（物业员工）

---

### 三、 B端内部细分角色的权限差异流（防呆与制衡设计）

同属于 B 端，业委会（物权所有者代表）与物业公司（受托企业）之间的利益经常存在冲突（例如物业希望虚增空置房面积、隐瞒共有资金）。因此前端页面必须做到严格的元素级隔离：

#### 1. 业委会主任（`COMMITTEE_DIRECTOR`） vs 业委会成员（`COMMITTEE_MEMBER`）

* **业务场景**：业委会主任是自治法人代表，拥有最终决策提交权；成员与秘书主要负责日常浏览和数据准备。
* **页面元素差异**：
* 在 **Tab 2（建筑名册）** 中，主任拥有物理台账（户数、车位数）的【保存变更】编辑权；成员登录时，整个卡片表单置灰只读。
* 在 **Tab 3（法定基数）** 中，主任右侧高亮展现大蓝按钮【申请重新对账】；成员登录时，该按钮直接隐藏（**N**），防止非负责人随意触发现场数据对账导致版本号混乱。



#### 2. 物业经理（`PROPERTY_MANAGER`） vs 物业成员（`PROPERTY_STAFF`）

* **业务场景**：物业经理作为 B 端辅助管理者，需要维护楼栋数、交付户数等工程物理台账，但绝不能允许其触及财务公示限制及核心计票规则。
* **页面元素差异**：
* **Tab 4（自治与财务规则）对物业侧全面降级**：物业经理和物业成员登录系统，进入社区设置页面时，**Tab 4 直接隐藏或不可点击**。物业既不需要看到“每季首月15日逾期锁号”的催办 Banner，也无权查看或修改任何公共收益共管账户配置。
* **物业成员最低权限**：物业员工（如普通客服、维修登记员）登录后，Tab 1 隐藏，Tab 2 只读，Tab 3 仅能查看表格。整个页面没有任何可点击的【保存】或【提交】按钮，杜绝了一线员工手误修改小区基数的可能。

### 四、 G端（街道/居委会）的“超级裁判”独占元素描述

当 G 端角色（如居委会干事）登录并切换到该小区时，页面将展现以下 **G 端独占（Exclusive）** 的特权操作元素：

1. **Tab 1 的【强行解锁治理锁】功能（独占 Banner 按钮）**：
* 如果业委会由于财务未公开被系统状态机卡死（`FINANCIAL_LOCKED`），业委会主任看到的只有一行绝望的红字警告。
* 而 G 端用户登录时，红字警告右侧会展现一个大红按钮：**【人工复核通过-强行解锁】**。点击后弹窗要求上传线下整改证明，提交后执行数据库事务更新 `governance_lock_status = 'UNLOCKED'`。


2. **Tab 3 的【审批对账申请】红点气泡与控制面板**：
* 当业委会主任点击了“申请重新对账”后，G 端用户的 Tab 3 顶部会高亮闪烁一个任务气泡：*“收到1条来自业委会的基数变更复核申请”*。
* G 端在此处可以看到 diff 对比（例如：申请将 V1 版的 $124,582.45\,\text{m}^2$ 修正为 $124,600.00\,\text{m}^2$）。G 端点击【同意并发布新版本】后，系统才会真正执行 `statistics_version` 自增，确立新的表决分母。

### 五、 前端开发（React / Taro）控制伪代码实现建议

为了让前端开发人员完美实现上述权限拆解，建议在社区设置页面的根组件采用 **角色权限高阶守卫（Role-Based Conditional Rendering）**。以下提供一段清晰的架构伪代码示例：

```javascript
// 社区设置根页面组件
function CommunitySettingsPage({ userRole }) {
  
  // 1. 定义控制变量
  const isGov = ['GOV_SUPER_ADMIN', 'COMMUNITY_ADMIN'].includes(userRole);
  const isDirector = userRole === 'COMMITTEE_DIRECTOR';
  const isProperty = ['PROPERTY_MANAGER', 'PROPERTY_STAFF'].includes(userRole);

  return (
    <div className="settings-container">
      <Tabs>
        {/* Tab 1 行政备案：物业完全不可见，业委会只读，G端可写 */}
        {!isProperty && (
          <TabPanel tab="行政区划与组织备案">
            <RegionForm disabled={!isGov} />
            <EstablishmentSwitches disabled={!isGov} />
            {isGov && <ForceUnlockBanner />}
          </TabPanel>
        )}

        {/* Tab 2 建筑名册：全员可见，但控制开发商隐私和保存按钮 */}
        <TabPanel tab="物业区域与建筑名册">
          <DeveloperCard maskAccount={!isGov} disabled={!isGov} />
          <PhysicalAssetForm disabled={!(isGov || isDirector || userRole === 'PROPERTY_MANAGER')} />
        </TabPanel>

        {/* Tab 3 计票基数：对账按钮仅属于G端直办或主任申请 */}
        <TabPanel tab="法定计票基数底座">
          <BaseMetricsViewer />
          {isGov && <Button>重新对账校对基数(直接生效)</Button>}
          {isDirector && <Button>提交重新对账申请(生成审批流)</Button>}
        </TabPanel>

        {/* Tab 4 财务规则：物业公司由于利益冲突，强行全隐 */}
        {!isProperty && (
          <TabPanel tab="自治与财务规则">
            <PolicySelect disabled={!isGov && !isDirector} />
            <FinancialWarningBanner visible={isDirector || userRole === 'COMMITTEE_MEMBER'} />
          </TabPanel>
        )}
      </Tabs>
      
      {/* 底部浮动通栏保存大条：普通物业员工和普通业委会成员直接不渲染 */}
      {['GOV_SUPER_ADMIN', 'COMMUNITY_ADMIN', 'COMMITTEE_DIRECTOR', 'PROPERTY_MANAGER'].includes(userRole) && (
        <StickyFooterButton role={isGov ? 'GOV' : 'BUSINESS'} />
      )}
    </div>
  );
}

```

通过这套详尽的权限拆解方案，UI 设计师可以开始为不同角色定制高亮、置灰或隐藏的视觉切片，研发团队也能精准锁死 API 接口层面的 `@PreAuthorize` 注解权限，确保系统完全满足最高级别的基层自治数字合规要求。
