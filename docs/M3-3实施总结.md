## 1. 目标 & 边界回顾

M2-2 期间 ELECTION 差额选举的引擎（`ElectionVotingEngine`：双过半 + 党员席位下限 + 多阶排序 + 抽签）、领域模型（`ElectionSubject` / `Candidate` / `ElectionVotingResult`）、`t_election_candidate` 表、`t_voting_subject.max_winners`、`t_vote_item.target_id` **全部已落地**；M3-2 又接通了 GENERAL/MAJOR 的"立项-公示-投票-结算"两端管道。

但 ELECTION 这一类型的**业务管道全程是断的**：

1. `DefaultVotingEngineRouter.settle` 对 ELECTION 抛 `UnsupportedSubjectTypeException`，引擎永远跑不到；
2. `ProposalLifecycleService.propose` 对 ELECTION 直接 reject（M3-2 占位），无法立项、无法写 `max_winners`；
3. `VoteSubmissionService.cast` 对 ELECTION 直接 reject，业主无法对候选人投票；
4. `t_election_candidate` 只有 `countActivePool` 一个只读查询，没有任何提名/审查写入；
5. `VotingSubjectRepositoryImpl.toAggregate` 丢弃 `max_winners`、不构造 `ElectionSubject`。

M3-3 把上述五处 plumbing 补齐。

| 维度 | 本期落地 |
|---|---|
| ✅ 管理端提名候选人 | 复用 `candidate:nominate`（GB 端），状态 `PENDING_REVIEW` |
| ✅ 街道办/居委会资格审查 | 复用 `candidate:approve`（G 端），`PENDING_REVIEW → APPROVED/REJECTED` |
| ✅ 选举立项 | 复用 `voting:subject:create`，放开 ELECTION 并要求 `maxWinners ≥ 1` |
| ✅ 业主选举投票 | 限 `maxWinners` 票，对 APPROVED 候选人投 SUPPORT |
| ✅ 结算路由进 `ElectionVotingEngine` | `toAggregate` 还原 `ElectionSubject` + router attach APPROVED 候选人 |
| ❌ 党组书记前置审查状态机 / HANDOVER_LOCK 换届熔断 | 推迟 M4 |
| ❌ ELECTION 投票强制 L3 | 默认与 GENERAL 同档不强制（选举非"重大事项电子大表决"），待确认 |
| ❌ MQ 定向推送 / 投票期实时票数曝光 | 沿用 M3-2，仅"已投/未投" |

### 已锁定的设计决策

1. **候选人申报来源 = 管理端提名**：由居委会/业委会筹备组录入名单（name + uid + 党员标记），C_USER 自检接口 `ElectionService.checkCandidateQualification` 保持独立不动。
2. **选举投票约束 = 限 maxWinners 票**：同一 opid 可对多个候选人各投 1 票 SUPPORT，总数不超过 `max_winners`；`UNIQUE(subject_id, opid, COALESCE(target_id,0))` 兜底"同一候选人不可重复投"。
3. **结算编排链路完全不改**：类型差异下沉到 router —— `toAggregate` 对 `subject_type=ELECTION` 构造带 `maxWinners` 的 `ElectionSubject`，router settle 时 attach `qualification_status=APPROVED` 候选人再调引擎。`VotingApplicationService.settle` 零改动。

## 2. 落地清单

### 2.1 V3.1 schema（`db/migration/V3.1__election_workflow.sql`）

```sql
-- 1. 候选人查询索引（settle 加载 APPROVED / 管理端列表 / 资格审查）
CREATE INDEX idx_election_candidate_subject_status
    ON t_election_candidate(subject_id, qualification_status);

-- 2. trigger 13：ELECTION（subject_type=1）必须携带 max_winners >= 1
--    非 ELECTION（GENERAL=3 / MAJOR=2）不约束，留 NULL（与 M3-2 insert 行为一致）
CREATE OR REPLACE FUNCTION fn_election_max_winners_check() RETURNS TRIGGER AS $$
BEGIN
    IF NEW.subject_type = 1 AND (NEW.max_winners IS NULL OR NEW.max_winners < 1) THEN
        RAISE EXCEPTION '[trigger 13] ELECTION 议题 max_winners 必须 >= 1，...';
    END IF;
    RETURN NEW;
END;
$$ LANGUAGE plpgsql;
CREATE TRIGGER trg_election_max_winners
    BEFORE INSERT OR UPDATE ON t_voting_subject
    FOR EACH ROW EXECUTE FUNCTION fn_election_max_winners_check();
```

> **权限无需新增**：`candidate:nominate`（V1.4，GB，授 role 3/4/5/6/8）/ `candidate:approve`（V1.4，G，授 role 1/2/3）/ `voting:subject:create`（M3-2 扩到 GBS）已齐备。
> **trigger 编号**：V1..V2.x 用 1-11，M3-2 用 12，M3-3 起占用 **13**。

### 2.2 Domain 层（`pangu-domain/.../model/voting/`，纯 Java 无 Spring）

- **`Candidate`** —— 加 `uid` + `qualificationStatus` 两字段（原仅 `candidateId/name/partyMember`），审查与归属校验需要。
- **新增 `CandidateStatus`** —— `PENDING_REVIEW(1)/APPROVED(2)/REJECTED(3)/WITHDRAWN(4)`，仿 `SubjectStatus` 的 `fromDbValue/getDbValue`。
- **新增 `ElectionCandidateActions`** —— 候选人状态机（仿 `VotingSubjectActions`）：`PENDING_REVIEW → APPROVED/REJECTED` 合法，终态再迁移抛内部静态 `IllegalCandidateTransitionException`。
- **`VotingSubject`** —— 加可空 `maxWinners` 透传字段（避免 propose 链路对 `ElectionSubject` 强转分叉；`toRow` 直接取值）。
- **`ElectionCandidateRegistry`** port 扩 6 方法：`findApprovedCandidates` / `nominate`（内部静态 `DuplicateCandidateException`）/ `updateQualification` / `findById` / `findBySubject` / `countSupportByOpid`。`countActivePool` 不动。

> `ElectionSubject` / `ElectionVotingEngine` / `ElectionVotingResult` 引擎与结果模型零改动。

### 2.3 Application 层（`pangu-application/.../voting/`）

- **新增 `ElectionCandidateService`**（`@Transactional`）：
  - `nominate` —— 校验议题存在且为 ELECTION 且 status ∈ {DRAFT, PUBLISHED}（否则 `SUBJECT_NOT_NOMINATABLE`）；`registry.nominate` 捕获 `DuplicateCandidateException` → `CANDIDATE_ALREADY_NOMINATED`。
  - `review` —— 候选人不存在 → `CANDIDATE_NOT_FOUND`；`ElectionCandidateActions` 迁移；`updateQualification` 返回 0 → `CANDIDATE_REVIEW_CONFLICT`。
  - `listCandidates` —— 复用 `registry.findBySubject`。
- **`ProposalLifecycleService.propose`** —— 删 ELECTION reject 分支；ELECTION 时 `maxWinners` 非空且 ≥1（否则 `ELECTION_MAX_WINNERS_REQUIRED`），透传落库。
- **`VoteSubmissionService.cast`** —— 删 ELECTION reject 分支；ELECTION 闸门：缺 `targetId` → `ELECTION_TARGET_REQUIRED`；候选人不存在/不属本议题/非 APPROVED/非 SUPPORT → `CANDIDATE_NOT_VOTABLE`；`countSupportByOpid >= maxWinners` → `VOTE_LIMIT_EXCEEDED`；opid 归属/scope/UNIQUE 防重沿用现有链路。
- **`command/`** 新增 `NominateCandidateCommand` / `ReviewCandidateCommand`；`ProposeSubjectCommand` 加 `Integer maxWinners`。
- **`VotingApplicationException.Reason`** 加 8 个：`ELECTION_MAX_WINNERS_REQUIRED` / `ELECTION_TARGET_REQUIRED` / `CANDIDATE_NOT_FOUND` / `CANDIDATE_NOT_VOTABLE` / `CANDIDATE_ALREADY_NOMINATED` / `CANDIDATE_REVIEW_CONFLICT` / `VOTE_LIMIT_EXCEEDED` / `SUBJECT_NOT_NOMINATABLE`。

### 2.4 Infrastructure 层（结算路由是关键）

- **`VotingSubjectRepositoryImpl`**：`toAggregate` 对 ELECTION 构造 `ElectionSubject`（`@SuperBuilder` 拷 base 字段 + `maxWinners`，candidates 留空）；落库经 `VotingSubject.maxWinners` 透传字段写 `max_winners`。
- **新增 `ElectionCandidateMapper`(.java/.xml)** + **`ElectionCandidateRow`**：`insertCandidate` / `updateQualification`（`WHERE candidate_id=? AND qualification_status=1`）/ `selectById` / `selectBySubject` / `selectApprovedBySubject` / `countSupportByOpid`。
- **`ElectionCandidateRegistryImpl`** 实现新端口，`nominate` catch `DuplicateKeyException` → `DuplicateCandidateException`（仿 `VoteItemRepositoryImpl`）。
- **`DefaultVotingEngineRouter`** —— 注入 `ElectionVotingEngine` + `ElectionCandidateRegistry`；ELECTION 分支强转 `ElectionSubject`、`setCandidates(findApprovedCandidates(...))` 后调 `settle`。

## 5. 已识别的剩余风险

1. **ELECTION 投票是否要求 L3**：默认与 GENERAL 同档不强制 L3。若业务要求选举走 L3，cast ELECTION 分支需补 `evaluateVoting`，本期未加，待确认。
2. **maxWinners 计数门并发**：`countSupportByOpid` 与 insert 间存在 TOCTOU 窗口。业主投票低频，沿用"读后写"软约束 + `UNIQUE` 兜底重复投；严格场景需 DB 侧约束，推迟评估。
3. **`ElectionSubject` 字段拷贝**：`@SuperBuilder` 拷 base 字段需逐字段，遗漏即丢数据。已在 `VotingSubject` 加可空 `maxWinners` 透传字段以避免 propose 链路强转分叉。
4. **trigger 13 与种子数据**：仅 INSERT/UPDATE 触发，不影响历史行；既有 ELECTION 测试种子（`ElectionWorkflowIntegrationTest`、矩阵测试）均已显式写 `max_winners`。
5. **候选人 uid 不加 FK**：跨 tenant 删 user 不级联，仅审计 join，沿用 M3-2 `proposed_by_user_id` 处理。
