# 维修与表决能力改造计划

> 业务关联：把当前分裂的维修表决、业主大会纸质投票和普通线上投票能力改造成符合统一业务规范的端到端流程。
>
> 状态：实施中，前三个切片及纸质投票后端闭环已完成
>
> 业务依据：[维修与业主大会表决业务规范](../business/维修与业主大会表决业务规范.md)
>
> 最近更新：2026-07-19

## 1. 当前问题

### 1.1 表决能力分裂

- 业主大会领域模型和表结构已经出现纸质、线上、混合三种方式，但正式办理服务只允许纸质；
- 维修事项表决又只允许线上；
- 普通线上投票与业主大会表决包不是同一条证据链；
- 管理端显示“线上实名投票”，与正式业主大会当前纸质能力矛盾；
- 小程序识别业主大会事项后只展示纸质办理说明，不能在线投票。

### 1.2 议事规则不能真正驱动流程

- 当前维修事项规则把“送达方式”保存为自由文本；
- 上传后直接设为当前版本，缺少草稿、逐项核对和确认启用；
- 送达文字进入快照但不校验逐户送达记录；
- 表决渠道被社区默认值或服务端硬编码覆盖，没有从有效规则和本次表决安排生成。

### 1.3 维修流程仍混入实现概念

- “参考估算”作为小区规则开关，但只约束旧工单，不约束当前工程项目；
- “定商受阻”“当前授权状态”“待后端核验”等技术状态暴露给物业；
- 责任初判、资金、决定、施工单位选择和最终方案锁定的业务顺序仍有交叉。

## 2. 改造原则

1. 先修领域和后端真相，再改页面文案；
2. 纸质、线上和混合投票共用一套表决包、名册、送达、有效票和计票模型；
3. 业主大会、部分共有维修等上层流程可以不同，但不复制投票执行引擎；
4. 所有渠道按“表决事项 + 专有部分”保证只有一张有效票；
5. 当前系统未完整支持的规则在正式办理前阻断，不做前端假流程；
6. 不修改已经执行的 Flyway 迁移，只增加后续迁移；
7. 不根据样本补造江湾议事规则、维修资金授权、账户余额或验收角色。

## 3. 实施切片

### 切片一：文档和领域语言收口

状态：已完成（2026-07-19）。

- 建立唯一维修与表决业务规范；
- 精简 `CONTEXT.md`，只保留领域词汇；
- 将旧 PRD、路线图和实施总结移入归档；
- 修正议事规则 ADR，废止无依据的付款留存和物业模式表决默认规则。

验收：活跃文档不再互相引用冲突结论，历史材料明确标记为非业务依据。

### 切片二：议事规则版本治理

状态：已完成并复核（2026-07-19）。

- 规则版本支持 `DRAFT -> PENDING_CONFIRMATION -> ACTIVE -> SUPERSEDED`；
- 原件、结构化配置、页码/条款、逐项确认和审计记录同时保存；
- 配置表决允许方式、有效送达方式、未反馈处理、纸质用印、线上身份要求及重复票处理；
- 当前有权确认人从真实任期和授权读取，不能由前端传角色；
- 每次正式表决生成不可变规则快照。

验收：上传原件不能直接启用；缺少确认或系统不支持的配置不能进入正式办理。

实施记录：规则状态机和会次快照由 `V3.99`、`V3.100` 及对应服务承载；切片级需求、设计和验证记录见 [业主大会议事规则版本治理 SPEC](../specs/owners-assembly-rule-governance/requirements.md)。聚焦测试 11 项、全量测试 653 项通过。

### 切片三：统一表决执行内核

状态：已完成（2026-07-19）。

抽取可供业主大会和维修决定复用的表决内核，统一管理：

- 表决包及方案版本；
- 冻结表决人名册和计票基数；
- 逐户送达记录；
- 纸质和线上有效票；
- 跨渠道唯一约束；
- 计票、结果和审计事件。

上层流程只负责确定谁有权发起、何时冻结以及结果如何推进项目，不自行保存另一套票。

验收：同一房屋的纸质票与线上票不能同时有效，所有结果可追溯到同一表决包。

实施记录：`V3.110` 建立通用表决包、冻结名册、送达、有效票和审计台账；业主大会纸质送达与收票已改走统一内核，通用线上入口遇到正式表决事项时也必须经过冻结名册、渠道、送达和跨渠道唯一性校验。切片级需求、设计和验证记录见 [统一表决执行内核 SPEC](../specs/unified-voting-execution-core/requirements.md)。聚焦测试 35 项、全量测试 661 项通过，1 项跳过。

### 切片四：纸质投票闭环

状态：后端能力完成；管理端与业主端完整页面纳入切片八（2026-07-19）。

- 选择并锁定已经按生效规则完成用印的纸质表决票模板；
- 逐户记录送达、签收或其他有效送达证据；
- 回收票面原件，由一人录入、另一人对照原件复核；
- 记录无效票、重复票和原因；
- 业主端只展示本人送达、参与状态和公开材料，不披露其他业主选择。

验收：从表决包发布到纸票计票、结果公告和归档可以完整办理。

实施记录：`V3.111` 建立纸质送达、回收纸票、不可变录入版本、复核和逐事项处理结果台账；送达核对通过后才进入统一送达，纸票由不同人员复核确认后才进入统一有效票。无效票、退回修订、重复材料和错误登记作废均保留历史，旧 `/paper-votes` 即时计票入口已删除。切片级需求、设计和验证记录见 [纸质投票闭环 SPEC](../specs/paper-voting-closed-loop/requirements.md)。结果公开与三端完整交互仍由后续切片完成。

### 切片五：线上实名投票闭环

- 业主端按当前账号、业主资格、专有部分和表决权代表核验资格；
- 投票内容绑定表决包、事项、方案版本和投票时间；
- 服务端在提交前检查该专有部分是否已有有效票；
- 为按有效规则提出纸质办理要求的个别业主保留纸质例外路径，并与线上票统一去重；
- 保存签名或等效可信确认、提交时间和审计记录；
- 截止后不可修改历史票。

验收：有资格业主可以在线完成投票，无资格账号、重复房屋、过期或旧版本请求均被后端拒绝。

### 切片六：纸质与线上并行

- 本次表决方式为混合时，同时开放纸质和线上采集；
- 两个渠道共享同一名册、时间窗口和计票规则；
- 纸票录入前检查线上有效票，线上提交前检查纸质有效票；
- 数据库唯一约束处理并发竞争；
- 重复收到的材料只形成冲突记录，不覆盖有效票；
- 线上办理困难的业主可以转纸质路径，并保留渠道变更及送达证据。

验收场景：同一场表决中，业主甲线上投票、业主乙纸质投票，均正常计票；甲的重复纸票被识别并保留审计记录但不重复计票。

### 切片七：维修工程接入统一表决

- 部分共有维修和全体共有维修根据确认的决定范围创建表决；
- 维修项目只保存与统一表决包的关联，不再维护独立线上票或微信接龙结果表作为最终真相；
- 授权提案冻结实施方案、预算、费用承担范围和施工单位选择方式；
- 区分“公共收益按规约补充专项维修资金”和“具体项目直接列支”，不得在维修资金不足时临时切换付款来源；
- 使用上海住宅维修资金时，将首次支取 30% 和审价前累计 90% 实现为支取上限校验，不自动生成项目付款比例；
- 通过后才进入确定施工单位、合同和最终实施锁定；
- 微信等外部征询材料仅在有效规则允许且完成逐户核验时作为纸质/外部证据归档，不能只上传汇总截图代替逐户票。

验收：维修方案修订不会覆盖已表决版本；未通过或未完成计票的项目不能确定施工单位或签订合同。

### 切片八：管理端和业主端重构

管理端：

- 删除“小区议事规则”中的参考估算开关；
- 项目确需前期成本测算时，在实施方案中填写测算金额、依据和编制人；物业内部是否强制编制，只能由服务组织的有效内部要求决定，不再伪装成业主大会议事规则；
- 展示“本小区规则允许的表决方式”和“本次实际表决方式”；
- 用结构化表单登记有效送达、未反馈、纸质和线上规则；
- 分别提供纸票送达、回收、录入、复核和混合渠道冲突处理工作台；
- 将“定商”统一改为“确定施工单位”。

业主端：

- 线上方式显示实名投票入口；
- 纸质方式显示本人纸票办理状态；
- 混合方式允许未产生有效票的业主选择在线办理，并说明纸质办理路径；
- 只展示本人参与状态，不披露其他业主的票面选择。

### 切片九：迁移和切换

- 盘点旧维修在线表决、微信接龙、业主大会纸票及普通线上票数据；
- 仅在证据能够一一对应表决包、房屋、事项和方案版本时迁移为统一票；
- 无法证明的历史记录保留原样只读并标记来源，不补造有效票；
- 新流程启用后关闭旧写入口，避免长期双写；
- 生产切换前备份并核对表决包、有效票和计票结果数量。

## 4. 测试与验收矩阵

后端至少覆盖：

- 三种表决方式的正向流程；
- 规则未启用、方式不允许、规则换版和快照不变；
- 纸质、线上及并发跨渠道重复票；
- 多共有权人同一专有部分的表决权代表约束；
- 有效送达与未反馈规则；
- 一般事项与特别重大事项的不同计票阈值；
- 租户、决定范围和数据权限隔离；
- 历史表决结果不可被方案或规则换版改写。

端到端至少覆盖：

1. 物业形成维修方案并冻结授权提案；
2. 有权组织主体在有效规则允许范围内确定本次采用混合投票；
3. 部分业主在线投票，部分业主收到并提交纸票；
4. 重复票被阻断；
5. 截止后合并计票并发布结果；
6. 通过后按实施方案确定施工单位并进入合同；
7. 业主端可查看本人参与状态和公开结果。

## 5. 非本轮范围

- 多个独立楼栋决定范围的跨楼栋聚合；
- 根据示范文本自动生成江湾生效规则；
- 没有真实账户来源时推算维修资金或公共收益余额；
- 没有有效依据时预设验收角色、付款比例或留存款释放规则；
- 仅做前端三选一下拉框但后端仍分别硬编码纸质或线上。

## 6. 上线前置事实

代码可以先完成通用规则登记和三渠道能力，但江湾生产启用正式表决前必须取得并核对：

- 当前生效议事规则原件；
- 有效送达、未反馈和纸质/线上办理条款；
- 当前表决权代表及房屋名册；
- 专项维修资金规约和直接授权范围；
- 当前物业服务合同及维修责任边界。
