# 纸质与线上并行表决闭环设计

> 业务关联：让两个采集渠道在同一冻结名册上竞争同一有效票位置，并完整保留后到纸质原件和线上冲突审计。
>
> 状态：实施中（2026-07-19）
>
> 需求依据：[requirements.md](requirements.md)

## 1. 设计原则

```text
同一表决包 + 同一冻结名册 + 同一时间窗
           |
      +----+----+
      |         |
   线上整包   纸票双人复核
      |         |
      +----+----+
           |
  表决事项 + 名册项唯一有效票
```

`PAPER_AND_ONLINE` 只控制采集入口，不建立第二份票或第二套计票结果。先形成的统一有效票保留；后到纸票转为逐事项重复结果，后到线上请求写冲突审计后拒绝。

## 2. 规则与表决包

- `ONLINE_AND_OFFLINE + PAPER_AND_ONLINE + FIRST_VALID_WINS` 映射为统一表决包 `CollectionMode.PAPER_AND_ONLINE`；
- 冻结摘要包含规则版本、实际方式、全部事项、材料、模板、时间窗口和名册摘要；
- 纸质送达无需纸质协助申请；线上仍须本人阅读确认和活体实名；
- 表决开始后不得切换为互联网主渠道或纯纸质方式。

## 3. 跨渠道冲突

### 3.1 线上先形成

纸票原件、录入版本和复核记录照常保存。最终转换逐事项检查统一有效票：若已存在不同纸票或线上票，则写 `PaperBallotOutcome.DUPLICATE`，关联既有有效票编号并保留原因。

### 3.2 纸票先形成

线上服务在幂等重试判断之后、写入前检查统一有效票。发现既有票时，通过独立事务写 `ONLINE_BALLOT_CONFLICT` 审计，记录表决包、专有部分、当前自然人、请求内容摘要和时间，但不保存或展示票面选择；随后返回业务冲突。

### 3.3 并发

统一有效票表继续以“表决事项 + 冻结名册项”的活动票唯一索引兜底。线上整包任一事项冲突时整个在线事务回滚；纸票复核捕获逐事项冲突并把后到原件标记为重复材料。

## 4. 工作台投影

统一办理台返回：

- 纸票送达和复核状态；
- 回收纸票、录入版本和逐事项处理结果；
- 互联网表决中的纸质协助请求；
- 线上已完成的数量汇总和跨渠道冲突数量。

办理台不返回在线选择。业主端按每个专有部分返回渠道可用性、完成状态和本人回执，同样不回显选择。

## 5. 测试策略

- 规则测试验证不允许并行或非 `FIRST_VALID_WINS` 时阻断；
- 集成测试同时完成一户线上和一户纸票；
- 顺序冲突测试覆盖线上先、纸票先和线上幂等重试；
- 并发与数据库测试验证最多一张有效票；
- Web 投影测试验证管理端和业主端不泄露线上选择。
