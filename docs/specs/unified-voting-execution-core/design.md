# 统一表决执行内核设计

> 业务关联：将表决包、表决人名册、有效票和结果从业主大会专用流程中下沉为可复用内核。
>
> 状态：已完成（2026-07-19）
>
> 需求依据：[requirements.md](requirements.md)

## 1. 现有实现审计

现有代码已有可复用的通用议题、`t_vote_item`、计票引擎和结果快照，也已在业主大会模块保存表决包、送达和票据留痕。但这些事实尚未形成同一个执行边界：

- `t_owners_assembly_package`、送达和票据留痕被业主大会命名和仓储绑定，维修决定无法直接复用；
- 分母快照通常在结算时才生成，而正式表决应在表决包锁定时冻结名册；
- 投票提交仍从实时 `c_owner_property` 校验资格，没有以本次冻结名册作为唯一依据；
- 现有分母查询会根据 `account_status` 排除记录，但权威业务规范没有给出“欠费或账号冻结即丧失表决权”的依据；
- 通用线上投票入口可直接写入业主大会的正式议题，却没有验证本次表决包是否允许该渠道。

本切片不覆废通用计票引擎，而是为它补上正式业主共同决定所需的表决包和冻结名册边界。

## 2. 模块边界

### 2.1 通用内核

`voting` 领域拥有：

- `VotingExecutionPackage`：表决包聚合根；
- `VotingElectorateSnapshot`：包级表决人名册和计票基数；
- `VotingPackageSubject`：表决包与通用 `VotingSubject` 的唯一关系；
- `VotingDeliveryRecord`：逐户送达事实；
- `VotingBallotRecord`：票面、渠道和 `t_vote_item` 结构化有效票的对应关系；
- `VotingExecutionAuditEvent`：包状态及可审计动作。

### 2.2 上层流程

`assembly` 保留会次、公示材料和议事规则快照，并把这些依据交给通用内核创建表决包。它不再直接写入自有送达或票据台账。

`repair` 在切片七仅保存维修提案与通用表决包的关联，不新建维修专用有效票表。

## 3. 领域状态和不变式

```text
DRAFT -> FROZEN -> VOTING -> CLOSED -> SETTLED
   \          \          \-> VOIDED
    \----------\--------------> VOIDED
```

- `DRAFT`：只能绑定事项和完整依据，不能送达或收票；
- `FROZEN`：方案、规则、范围、渠道、时间和名册均已冻结；
- `VOTING`：在冻结时间窗口内接受该包允许的有效票；
- `CLOSED`：停止收票，允许以冻结事实结算；
- `SETTLED`：结果已固化；
- `VOIDED`：作废，历史只读。

核心不变式：

1. 一个表决事项最多属于一个通用表决包；
2. 一个已冻结表决包恰好对应一份表决人名册快照；
3. 有效票中的 `opid`、`uid` 和面积必须与名册快照一致；
4. 一般和重大决议中，同一 `packageId + subjectId + opid` 只能有一张有效票；
5. 通道只是有效票的属性，不参与唯一键。

## 4. 持久化设计

新增 `V3.110` 迁移，不修改历史迁移：

- `t_voting_execution_package`：通用表决包；
- `t_voting_package_subject`：表决包与议题关系；
- `t_voting_electorate_snapshot`、`t_voting_electorate_item_snapshot`：包级冻结名册；
- `t_voting_delivery_record`：通用送达记录；
- `t_voting_ballot_record`：通用票据留痕，继续关联 `t_vote_item`；
- `t_voting_execution_audit`：表决包执行审计；
- `t_voting_result` 增加表决包、名册、方案和规则摘要。

历史业主大会数据只在能够从现有外键和摘要无损还原时迁移。缺少名册或规则快照的记录保留原历史状态，不填充伪造摘要。

## 5. 应用服务

`VotingExecutionService` 作为唯一写入入口：

- `createPackage`：从上层已确认的依据创建草稿；
- `attachSubject`：在草稿阶段绑定通用议题；
- `freeze`：校验完整依据并生成不可变名册快照；
- `open`：到达时间后进入收票；
- `recordDelivery`：写入已由渠道适配器验证的送达事实；
- `cast`：以冻结名册校验资格并写入唯一有效票；
- `close` 与 `settle`：截止收票并使用冻结计票基数固化结果。

`VoteSubmissionService` 遇到已绑定通用表决包的正式事项时，必须转入 `VotingExecutionService`；不得绕过表决包渠道、名册和内容版本校验。

## 6. 业主大会接入

1. 确认公示与表决安排时，`assembly` 以会次、已锁定材料、议事规则快照和事项创建通用表决包；
2. 发布公示时由通用内核锁定表决包并冻结名册；
3. 开始投票时同步进入内核 `VOTING`；
4. 纸质送达和纸票录入不再写入业主大会专用表，改为调用通用内核；
5. 业主大会展示模型继续使用业务语言，不向物业或业主暴露内核表名和技术状态。

## 7. 测试策略

- 领域测试验证表决包状态机和依据完整性；
- 持久化集成测试验证名册在锁定时生成且不随实时房屋变化；
- 应用服务测试验证非名册人员、错误渠道和时间窗口被拒绝；
- 并发测试验证纸质票与线上票不能对同一专有部分同时生效；
- 结算测试验证结果使用表决包冻结的名册和规则。
