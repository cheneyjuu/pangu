# 维修报修至工程验收全流程需求

> 业务关联：把报修、工程、表决、施工和验收串成一条可执行、可审计的维修业务链。
>
> 验收依据：[维修报修至工程验收全流程验收规则](../../acceptance/维修报修至工程验收全流程验收规则.md)
>
> 状态：已确认目标，待实施（2026-07-20）

## 1. 范围

本需求从业主提交报修开始，以工程验收通过为终点。付款、资金支取、完工公示和质保办理不在本次范围。

本轮先覆盖单一楼栋费用承担范围，不实现跨楼栋聚合。报修事项与工程项目继续分离，一个工程可以关联一个或多个已经勘验的报修事项。

## 2. 报修与工程立项

1. 业主提交标题、位置、问题说明和现场证据，系统不得把附件数量说明写成勘验结论。
2. 物业完成受理、位置核实和现场勘验后，才能把共有部分事项交接为工程项目。
3. 工程必须保存来源工单关系；后续状态变化不得反向改写业主原始报修内容和勘验记录。
4. 实施方案必须明确工程范围、预算、费用承担范围、施工单位选择方式、施工管理、验收要求和实施时间；无法确认的事实保持待确认，不使用平台默认值。

## 3. 表决方式与重复票

1. 单次实际表决方式分为纸质、线上、纸质与线上并行三种；三者属于采集方式，不等同于重复票裁决策略。
2. 混合表决的重复票策略独立冻结为：首张有效票优先、线上有效票优先或纸质有效票优先。
3. 重复票策略必须来自当前生效规则，不允许项目经办人在发起时临时选择，也不允许系统自动补默认值。
4. 规则登记校验与运行支持能力必须共用同一能力判断。能够启用的规则不得在项目准备表决时因同一配置再次被判定为不可执行。
5. 发生替代时，原有效票转为被替代状态并保留票据、渠道、时间、替代理由和审计；最终计票只能读取当前有效票。
6. 并发提交必须保证同一表决包、事项和房屋最多一张当前有效票，且裁决结果符合冻结策略。

## 4. 规则就绪与办理

1. 准备接口一次返回所有缺口和当前规则允许的实际方式，不逐个暴露技术闸门。
2. 公示/通知期限只决定最早开始时间，不应掩盖其他规则配置错误。
3. 页面以“本小区规则约定的重复票处理方式”等业务语言说明结果，不展示内部枚举。
4. 既有尚未准备表决的项目直接读取修复后的当前有效规则，无需重建工程。
5. 已冻结表决继续使用当时规则快照，不被后续规则换版改写。

## 5. 施工与验收

1. 表决通过或有效授权形成后，才能按实施方案确定施工单位并归档合同。
2. 开工、施工记录、材料核验、竣工结算均必须绑定同一锁定方案和合同。
3. 验收要求必须冻结自实施方案、合同或有效决定，至少保存要求内容、依据附件和必需参与方；工作流类型不能替代依据。
4. 验收参与方提交结论和证据后，系统按冻结要求判断通过、未完成或需要整改。
5. 需要整改时，当前验收轮次只读保留，工程回到施工处理；重新提交后建立新验收轮次。
6. 验收通过时项目进入完成状态，生成不可变验收结果；本次流程到此结束，不自动创建付款请求。

## 6. 角色与隐私

1. 业主提交报修、本人表决及实施方案明确要求的本人验收动作。
2. 物业办理工单、勘验、方案、询价、施工过程和材料归档，但不能修改规则、计票结果或代替有权主体定案。
3. 业委会按权限核对责任与费用、确认表决安排、确定施工单位及履行方案约定的验收职责。
4. 施工单位提交报价、合同签署材料、施工和完工材料，不能确认自己的验收结果。
5. 管理端默认只展示汇总和办理状态，不展示其他业主的线上票面选择。

## 7. 验收标准

1. 主 BDD 场景从业主报修接口开始，经管理端和施工单位接口，到验收定案接口结束。
2. `ONLINE_PREVAILS` 的生效规则可以准备混合表决，并正确处理纸票先、线上票后的替代。
3. `PAPER_PREVAILS`、`FIRST_VALID_WINS` 均有独立行为测试；未知或不完整策略不能启用。
4. 规则登记与表决准备使用同一能力判定，测试不存在“登记成功但无可办理方式”的组合。
5. 验收通过后的工单、项目、方案、表决、施工单位、合同、施工记录、结算、验收和审计链可相互追溯。
