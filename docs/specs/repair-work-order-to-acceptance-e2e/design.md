# 维修报修至工程验收全流程设计

> 业务关联：以一条纵向业务链统一维修上下文，并修复规则登记与表决执行、方案验收要求与验收定案之间的断层。
>
> 需求依据：[requirements.md](requirements.md)
>
> 状态：待实施（2026-07-20）

## 1. 总体边界

```text
业主报修 -> 物业勘验 -> 工程方案 -> 前期询价 -> 责任与费用确认
       -> 相关业主表决 -> 确定施工单位 -> 合同 -> 施工留痕
       -> 竣工结算核验 -> 验收办理 -> 验收通过
```

每一段拥有自己的聚合和状态，但通过不可变标识连接：`workOrderId -> projectId -> planId -> votingPackageId -> selectionId -> contractId -> acceptanceId`。跨段推进只消费前一段已经确认的事实，不复制或重算前段结果。

## 2. 统一表决能力

### 2.1 能力判定单一来源

抽取 `VotingCollectionModeCapability` 领域策略，同时供以下入口使用：

- 议事规则提交/启用校验；
- 维修表决准备选项；
- 业主大会表决准备；
- 表决包冻结。

能力策略输入完整规则配置，输出允许的单次方式和全部不支持原因。调用方不得再各自维护一套渠道组合判断。

### 2.2 重复票裁决

新增 `DuplicateBallotResolutionPolicy` 接口，以冻结的 `DuplicateVotePolicy` 选择实现：

- `FirstValidWins`：已有有效票保留，后到材料记录为重复；
- `OnlinePrevails`：线上票到达时可替代纸质有效票；纸质票到达且已有线上票时记录为重复；
- `PaperPrevails`：纸质票复核时可替代线上有效票；线上票到达且已有纸质票时拒绝计入并记录冲突。

裁决输出 `KEEP_EXISTING`、`REPLACE_EXISTING` 或 `ACCEPT_NEW`。应用服务负责在同一事务内更新有效票状态、写入新票和记录审计。数据库约束保证同一事项和名册项最多一张当前有效票。

被替代票不是删除：`t_vote_item` 和统一票据记录增加明确失效状态、替代票引用、策略、原因和时间。计票、回执和汇总只读取当前有效票；审计接口可读取完整替代链，普通页面不展示其他业主的选择。

### 2.3 兼容

- 已冻结且采用 `FIRST_VALID_WINS` 的历史表决保持原语义；
- 尚未准备表决的项目读取当前有效规则，修复后无需重建；
- 已有 `ONLINE_PREVAILS` 或 `PAPER_PREVAILS` 规则可以继续使用，不修改规则原文或摘要；
- 不从历史汇总数据补造逐户替代链。

## 3. 验收要求模型

当前 `requiredAcceptanceRoles`、`acceptanceMethod` 与项目验收策略未形成同一可执行快照，且代码按楼栋/全小区硬编码参与人。目标设计以方案为真相建立 `RepairAcceptanceRequirementSnapshot`：

- `requirementId / projectId / planId / tenantId`；
- 业务名称、要求说明、参与方类型、是否必需；
- 通过条件和最低数量/比例（仅规则确有该口径时）；
- 依据附件、来源类型和来源摘要；
- 冻结人、冻结时间和快照摘要。

`RepairProjectAcceptanceService` 按快照生成可办理任务，并通过 `RepairAcceptanceRequirementEvaluator` 计算未完成、整改或通过。楼栋/全小区只决定可能的范围解析器，不能直接决定固定角色组合。

施工单位的完工提交与验收方结论分离；施工单位不能验收自己的工程。方案要求物业、业委会、业主代表、受影响业主或第三方时，分别由对应身份或原件代录适配器提交并保留证据。

## 4. 状态与事务

```text
工程草稿 -> 授权程序中 -> 已获授权 -> 施工中 -> 待验收 -> 已完成
                                      ^          |
                                      +--整改----+
```

- 表决结算和工程授权在同一事务中写治理依据并推进项目；
- 施工单位选择和合同不得早于授权；
- 核验竣工结算后创建验收轮次；
- 验收定案锁定项目和轮次，检查版本及全部要求；
- 整改使结算和验收要求按既有规则失效或进入新版本，不覆盖旧轮次；
- 通过后只推进到 `COMPLETED`，不触发付款或完工公示。

## 5. BDD 设计

扩展现有 `RepairProjectBuildingE2eFlowTest` 为主业务验收测试，但拆出可读步骤对象，避免单个测试方法堆积数据库细节。测试只通过公开 HTTP 接口创建业务事实；测试时钟用于推进公示、表决和截止时间。

纵向主场景使用 `ONLINE_PREVAILS`，完成：报修、勘验、立项、报价、责任确认、冻结方案、混合表决、线上替代纸票、结算、选择施工单位、合同、开工、过程证据、结算、验收要求办理和验收通过。

横向规则场景分别覆盖：

1. 三种采集方式；
2. 三种重复票策略的两个到达顺序；
3. 并发竞争；
4. 规则启用与准备选项一致；
5. 整改后新轮次；
6. 权限和隐私。

## 6. 三端职责

- 神农：业主报修、本人材料查看、投票/纸质协助、方案要求的本人验收任务；
- 盘古：物业及治理主体办理工单、方案、表决、施工和验收；
- 瑶池：施工单位报价、合同材料、施工和完工材料提交；
- Pangu 后端：身份、规则、有效票、状态、审计和验收结果的唯一可信来源。
