# 维修工程接入统一表决要求

> 业务关联：把已经冻结的共有部分维修提案，按本小区已生效议事规则交由相关业主表决，并把结果回写为不可变的工程授权依据。
>
> 状态：统一办理、未反馈票计票和书面委托代理能力已实现并完成本地回归，待部署验收（2026-07-20）

## 1. 范围与前提

1. 本能力只接收状态为 `AUTHORIZATION_IN_PROGRESS`、方案状态为 `AUTHORIZATION_FROZEN` 且具有授权提案摘要的维修工程。
2. 本阶段只办理共有部分维修及维修资金使用等一般共同决定事项，不办理筹集维修资金、改建重建、改变共有部分用途等重大决定；出现重大事项时必须另行建模并阻断发起，不能自动套用一般事项阈值。
3. 表决范围必须等于方案冻结时固化的费用承担房屋集合。工程位置、楼栋标签、报修人房屋和自由文本均不能替代该集合。
4. 必须存在本小区当前已启用的结构化议事规则。历史维修征询规则、平台默认值和自由文本规则不能用于新表决。
5. 本阶段不支持多个独立决定范围的跨楼栋聚合。

## 2. 本次办理方式

1. 经办人只能从当前已启用规则明确允许的方式中选择本次实际方式：纸质投票、互联网表决并提供纸质协助、纸质与线上并行。
2. 纸质与线上并行还必须确认规则中的重复票处理方式为“首张有效票生效”。平台不支持的优先覆盖规则必须在发起前阻断。
3. 表决开始和截止时间、规则版本、提案版本、费用承担房屋、代表关系和计票基数在冻结后不可修改。
4. 互联网表决并提供纸质协助时，纸质办理申请只改变该专有部分的采集渠道，不改变名册、时间窗口或计票规则。

## 3. 名册和计票基数

1. 每个冻结的费用承担房屋必须在当前有效房屋名册中存在，面积必须与方案冻结时的分摊快照一致。
2. 每个房屋必须有唯一、已核验的表决权代表；共有产权但未确认唯一代表时不得发起。
3. 方案冻结后产权、代表或面积发生变化时，不得静默刷新表决分母；应提示重新核验并形成新方案版本。
4. 人数按唯一表决权代表去重，面积按专有部分去重；逐房屋明细和聚合摘要必须可回查。

## 4. 办理闭环

1. **准备表决**：校验项目、方案、规则、方式和精确房屋集合，创建一个维修工程表决事项和一个统一表决包，并原子冻结名册。
2. **开始表决**：到达开始时间后开启同一表决包；全部渠道只能向该包写入。
3. **送达与收票**：复用统一纸票、线上实名和混合去重能力；管理端不得看到线上个人选择，审计权限接口除外。
4. **截止与结算**：截止后只读取冻结名册、有效票和规则快照，分别计算参与及同意的人数、面积并判断是否通过。
5. **通过**：生成指向表决包、表决事项、授权提案摘要和结果证明的治理依据，项目进入 `AUTHORIZED`。
6. **未通过**：项目回到可修订状态，原表决包和结果只读保留；再次发起必须创建新方案和新表决包。
7. 表决通过后才能确认最终实施方案、确定施工单位并签订合同；冻结或开始表决均不产生施工、合同或付款资格。
8. 结算后由系统从不可变计票快照自动形成公开汇总结果和计票存证；公开结果不得包含逐户选择，也不得由经办人手工改写。
9. 结算后，纸票送达、原件、录入版本、异人复核、无效或重复材料记录以及线上汇总仍须只读可查；结束收票只关闭写入动作，不能关闭归档查阅。

## 5. 未反馈票认定与计票

1. 未反馈票只能依据本次表决冻结的议事规则处理，系统支持“未反馈不计入参与”“未反馈按多数意见认定”“未反馈认定为弃权”三种方式，不得在结算时临时改选。
2. 可被认定为未反馈的表决权必须同时满足：属于冻结名册、至少存在一条规则认可的有效送达记录、截止时对该事项没有有效表决票。未送达、名册外房屋和已有有效票的表决权不得进入推定。
3. “按多数意见认定”的多数意见只从截止时已经形成的有效实际票中计算，推定票不得反向参与多数意见判断，避免循环放大。
4. 人数维度和面积维度必须分别存在唯一最高选项，且两者指向同一选项，系统才可形成多数意见；任一维度并列、两个维度结论不一致或没有实际有效票时，必须停止结算并提示治理主体核对，系统不得自行选择排序口径或随机决定。
5. “认定为弃权”不需要先形成多数意见；所有满足第 2 条的未反馈表决权按弃权计入参与人数、参与面积和弃权汇总。
6. 实际票与认定票必须分别统计、分别留痕。认定票不是业主提交的票，不得写入实际票表、不得生成业主投票回执，也不得在页面上表述为业主本人选择。
7. 每条认定记录必须固化表决包、事项、冻结名册项、规则版本与摘要、有效送达证据摘要、认定方式、认定选项、认定原因和结算时间；公开结果只展示实际票、认定票和合计汇总，不公开逐户选择。
8. 结果摘要必须包含实际票汇总、认定票汇总、合计汇总、未反馈资格数量与面积、认定方式、多数意见计算结果及认定记录聚合摘要，并与方案、规则、名册和表决包摘要共同存证。
9. 同一表决包内各事项独立判断是否存在实际票、未反馈资格和多数意见；不得用一个事项的多数意见替代另一个事项。

## 6. 角色与界面

1. 物业可以编制方案、准备材料和办理纸票，但不能启用议事规则、修改计票阈值或手工填写表决结果。
2. 有权治理主体负责确认并启用议事规则、确认发起正式表决及查看结果。
3. 业主端只展示本人有表决权的维修事项、完整提案材料、本人可用渠道、完成状态和回执。
4. 页面使用“准备相关业主表决”“开始表决”“确定施工单位和签订合同”等业务语言，不出现“定商”“服务端快照”“技术参与”等技术术语。

## 6A. 书面委托代理

1. 议事规则约定需要书面委托时，表示业主可以选择委托代理人办理纸质投票，不表示每名业主都必须登记委托；业主本人线上或纸质表决仍可照常办理。
2. 书面委托必须绑定本次冻结表决包、名册中的原业主房屋、代理人身份、覆盖完整表决窗口的有效期和原始文件。代理人不取得独立表决权，计票人数、面积和票的归属始终使用原业主房屋。
3. 一名工作人员登记后，必须由另一名工作人员核对委托人签名、代理人身份、授权范围和有效期；登记人不能核对自己的登记。
4. 尚未用于送达或收票的已确认委托可以撤销并保留原因与审计记录；已经绑定纸质表决票后不能撤销。
5. 代理人办理送达和收票时必须显式引用已确认委托；系统同时校验表决包、原业主房屋、状态和有效期，不能只凭代理人姓名放行。
6. 发起前就绪检查必须一次返回全部规则缺口。已支持的书面委托不得再作为晚到的技术阻断，页面也不得直接显示 `Access Denied` 或内部异常文案。

## 7. 历史兼容

1. 旧楼栋维修决定、微信接龙和手工计票只作为历史记录读取，切换后禁止新写入。
2. 旧的维修工程关联业主大会事项只读保留；尚未完成的记录必须显式迁移为统一表决包或人工终止，不得双轨继续收票。
3. 迁移必须有逐项目报告，记录旧流程状态、迁移结果、无法迁移原因和审计时间。
