# 维修工程接入统一表决任务

> 业务关联：跟踪维修授权提案接入统一表决、三端办理和历史迁移的完成情况。
>
> 状态：实施中（2026-07-19）

## 1. 精确范围与内核扩展

- [ ] 增加 `REPAIR_ALLOCATION` 范围和数据库约束；
- [ ] 增加按精确房屋集合读取候选名册的仓储能力；
- [ ] 抽取统一表决的精确名册冻结入口，保留单一校验和审计链；
- [ ] 校验方案分摊房屋与当前有效名册、面积和唯一代表完全一致；
- [ ] 覆盖范围漂移和原子回滚测试。

## 2. 维修表决编排

- [ ] 新增维修表决关联模型、仓储和迁移；
- [ ] 校验项目、冻结方案、活动规则和本次方式；
- [ ] 原子创建事项、表决包、精确名册和关联；
- [ ] 开始、截止、结算并将通过结果写成治理依据；
- [ ] 未通过时回到方案修订且保留历史记录；
- [ ] 覆盖通过、未通过、并发和重复发起测试。

## 3. 管理端与业主端接口

- [ ] 提供维修表决准备、开始、结算和进度接口；
- [ ] 提供维修纸票办理薄适配接口；
- [ ] 提供业主维修表决任务、阅读确认、投票、纸质协助和回执接口；
- [ ] 确认普通管理接口不泄露个人线上选择。

## 4. 三端交互

- [ ] 盘古以业务语言展示“准备相关业主表决”和渠道办理台；
- [ ] 瑶池如承担同一管理能力，复用相同契约且不建立第二套规则；
- [ ] 神农业主端展示材料、本人表决权、渠道、提交确认和回执；
- [ ] 删除“定商”“当前授权状态”“服务端快照”等面向业务人员的技术文案。

## 5. 历史迁移与切换

- [ ] 盘点旧楼栋决定和业主大会关联的在途数据；
- [ ] 生成逐项目迁移或终止报告；
- [ ] 关闭旧维修决定、微信接龙和手工结果的新写入口；
- [ ] 验证历史详情仍可读取且不会进入新计票。

## 6. 验证与发布

- [ ] 后端全量测试通过；
- [ ] 盘古、瑶池和神农构建通过；
- [ ] SSH 部署后验证数据库迁移、健康检查和静态资源；
- [ ] 在生产环境完成一条纸质、一条线上和一条混合的真实维修表决流程；
- [ ] 保存验收证据并更新权威规范和任务状态。
