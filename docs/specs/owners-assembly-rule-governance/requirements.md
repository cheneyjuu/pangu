# 业主大会议事规则版本治理需求

> 业务关联：登记本小区已经生效的业主大会议事规则，并在正式表决开始前形成可追溯、不可变的执行依据。
>
> 状态：已实现并复核（2026-07-19）
>
> 上位业务规范：[维修与业主大会表决业务规范](../../business/维修与业主大会表决业务规范.md)
>
> 对应改造切片：[维修与表决能力改造计划 - 切片二](../../plans/维修与表决能力改造计划.md#切片二议事规则版本治理)

## 1. 范围

本切片只解决议事规则版本的登记、核对、启用、替代和单次会议快照，不实现纸质、线上或混合投票的完整执行流程。

规则原件、结构化配置、来源页码与条款、逐项核对记录和状态审计共同构成一个可执行版本。任何单独的文件上传、表单保存或账号角色名称都不能使规则生效。

## 2. 参与主体

- **规则经办人**：具有规则草稿权限的物业或自治组织工作人员，可以上传原件、录入配置、修改草稿并提交核对；不能确认启用。
- **规则确认人**：系统根据当前小区的有效业委会职务记录识别出的主任或副主任，可以逐项核对并确认启用；客户端不能提交或覆盖其职务。
- **规则使用方**：正式业主大会流程只能读取当前已启用版本，并在确认会议安排时复制为该会次的不可变规则快照。

## 3. 功能需求

### R1 草稿不产生效力

1. 上传原件和结构化配置后只创建 `DRAFT`；
2. 草稿可以修改，待核对、已启用和已替代版本均不可修改；
3. 查询当前有效规则时不得返回草稿或待核对版本。

### R2 提交核对必须具备完整依据

提交核对前必须由服务端校验：

- 规则名称、版本、生效日期和变更原因完整；
- 原件已经私有存储，并保存文件名、类型、大小、ETag 和 SHA-256；
- 会议形式、期限、有效送达方式、未反馈处理、代理规则、表决方式、纸质与线上要求、重复票处理、计票阈值和公告期限均已明确；
- 每个结构化字段都关联有效页码和具体条款；
- 表决方式与线上身份核验、重复票处理之间不存在自相矛盾的组合。

系统不得用平台默认值补齐缺失内容。

### R3 逐项核对和启用权限

1. `PENDING_CONFIRMATION` 版本中的每个结构化字段分别保留核对状态、配置摘要、原件页码、条款、确认人职务和时间；
2. 只有当前小区有效职务为主任或副主任，并同时具有敏感操作权限的工作身份可以核对和启用；
3. 规则全部字段核对完成前不能启用；
4. 尚未到生效日期的版本不能提前启用。

### R4 版本替代和并发一致性

1. 每个小区同一时刻最多只有一个 `ACTIVE` 版本；
2. 新版本启用时，原 `ACTIVE` 版本转为 `SUPERSEDED`；
3. 启用过程必须按小区串行化，并由数据库唯一约束处理并发竞争；
4. 启用或替代失败时，整个状态迁移必须回滚。

### R5 不可变规则快照

1. 正式会议确认安排时，从当时的 `ACTIVE` 版本生成会次专属快照；
2. 快照保存规则身份、原件摘要、结构化配置及配置摘要；
3. 后续规则换版不能改变已生成的会次快照；
4. 没有有效规则或系统尚不能执行其配置时，可以继续会前准备，但不能进入正式送达、投票或计票。

### R6 租户、权限和原件访问

1. 所有读写按当前工作身份和小区租户隔离；
2. 普通租户工作身份不能通过请求参数访问其他小区规则；
3. 原件只能通过有权限的短时预览凭证访问，不返回长期对象存储地址；
4. 草稿创建、修改、提交、启用和替代动作必须保留审计记录。

## 4. 验收场景

1. 物业上传完整规则后得到草稿，不能直接启用；
2. 缺少任一结构化字段或来源条款时，提交核对失败；
3. 物业即使持有草稿权限，也不能逐项核对或启用；
4. 非当前主任或副主任的自治账号不能启用；
5. 主任逐项核对后启用，新版本启用会替代旧版本；
6. 其他小区的规则编号不能被当前小区账号读取或操作；
7. 已冻结会次继续显示旧规则版本，新启用版本只影响以后创建的会次；
8. 规则允许线上或混合表决，但对应证据状态机尚未实现时，正式流程明确阻断。

## 5. 非本切片范围

- 统一纸质、线上和混合投票执行内核；
- 将历史维修事项专用规则直接转换为业主大会议事规则；
- 根据示范文本或样本材料推定江湾国际公寓的有效规则；
- 自动解析原件并跳过人工逐项核对；
- 业委会成员任期备案资料的产生和审核。
