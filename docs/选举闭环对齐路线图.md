## 0. 目标

把业务设计稿《选举闭环.md》定义的「角色矩阵、状态机、组织拓扑、双签流程、监控基线」全面落地到 pangu 后端、yaochi 管理端、shennong-app C 端。设计稿与现状存在 5 类差距，需按 **梯度 A → E** 顺序推进；每梯度独立 PR + 用户验证后才进入下一梯度，避免一次性大改导致回滚困难。

## 4. 梯度 B：议题状态机扩展（双签流程）

### 4.1 目标 + 决策落地（2026-06-27 已对齐）

引入设计稿的 `PENDING_COMMITTEE → PENDING_STREET → APPROVED` 三个审批中状态，但按以下决策落地：

| 决策点 | 选择 | 说明 |
|---|---|---|
| Q1 适用范围 | **α 仅 ELECTION** | 一般决议 / 重大决议保持 DRAFT → PUBLISHED 一步直跳；ELECTION 强制走双签 |
| Q2 与候选人审查 | **α 保留独立** | 候选人三段审查（PARTY_REVIEW → COMMITTEE_REVIEW → APPROVED）与议题双签**两个状态机并行**，互不合并 |
| Q3 与 Waiver | **α Waiver 独立** | Waiver 仅在党员比例红线触发时启用；议题双签是 ELECTION 立项的常规流程 |
| Q4 是否引入 APPROVED 中间态 | **β 不引入**（实施决策） | 合并设计稿的 APPROVED + PUBLISHED 为 PUBLISHED 单一状态；街道办的"行政特批通过"动作 = 直接推到 PUBLISHED（即"终审通过 + 一键发榜"合并为一步）。原因：最小破坏现状状态机，避免引入 4 个新状态导致复杂度爆炸 |

### 4.2 最终状态机（梯度 B 后）

```

  DRAFT ──submit──> PENDING_COMMITTEE ──committee_approve──> PENDING_STREET ──street_approve──> PUBLISHED ──open──> VOTING ──close──> CLOSED ──settle──> SETTLED
                          │ committee_reject(reason)                │ street_reject(reason)
                          v                                          v
                       DRAFT                                       DRAFT

GENERAL / MAJOR 议题（不变）：
  DRAFT ──publish──> PUBLISHED ──open──> VOTING ──close──> CLOSED ──settle──> SETTLED
```
