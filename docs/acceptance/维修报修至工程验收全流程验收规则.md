# 维修报修至工程验收全流程验收规则

> 业务关联：验证一笔共有部分维修从业主报修进入物业办理，经过工程立项、相关业主表决、施工和竣工验收，最终形成可追溯的验收通过事实。
>
> 验收终点：工程验收通过；付款申请、付款审批、资金支取、完工公示和质保期管理不属于本次验收范围。
>
> 法规与事实依据：本小区已生效议事规则、项目实施方案及合同是项目级真相；平台不得用默认值替代。上海市维修资金规定要求实施方案包含实施范围、预算、分摊范围、施工单位选择方式、施工管理、工程验收和实施时间，并在表决通过后确定施工单位、签订合同和组织验收。

## 1. 验收目标

1. 同一笔业务可以从业主报修连续办理到工程验收通过，不靠修改数据库、跳过状态或补造快照放行。
2. 神农、盘古、瑶池共用同一工单、工程、报价、表决、合同和验收事实，不各自保存冲突结果。
3. 规则登记时被系统接受的配置，在对应业务前提满足后必须可执行；系统不得在流程后段才暴露新的技术闸门。
4. 表决支持纸质、线上、纸质与线上并行三种实际办理方式。混合表决发生重复票时，按本次冻结的议事规则执行，不得硬编码为某一种策略。
5. 验收参与人、通过条件和所需材料来自已批准实施方案、合同及有效规则。楼栋维修或全小区维修标签不能单独推导固定验收角色。
6. 每一步失败都返回业务人员可理解、可处理的完整原因；页面不显示内部枚举、`Access Denied`、服务端快照等技术术语。

## 2. 主验收场景

### 场景：楼栋共有部位维修采用线上线下混合表决并完成工程验收

**Given**

- 小区存在有效房屋、业主、物业、业委会和施工单位身份；
- 小区存在已经核对并启用的议事规则，允许线上线下结合，重复票规则为“线上有效票优先”；
- 议事规则明确送达、未反馈、委托代理、计票阈值及公示/通知期限；
- 工程验收要求已在实施方案中明确，并可追溯到方案或合同附件；
- 测试数据与真实业务隔离，且测试结束后可清理。

**When**

1. 业主在神农提交包含标题、位置、问题说明和现场照片的报修；
2. 物业在盘古受理、核实位置、完成现场勘验并确认属于共有部分维修；
3. 物业由该工单创建维修工程，填写维修点位、方案说明、预算和费用承担房屋；
4. 物业发出前期询价，施工单位在瑶池提交报价原件和结构化报价，物业完成核对；
5. 物业提交责任与费用初步意见，业委会核对后冻结供相关业主审议的实施方案；
6. 业委会选择“纸质与线上并行”，系统冻结规则、方案、费用承担房屋、计票基数和表决时间；
7. 一户先提交纸质票后再提交线上票，另一户只提交纸质票，其余有权业主线上表决；
8. 系统按“线上有效票优先”保留第一户线上票为有效票，纸质票保留原件并记录被替代原因；
9. 表决截止后系统按冻结规则自动计票，形成不可变结果并推动工程获得授权；
10. 按已批准的施工单位选择方式确认施工单位并归档合同；
11. 登记开工、施工前/中/后过程证据和必要的材料核验；
12. 施工单位提交竣工结算和完工材料，物业完成核验；
13. 按实施方案和合同冻结的验收要求收集参与人结论与证据；
14. 所有必需条件满足后，有权定案人完成验收定案。

**Then**

- 工单可追溯到工程，工程可追溯到来源工单；
- 报价、表决、施工单位选择、合同、施工记录、结算和验收均指向同一已锁定方案；
- 同一事项同一房屋最终只有一张有效票，纸质和线上原始材料均保留；
- 表决结果能够追溯到规则、名册、送达、实际票、认定票及重复票裁决；
- 工程状态为“验收通过”，验收轮次状态为 `PASSED`；
- 未进入付款、资金支取、完工公示或质保状态；
- 普通物业人员和业主看不到其他业主的票面选择。

## 3. 必验矩阵

| 能力 | 必验结果 |
| --- | --- |
| 纸质表决 | 纸票送达、回收、录入、异人复核、有效票和归档可追溯 |
| 线上表决 | 实名身份、房屋表决权、材料确认、整包提交和回执可追溯 |
| 混合表决 | 两种渠道共用一个表决包、名册和计票结果 |
| 首张有效票优先 | 后形成的有效材料不覆盖先形成票，但原件和冲突留痕保留 |
| 线上有效票优先 | 线上票替代既有纸质有效票；后到纸票不得覆盖线上票 |
| 纸质有效票优先 | 纸质票复核后替代既有线上有效票；后到线上票不得覆盖纸票 |
| 委托代理 | 委托原件、异人核对、有效期和代理纸票绑定完整，票仍归原房屋 |
| 未反馈处理 | 按冻结规则执行，实际票和认定票分开统计和留痕 |
| 整改 | 任一必需验收结论要求整改时不能验收通过；整改后形成新验收轮次 |
| 权限 | 物业不能修改规则或计票结果，施工单位不能验收定案，无权业主不能投票或验收 |

## 4. 一次性就绪检查

正式表决前一次返回全部缺口，至少包括：有效规则、系统可执行的实际办理方式、送达方式、未反馈处理、委托代理、重复票规则、计票阈值、唯一表决代表、费用承担房屋、法定面积、方案和表决时间。

施工前一次返回全部缺口，至少包括：已通过的授权、施工单位选择结果、有效合同、施工范围和开工条件。

验收前一次返回全部缺口，至少包括：已核验竣工结算、施工证据、验收要求来源、必需参与人、必需附件及尚未满足的通过条件。

## 5. 验收门禁

1. 主场景必须通过公开 HTTP 接口完成，BDD 不得直接插入业务结果或修改状态；仅允许通过测试时钟推进等待期限。
2. 领域和接口测试必须覆盖三种表决方式及三种重复票策略。
3. Pangu 全量测试、Yaochi 类型检查和构建、Shennong 类型检查和体验构建全部通过。
4. 生产部署后使用隔离测试数据跑同一主场景，核验 Flyway、服务、静态资源、状态和审计记录。
5. 任一环节仍需手工改库或出现新的晚到技术阻断，均判定本次目标未完成。
