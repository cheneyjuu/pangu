## 1. 目标 & 边界回顾

M3-3 把 ELECTION 候选人资格审查接通为**单段闸**：

```
提名 → PENDING_REVIEW(1) --candidate:approve--> APPROVED(2) / REJECTED(3)
```

按「党建引领 + 居委会把关」的治理语义，业委会候选人在进入居委会资格审查之前，应先由**党组书记**做一道政治 / 资格初筛（前置政审）。本期在原单段闸前插入这道前置审查，把资格审查改造为**两段状态机**：

```
提名 → PENDING_PARTY_REVIEW(1)
       --党组书记前置审查 approve--> PENDING_COMMITTEE_REVIEW(5)
       --党组书记前置审查 reject -->  REJECTED(3)
PENDING_COMMITTEE_REVIEW(5)
       --居委会资格审查 approve--> APPROVED(2)
       --居委会资格审查 reject -->  REJECTED(3)
```

| 维度 | 本期落地 |
|---|---|
| ✅ 党组书记前置审查 | 新增 `candidate:review:party`（G 端），`PENDING_PARTY_REVIEW → PENDING_COMMITTEE_REVIEW / REJECTED` |
| ✅ 居委会资格审查改源态 | 复用 `candidate:approve`，源态从 `PENDING_REVIEW` 收窄为 `PENDING_COMMITTEE_REVIEW` |
| ✅ 严格分权 | `candidate:approve` 从 `PARTY_SECRETARY(3)` 收回；党组只做前置政审、居委会只做资格把关 |
| ✅ 阶段化乐观锁 | `updateQualification(id, expectedFrom, newStatus)`，防并发 + 兜底跳过党组前置审查直接资格通过 |
| ✅ 数据零迁移 | 历史值 `1` 语义平移为 `PENDING_PARTY_REVIEW`，仅新增待办值 `5` |
| ❌ 党组前置审查的独立审计/留痕表 | 沿用 `qualification_status` 单列状态，不建审查流水表，推迟 |
| ❌ 前置审查驳回理由 / 退回重审 | 本期前置审查仅 approve/reject 二态，无"退回补正"，推迟 |

### 已锁定的设计决策

1. **严格分权（用户一对一确认）**：新增 `candidate:review:party` 授 `GOV_SUPER_ADMIN(1)` + `PARTY_SECRETARY(3)`；`candidate:approve` 从 `PARTY_SECRETARY(3)` **收回**，仅留 `GOV_SUPER_ADMIN(1)` + `COMMUNITY_ADMIN(2)`。党组书记自此**只能**做前置政审、不能直接做居委会资格审查；街道办超管保留双闸 override。
2. **dbValue 平移、零数据迁移**：历史 `PENDING_REVIEW=1` 的语义平移为 `PENDING_PARTY_REVIEW=1`——既有 `qualification_status=1` 的行天然落到"待党组前置审查"，无需任何 `UPDATE`。新增 `PENDING_COMMITTEE_REVIEW=5`（而非插队改号），保证旧值含义不变。
3. **源态守卫下沉领域 + DB 双保险**：两个驳回动作（`partyReject` / `reject`）都落到同一终态 `REJECTED`，单凭 `(from→to)` 跳转表无法区分阶段归属，故在 `ElectionCandidateActions` 内显式 `requireCurrent(candidate, expected)` 点名各自唯一允许的源态；service 层捕获 `from` 状态作为 `expectedFrom` 传给阶段化乐观锁 `updateQualification`，DB 侧 `WHERE qualification_status = #{expectedFrom}` 兜底并发与跨阶段越权。

## 2. 落地清单

### 2.1 V3.2 schema（`db/migration/V3.2__candidate_party_prereview.sql`）

```sql
-- 1. 放开 CHECK 约束，允许新增待办值 5
ALTER TABLE t_election_candidate DROP CONSTRAINT chk_candidate_qualification;
ALTER TABLE t_election_candidate ADD CONSTRAINT chk_candidate_qualification
    CHECK (qualification_status IN (1, 2, 3, 4, 5));

-- 2. 新增能力点（G 端，redline=0，与 candidate:approve 一致）
INSERT INTO sys_permission (...) VALUES
    ('candidate:review:party', '党组书记前置审查（政治/资格初筛）', 'VOTING', 'G', 0);

-- 3. 授权 candidate:review:party → GOV_SUPER_ADMIN(1) + PARTY_SECRETARY(3)
INSERT INTO sys_role_permission (role_id, permission_key) VALUES
    (1, 'candidate:review:party'), (3, 'candidate:review:party');

-- 4. 严格分权：从 PARTY_SECRETARY(3) 收回 candidate:approve
DELETE FROM sys_role_permission WHERE role_id = 3 AND permission_key = 'candidate:approve';
```

> **无新增 trigger**（仅 `ALTER` 既有 CHECK 约束）；trigger 编号沿用 V3.1（已占用至 13）。
> **无新增索引**：`idx_election_candidate_subject_status`（V3.1）已覆盖按 `(subject_id, qualification_status)` 的审查/列表查询。

### 2.2 Domain 层（`pangu-domain/.../model/voting/`，纯 Java 无 Spring）

- **`CandidateStatus`** —— `PENDING_PARTY_REVIEW(1)` / `APPROVED(2)` / `REJECTED(3)` / `WITHDRAWN(4)` / **新增 `PENDING_COMMITTEE_REVIEW(5)`**；`fromDbValue/getDbValue` 不变。
- **`ElectionCandidateActions`** —— 两段化跳转表：
  - `PENDING_PARTY_REVIEW → {PENDING_COMMITTEE_REVIEW, REJECTED, WITHDRAWN}`
  - `PENDING_COMMITTEE_REVIEW → {APPROVED, REJECTED, WITHDRAWN}`
  - 新增 `partyApprove`（→`PENDING_COMMITTEE_REVIEW`）/ `partyReject`（源态守卫后 →`REJECTED`）；`approve` 现**只**能从 `PENDING_COMMITTEE_REVIEW` 触发（跳过党组前置审查直接 `approve` 被状态机拒绝）；`reject` 源态守卫为 `PENDING_COMMITTEE_REVIEW`。
  - 新增私有 `requireCurrent(candidate, expected)`：两个驳回动作落同一终态，借此点名各自源态。

### 2.3 Application 层（`pangu-application/.../voting/`）

- **`ElectionCandidateService`**：
  - 新增 `partyReview(PartyReviewCandidateCommand)` —— 取候选人（不存在 → `CANDIDATE_NOT_FOUND`），捕获 `from` 状态，`partyApprove/partyReject` 迁移（非法迁移 → `CANDIDATE_REVIEW_CONFLICT`），`updateQualification(id, from, newStatus)` 返回非 1 → `CANDIDATE_REVIEW_CONFLICT`。
  - `review(ReviewCandidateCommand)` —— 同上模式，源态现为 `PENDING_COMMITTEE_REVIEW`。
- **`command/`** 新增 `PartyReviewCandidateCommand(candidateId, approve, operatorUserId)`。
- **`VotingApplicationException`** —— `CANDIDATE_REVIEW_CONFLICT` 复用于两段冲突（无需新增 Reason），注释更新为"候选人已不在期望的审查阶段"。

### 2.4 Infrastructure 层

- **`ElectionCandidateMapper`(.java/.xml)** —— `updateQualification` 的 `WHERE` 改为 `candidate_id = #{candidateId} AND qualification_status = #{expectedFrom}`；`insertCandidate` 仍写字面量 1（提名落 `PENDING_PARTY_REVIEW`）。

## 5. 已识别的剩余风险

1. **审查无独立留痕**：前置/资格审查只翻 `qualification_status` 单列，不记"谁在何时以何理由审查"的流水。当前满足"两段把关"的状态需求，但无法回溯审查链；若后续需要审计/驳回理由/退回补正，需升级为独立审查流水表（本期评估后不做）。
2. **严格分权对存量授权的影响**：V3.2 `DELETE` 收回了 `PARTY_SECRETARY(3)` 的 `candidate:approve`。若生产环境曾依赖党组书记直接做资格审查，迁移后该路径将 403——这是分权决策的预期结果，需在上线说明中提示运营调整流程。
3. **跨阶段越权的双重防护边界**：领域 `requireCurrent` + 跳转表是主校验，DB `expectedFrom` 乐观锁是并发与兜底；二者一致性靠 service 层"先取 `from` 再迁移再以 `from` 更新"的写法保证。若未来新增绕过 `ElectionCandidateService` 的直写路径，需同样遵守该取值约定。
4. **dbValue 5 的非连续编号**：`PENDING_COMMITTEE_REVIEW=5` 排在 `WITHDRAWN=4` 之后而非按流程顺序插队，是为保历史值不变；阅读状态机时需以 `ElectionCandidateActions` 跳转表为准，不可凭 dbValue 大小推断流程先后。
