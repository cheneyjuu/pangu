## 0. 目标与非目标

### 0.1 目标

M1 把"**人 / 权 / 部门 / 租户**"四件套搬上轨道，但在三个维度还留了短板：

1. **治理动作不可锁** —— 财务公示、选举公示、资金台账发布只要 admin 想改还能直接 `UPDATE`，缺少"业委会主任 + 街道办双签解锁"的通用范式
2. **审计不可追溯** —— 业主拿到的"上一期 vs 这一期公示"完全要靠肉眼对比，没有统计版本号一致性、也没有 W/R/N 差分
3. **角色不可动态扩展** —— SaaS 管理员只能用预置的 13 个 system role；业主自治场景里"志愿队队长 / 楼栋长 / 维修组组长" 这类长尾角色没法在管理后台建出来

M2 解决这三件事，并清理 M1 留下的四个**已知遗留**。

### 0.2 非目标

- 不动 M1 的 RBAC schema（V1.x / V1.4）—— M1 一旦合并到 main 就**冻结**，从 V2.x 开始向前兼容
- 不接真实司法链（区块链 SDK 集成）—— 留给 M3
- 不改前端

## 2. M2-1：governance_lock 通用机制

### 2.1 痛点

M1 的 `PartyRatioWaiver` 已经验证了"业委会 / 街道办双签"范式可以工作，但只针对**党员比例放宽**这一个 entity。PRD §3.3 备案与刻章、§5.1 共有资金、§7.1 投票表决报告 都要求"一旦公示锁定，非双签解锁不能改"——这套机制不能为每个 entity 复制 schema。

### 2.2 Schema（V2.5）

```sql
CREATE TABLE t_governance_lock (
    lock_id BIGSERIAL PRIMARY KEY,
    tenant_id BIGINT NOT NULL,
    entity_type VARCHAR(32) NOT NULL,        -- FINANCE_DISCLOSURE / ELECTION_DISCLOSURE / FUND_LEDGER_PUBLISH
    entity_id BIGINT NOT NULL,
    locked_by_user_id BIGINT NOT NULL,
    locked_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
    lock_payload_hash VARCHAR(64) NOT NULL,   -- 锁定瞬间整体快照 SHA256

    -- 双签解锁
    unlock_committee_user_id BIGINT,
    unlock_committee_at TIMESTAMP,
    unlock_committee_signature VARCHAR(128),
    unlock_street_user_id BIGINT,
    unlock_street_at TIMESTAMP,
    unlock_street_signature VARCHAR(128),
    unlock_at TIMESTAMP,                      -- 双签齐备瞬间填充

    version INT NOT NULL DEFAULT 0,           -- 乐观锁
    UNIQUE (tenant_id, entity_type, entity_id),
    CONSTRAINT chk_unlock_signers_diff CHECK (
        unlock_committee_user_id IS NULL OR
        unlock_street_user_id IS NULL OR
        unlock_committee_user_id != unlock_street_user_id
    )
);

-- Trigger 8：unlock_at 必须在双签齐备瞬间填充，禁止单独 UPDATE
CREATE OR REPLACE FUNCTION fn_governance_lock_unlock_atomicity() ...;
CREATE TRIGGER trg_governance_lock_unlock BEFORE UPDATE ON t_governance_lock ...;
```

### 2.3 Domain / Application 改动

- 新增 `pangu-domain/.../model/lock/GovernanceLock.java` 聚合根
- 新增 `pangu-domain/.../model/lock/LockEntityType.java` enum（FINANCE_DISCLOSURE / ELECTION_DISCLOSURE / FUND_LEDGER_PUBLISH）
- 新增 `pangu-application/.../lock/GovernanceLockApplicationService`：`lock(...)` / `committeeSign(...)` / `streetSign(...)` / `verifyUnlocked(entityType, entityId)`
- 改造 `PartyRatioWaiver` —— ⚠️ **不要破坏 M1 schema**，只是把 `WaiverApplicationService.streetReview` 通过时**额外**写一条 `t_governance_lock` 记录，让 `t_party_ratio_waiver.local_payload_hash` 与 `t_governance_lock.lock_payload_hash` 平行存在；后续两套并行一段，下个 M 再考虑统一

### 2.4 验收

- 单元：`GovernanceLockApplicationServiceTest`（双签矩阵 + 同人禁双签 + version 乐观锁冲突）
- 集成：`GovernanceLockMatrixTest`（三种 entityType 的锁定 / 半签 / 全签流程）
- 触发器反例：`SysGovernanceLockTriggerTest`（trigger 8 反例）

## 3. M2-2：statistics_version 注册中心 + ETag

### 3.1 痛点

M1 已经把 `t_voting_result.settle_version → statistics_version` 改名对齐，但仍**只在投票结果**这一个表里有版本号。业主拉取财务报表 / 选举名册时，前端没法判断"我手上的版本号还有效吗"——服务端旁路 `UPDATE` 后 ETag 不变，前端缓存错配。

### 3.2 Schema（V2.6）

```sql
CREATE TABLE t_statistics_version_registry (
    tenant_id BIGINT NOT NULL,
    entity_type VARCHAR(32) NOT NULL,
    entity_id BIGINT NOT NULL,
    version INT NOT NULL,
    published_at TIMESTAMP NOT NULL,
    published_by BIGINT NOT NULL,
    payload_hash VARCHAR(64) NOT NULL,
    PRIMARY KEY (tenant_id, entity_type, entity_id, version)
);

CREATE INDEX idx_stat_version_latest ON t_statistics_version_registry (tenant_id, entity_type, entity_id, version DESC);
```

### 3.3 Web 层改动

- 新增 `StatisticsVersionAdvice`（`@RestControllerAdvice`）：
  - 标了 `@VersionedEntity(type=...)` 的 controller 方法返回时拉最新 version 设进 `ETag`
  - 请求带 `If-None-Match` 时若版本未变直接 `304 Not Modified`
- 新增 `pangu-domain/.../repository/StatisticsVersionRepository`
- VotingResult 写入时同步 upsert 一条 registry 记录

### 3.4 验收

- `StatisticsVersionRegistryTest`（registry 表的写入语义）
- `EtagAdviceTest`（304 / 200 矩阵）
- 改造 `ElectionWorkflowIntegrationTest` 加一段 ETag 验证

## 4. M2-3：财务公示与 W/R/N 差分

### 4.1 痛点

PRD §5.1 / §5.2 要求物业 / 业委会按月或按季公示共有资金、维修资金动账，业主能看历史版本并做差分审计。现状是只有 `t_fund_ledger_entry` 流水，**没有"快照公示"** 概念，业主拉到的就是 raw 流水，不可比对。

### 4.2 Schema（V2.7）

```sql
CREATE TABLE t_finance_disclosure_snapshot (
    snapshot_id BIGSERIAL PRIMARY KEY,
    tenant_id BIGINT NOT NULL,
    period VARCHAR(20) NOT NULL,             -- '2026Q1' / '2026-06'
    disclosure_type VARCHAR(32) NOT NULL,    -- COMMON_FUND / MAINTENANCE_FUND
    data_payload JSONB NOT NULL,             -- 行级账户余额 + 流水汇总
    statistics_version INT NOT NULL,
    payload_hash VARCHAR(64) NOT NULL,
    locked_at TIMESTAMP,
    locked_by_user_id BIGINT,
    governance_lock_id BIGINT REFERENCES t_governance_lock(lock_id),
    published_at TIMESTAMP,
    UNIQUE (tenant_id, disclosure_type, period)
);

CREATE TABLE t_disclosure_audit_compare (
    compare_id BIGSERIAL PRIMARY KEY,
    prev_snapshot_id BIGINT NOT NULL REFERENCES t_finance_disclosure_snapshot,
    curr_snapshot_id BIGINT NOT NULL REFERENCES t_finance_disclosure_snapshot,
    diff_json JSONB NOT NULL,                -- {writes:[...], reads:[...], unchanged:[...]}
    audited_by BIGINT,
    audited_at TIMESTAMP
);
```

### 4.3 Application 改动

- 新增 `pangu-application/.../disclosure/FinanceDisclosureApplicationService`：
  - `compose(period, type)` —— 从 `t_maintenance_fund_account` + `t_fund_ledger_entry` 聚合生成 payload
  - `lockAndPublish(snapshotId)` —— 调 `GovernanceLockApplicationService.lock(FINANCE_DISCLOSURE, snapshotId)` 后 publish
  - `compare(prev, curr)` —— 跑 W/R/N diff 工具，落库 audit_compare
- 新增 `DisclosureDiffCalculator`（纯 Java，按 JSON path 比对）

### 4.4 Web 层

- `GET /api/v1/disclosures/{snapshotId}` —— 返回单期快照
- `GET /api/v1/disclosures/{snapshotId}/compare/{vsVersion}` —— 返回 W/R/N 差分
- `POST /api/v1/disclosures/{snapshotId}/publish` —— 走 governance_lock 锁住

### 4.5 验收

- `FinanceDisclosureComposeTest`（payload 聚合正确性）
- `DisclosureDiffCalculatorTest`（W/R/N 三路矩阵）
- 集成：`FinanceDisclosureWorkflowTest`（compose → lock → publish → 业主拉取 → compare）

## 6. M2-5：M1 遗留清理

### 6.1 `pangu-infrastructure → pangu-application` 反向依赖

`pangu-infrastructure/pom.xml` 当前依赖 `pangu-application`，违反 Clean Architecture 单向依赖图。M1 通过"新 port 一律放 domain"绕开，没有真正修。

**做法**：
- 排查 infrastructure 里所有引用 `application.*` 的类，把依赖归并到 domain（移 interface）或 interfaces（移 web 适配器）
- 删 `pangu-infrastructure` 对 `pangu-application` 的 maven 依赖
- 跑全量 mvn test 确认通过

### 6.2 生产空 UserContext 路径测试

M1 已经把 `DataScopeInterceptor` 的生产 mock 兜底移除了，但生产模式下 `UserContext == null` 时的处理路径还没 e2e 测试覆盖。

**做法**：
- 新增 `ProductionEmptyContextTest`：`@ActiveProfiles("prod")` + 不带 Authorization header → 触发 `@DataScope` mapper 应抛 `DATA_SCOPE_NO_CONTEXT`，不该回退到 ALL_COMMUNITY
- 增 `pangu-bootstrap/src/test/resources/application-prod.yml` 关键覆盖（mock loader 关闭）

### 6.3 `@Scheduled` 调度器依赖

M1 的 `VotingDeadlineScheduler` 在 `pangu-application` 模块用了 `@Scheduled`，但 application 当前只依赖 `spring-context + spring-tx`，并未启用调度（实际启动时由 bootstrap 的 `@EnableScheduling` 兜底）。

**做法**：要么把调度器迁移到 `pangu-bootstrap`，要么给 application 加一个 `@EnableScheduling` 的 `@Configuration`；选前者更干净。

## 8. 决策点与开放问题

> 进入 M2-1 实施前需要锁定的几个二选一：

| # | 议题 | 选项 A | 选项 B | 推荐 |
|---|---|---|---|---|
| Q1 | governance_lock 是替代还是并行 PartyRatioWaiver？ | 替代：把 Waiver 的双签字段迁到 lock，删除 Waiver 双签字段 | 并行：Waiver 保留自有双签，lock 只承载新 entityType | **B**（M1 schema 冻结，迁移代价高） |
| Q2 | statistics_version 是用 advice 自动注入 ETag 还是 controller 显式 set | A：`@VersionedEntity` + advice | B：每个 controller 手动 `ResponseEntity.ok().eTag(...)` | **A**（一致性 + 减重复代码） |
| Q3 | DisclosureDiffCalculator 比对粒度 | A：JSON path 级 | B：账户级（按 account_id 比对 balance） | **B**（业主关心账户余额变化，不关心字段名） |
| Q4 | M2-5.1 是否同 PR 修反向依赖 | A：单独 PR | B：跟 M2-1 一起修 | **A**（便于 reviewer 集中看依赖图变化） |

## 9. 风险登记

| 风险 | 严重度 | 缓解 |
|---|---|---|
| trigger 8 atomicity 在并发双签下失效 | 高 | DB 加 `version` 乐观锁 + Redis 分布式锁双保险（同 M1 Waiver 范式） |
| 财务 disclosure 锁定后业务想"撤回" | 中 | 走"双签解锁 → 修改 → 重新发布"标准流程；不开后门 |
| ETag advice 影响其它非 versioned 接口性能 | 低 | 只对 `@VersionedEntity` 标注的方法生效，零影响 |
| 角色管理后端被滥用 | 高 | trigger 6/7 兜底 + admin:role:manage 严格 PreAuthorize + 操作日志全量审计 |
