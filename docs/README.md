## 证据材料

`docs/sample` 保存报价、维修意见、验收单、送达表和公示材料等样本。样本只证明对应文件中出现的事实，不能自动推导全小区规则、合同关系、资金账户或平台通用流程。

`docs/certificate` 不是文档目录。证书和密码不得纳入版本控制；生产证书应由部署环境的受控密钥管理方式提供。

## 历史文档

[archive](archive/README.md) 保存历史 PRD、路线图、重构方案和实施总结。归档内容可能包含已被推翻、未经核验或只对应某个历史版本的结论，不得作为新代码的业务依据。

## 文档状态

- **权威业务规范**：当前目标业务的唯一依据；修改时必须核对来源并同步领域语言。
- **有效 ADR**：已经接受的技术或架构决策；涉及业务前提时仍服从权威业务规范。
- **待复核**：存在未经核验的法律、业务或实现前提，不得继续扩展。
- **已废止**：结论不再有效，仅为历史追溯保留。
- **运行手册**：描述部署、测试或运维操作，不定义业务规则。
- **历史归档**：只用于追溯当时讨论和实现。

## 维护规则

1. 一个业务主题只保留一份当前权威规范；新结论直接更新该规范，不再新增“修正版”“最终版”文件。
2. 业务规范只写目标业务和证据边界；当前代码差距写入对应实施计划。
3. 实施结果写入提交记录、测试和发布记录，不回填为业务真相。
4. 法律、本地规则、合同、项目材料和口述结论必须区分来源与确定性。
5. 无法确认的内容列入“待核验事实”，不得用默认配置或示范文本代替。
6. ADR 发现错误前提时标记为待复核或已废止，不继续沿用错误结论。
7. 历史文档移入归档后，不再修补其中的业务结论；需要复用的事实必须重新核验后写入权威规范。
8. SPEC 只服务一个当前实施切片；需求必须引用权威业务规范，完成后保留为变更追溯材料，不得升级为第二份业务真相。
