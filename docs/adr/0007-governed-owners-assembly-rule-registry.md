# 业主大会议事规则采用原件约束的治理配置

> 业务关联：业主大会筹备、方案公示、会议通知、选票送达、表决计票和结果公告所依据的议事规则版本管理。
>
> Status: accepted
> Date: 2026-07-18

## Context

各小区实际生效的《业主大会议事规则》可能对会议形式、公示和通知期限、有效送达、未表态处理、委托代理、计票及结果公告作出不同选择。上海市示范文本和平台默认值不能替代江湾国际公寓实际备案并在会议发生时有效的规则。

系统既需要保存具有证明力的规则原件，也需要使用结构化数据驱动状态机和校验。仅保存文件会迫使运行时反复解析且无法稳定执行；仅保存可编辑配置则无法证明配置来自哪份规则。扫描件质量、版式和条款表达也决定了 OCR 或大模型解析不能直接产生具有业务效力的配置。

## Decision

### 1. 原件、结构化配置和确认记录同时保存

每个议事规则版本由三类不可分割的记录组成：

1. **规则原件**：原始文件、文件摘要、备案信息、生效日期和来源；
2. **结构化规则配置**：允许的会议形式、公示和通知期限、有效送达方式、未表态规则、代理规则、纸质/线上表决约束、重复投票处理、计票与结果公告要求；
3. **确认与审计记录**：录入人、逐项核对人、确认角色、确认时间和变更原因。

结构化字段应尽可能记录对应原件页码和条款位置，使配置结论能够回查原文。

### 2. 上传原件是必需业务步骤，自动解析不是

规则启用流程固定为：

`上传规则原件 -> 人工填写或系统辅助解析 -> 主任/副主任逐项核对 -> 启用规则版本`

- 系统可以使用 OCR 或模型解析预填结构化字段，但解析结果只能是待核对草稿，不能自动启用；
- 系统必须支持完全人工录入，扫描件无法识别时不得阻断规则登记；
- 运行中的业务流程只读取已确认的结构化配置，不在公示、投票或计票时重新解析原件；
- 只有未来接入可信政府平台提供的带签名结构化规则数据时，才可以另行评估免上传原件，当前不采用该例外。

### 3. 启用权限与版本生命周期

- 物业经理或业委会经办人员可以代为上传原件、填写配置草稿；
- 业委会主任或副主任负责逐项核对并确认启用；
- 属地核验或备案信息如需补充，应作为独立核验记录保存；
- 物业服务企业不能单独使议事规则生效。

规则版本生命周期为 `DRAFT -> PENDING_CONFIRMATION -> ACTIVE -> SUPERSEDED`。`ACTIVE` 版本不可修改；规则变化必须新建版本并重新确认，不覆盖历史版本。

### 4. 每次业主大会冻结规则快照

业委会形成召开决定并进入正式办理、开始方案公示前，后端必须从当时有效的 `ACTIVE` 版本生成不可变的 `OwnersAssemblyRuleSnapshot`。后续规则版本变化不影响在办或历史会议。

没有有效议事规则版本时，可以保存内部筹备草稿，但不得确认正式召开、发布方案公示或会议通知、制作/送达选票、开启表决及计票。系统不得用平台默认天数或默认未表态处理代替缺失规则。

## Consequences

- 后端以已确认配置和会议快照作为唯一运行时规则来源，同时保留原件用于审计和争议复核；
- 管理端必须区分“录入草稿”和“确认启用”，不能把文件上传成功显示为规则已生效；
- C 端只展示会议快照对应的公开规则摘要和原件，不直接读取可变配置；
- 解析器可以独立演进或被关闭，不会改变既有会议的规则和计票结果；
- 江湾实际备案规则尚未取得前，只能实现规则登记能力和前期筹备，不能据示范文本补造江湾生效配置。

### 当前可执行能力边界

本次实现只把已建模的书面征求意见和纸质选票流程接入规则快照：方案公示期限、逐户送达方式、未表态处理、纸质选票用印和一般/重大事项的精确计票阈值均从本次快照读取。

若已确认规则要求独立的会议通知期限、结果公告期限、线下会议签到/代理、线上或纸电混合表决，系统必须在进入正式办理前明确阻断。该阻断不表示这些规则不应适用，而是防止系统在缺少独立状态、原始材料和身份/重复票证据链时伪造“已办理”。补齐相应状态机和证据模型后，才能解除对应能力闸门。

## Considered Alternatives

- **只上传文件、运行时自动解析**：拒绝。解析结果不稳定，无法保证同一会议多次执行得到相同规则。
- **只保存人工配置、不保存原件**：拒绝。无法证明字段来源，也无法在争议中回查原文。
- **解析成功后自动启用**：拒绝。把技术识别结果错误提升为具有治理效力的决定。
- **使用平台默认规则兜底**：拒绝。示范文本和历史办理习惯不能代替本小区实际备案规则。
