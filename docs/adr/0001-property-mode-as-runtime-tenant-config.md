# 物业管理方式作为租户级运行时配置

> 业务关联：同一套 SaaS 在不同物业管理区域按各自真实物业服务合同和账户安排运行。
>
> Status: accepted
>
> Date: 2026-06-19
>
> Revised: 2026-07-19

## Context

不同物业管理区域可能采用不同的物业服务计费方式、资金管理安排和信息公开要求。这些差异属于租户业务配置，不应通过为每种方式复制一套数据库和服务来表达。

本 ADR 只决定技术承载方式，不确认平台固定存在几种物业管理方式，也不确认任何方式对应的资金权限、签署流程、公开范围或切换门槛。上述业务内容必须来自真实物业服务合同、业主共同决定和适用规定。

## Decision

物业管理方式作为租户级运行时配置保存，并满足以下约束：

1. 当前配置引用一个有来源、有生效时间的业务配置版本；
2. 历史版本只追加、不覆盖，业务记录保存其发生时适用的版本；
3. 差异化行为通过 `PropertyModeStrategy` 等明确接口分派，不在控制器、Mapper 或页面中散落条件判断；
4. 未经核验的方式名称和业务规则不得仅因枚举已存在就自动启用；
5. 方式切换必须由独立业务流程验证决定依据、生效时间、在途事项和历史数据边界。

## Consequences

- 单库单服务可以复用身份、组织、审计等共同能力；
- 新增或修正一种方式时，必须补齐对应策略和契约测试；
- 数据可见性、资金动作和授权仍由后端依据真实配置校验，不能只靠菜单隐藏；
- 旧代码中的枚举名称属于实现遗留，不自动成为面向用户的业务术语。

## Considered Alternatives

- **每种方式物理分库并独立部署**：拒绝。会复制共同能力，并把业务方式变更扩大为跨库迁移；只有在监管、合同或安全隔离提出明确要求时才重新评估。
- **在各业务服务内直接判断枚举**：拒绝。规则会分散且难以验证，容易让历史记录随当前配置改变。
- **只在前端控制差异**：拒绝。身份、资金、权限和审计必须由后端执行。

## Non-decisions

本 ADR 不决定：

- 本小区当前采用何种物业服务计费或资金管理方式；
- 平台应向业务人员展示哪些方式名称；
- 方式切换由谁决定、适用何种表决阈值和公示期限；
- 是否采用第三方托管、电子签章或外部存证服务。
