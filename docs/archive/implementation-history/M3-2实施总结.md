## 1. 目标 & 边界回顾

M2-2 / M2-3 期间投票引擎（`AbstractVotingEngine` + 3 个具体引擎 + 党员比例豁免）和 `settle` 链路已完整落地，但**投票业务流的两端都没接通**：没有"发起议题"endpoint、没有"业主投票提交"endpoint、没有"我的议题"列表。现有 `ElectionWorkflowIntegrationTest` 只能靠 `JdbcTemplate.update("INSERT INTO t_voting_subject ...")` 才能跑通，前端永远没法接。M3-2 把这两端补齐。

| 维度 | 本期落地 |
|---|---|
| ✅ 议题生命周期 endpoint | 立项 / 公示 / 撤回 / 管理端详情 / "我的议题"列表 |
| ✅ 业主投票提交 endpoint | L3 face-auth + ABAC scope 校验 + 防重 |
| ✅ Scheduler 自动 PUBLISHED → VOTING | 按 `vote_start_at`；现有 VOTING → CLOSED → SETTLED 不动 |
| ❌ 党组书记前置审查 / HANDOVER_LOCK 换届熔断 | 推迟 M4 |
| ❌ ELECTION 类型（候选人申报 + 选举投票） | 立项 endpoint 直接 reject → `SUBJECT_TYPE_NOT_SUPPORTED`，推迟 M3-3 |
| ❌ MQ 千人千面定向推送 | 推迟 M4 |
| ❌ 投票期间实时票数曝光 | 业主端仅看"已投/未投"，防从众；SETTLED 后才暴露结果 |

### 已锁定的设计决策

1. **立项发起者按 SubjectType 分级授权**：ELECTION 本期直接 reject；MAJOR/GENERAL 复用 `voting:subject:create`，并把该能力追加授予 `PROPERTY_MANAGER`（承接"日常资金开支议案"）。
2. **PUBLISHED → VOTING = Scheduler 自动**：复用 `VotingDeadlineScheduler` 设计模式，新增 `VotingOpenScheduler` 监听 `vote_start_at` 到达 + `status=PUBLISHED` 的议题。
3. **议题撤回分阶段授权**：`DRAFT` 发起者本人可撤；`PUBLISHED` 仅街道办（`voting:subject:cancel`）强撤并附 `cancel_reason`；`VOTING/CLOSED/SETTLED` 物理不可撤。
4. **opid 前端显式传**：service 校验 `opid` 属于当前 `uid` + 在 `subject.scope` 范围内。
5. **重复投票 = 409 + `VOTE_ALREADY_CAST`**：DB UNIQUE 冲突翻译为业务异常。

## 2. 落地清单

### 2.1 V3.0 schema（`db/migration/V3.0__voting_lifecycle.sql`）

```sql
-- 1. 议题主表加 4 个审计字段 + 放开 CANCELLED 状态
ALTER TABLE t_voting_subject
    ADD COLUMN proposed_by_user_id BIGINT,
    ADD COLUMN cancelled_at TIMESTAMP,
    ADD COLUMN cancelled_by_user_id BIGINT,
    ADD COLUMN cancel_reason VARCHAR(500);
ALTER TABLE t_voting_subject DROP CONSTRAINT chk_subject_status;
ALTER TABLE t_voting_subject
    ADD CONSTRAINT chk_subject_status CHECK (status IN (1, 2, 3, 4, 5, 6));  -- 6=CANCELLED

-- 2. trigger 12：CANCELLED ↔ cancel_* 三件套一致性兜底（双向）
--    status=6 必须同时落 cancelled_at + cancelled_by_user_id + cancel_reason；
--    status<>6 不应携带任何 cancel_* 字段。

-- 3. 新增能力点：voting:subject:cancel（G 端街道办，redline=1）/ voting:subject:audit（GB 端）
-- 4. voting:subject:create / publish 的 allowed_dept_categories 由 'GB' 扩到 'GBS'，并授予 PROPERTY_MANAGER(role 10)
-- 5. 索引 idx_voting_subject_status_start(status, vote_start_at) 供 scheduler 扫描
```

> **trigger 编号**：V1..V2.x 已用 1-11；M3-2 起占用 **12**。
> **C_USER 降级**：业主投票（`voting:subject:cast`）走 c_user 链路，无 sys_role 关联，endpoint 用 `isAuthenticated()` + service 层强校验（沿用 M2-3 disclosure 降级方案）。

### 2.2 Domain 层（`pangu-domain/.../model/voting/`，纯 Java 无 Spring）

- **`SubjectStatus`** —— 加 `CANCELLED(6)`。
- **`VotingSubject`** —— 加 `proposedByUserId / cancelledAt / cancelledByUserId / cancelReason` 4 字段。
- **新增 `VotingSubjectActions`** —— 聚合根行为，状态机入口（`EnumMap<SubjectStatus, EnumSet<SubjectStatus>>` 转移表）：
  - `open(...)` 工厂，status=DRAFT；`Objects.requireNonNull` 校验关键入参，`IllegalArgumentException` 校验标题空白 / `voteEndAt` 不晚于 `voteStartAt` / BUILDING 缺 `scopeReferenceId` / UNIT 暂未实现。
  - `publish(subject)`：`DRAFT → PUBLISHED`。
  - `openVoting(subject, now)`：scheduler 调用，`PUBLISHED → VOTING`，要求 `now ≥ voteStartAt`，否则 `IllegalSubjectTransitionException`。
  - `cancelByProposer(subject, currentUserId, reason, now)`：仅 `DRAFT` + 本人。
  - `cancelByGovernment(subject, currentUserId, reason, now)`：仅 `PUBLISHED`。
  - `applyCancellation`：reason 非空白且 ≤500 字。
- **新增 `OwnerPropertyVotingView`**（`model/asset/`）—— 投票资格只读视图：`opid / uid / tenantId / buildingId / buildArea / votingDelegate / accountStatus`；`isValidForVoting()` = `votingDelegate && accountStatus==1`。
- **`VotingSubjectRepository`** 加 `insert / updateStatus / cancel / findByIdForUpdate / findPublishedReadyForOpen`；**`VoteItemRepository`** 加 `insert`（内部静态 `DuplicateVoteException`）；新增 **`OwnerPropertyVotingRepository`** port。

### 2.3 Application 层（`pangu-application/.../voting/`）

- **`ProposalLifecycleService`**（6 use case，`@Transactional`）：
  - `propose` —— ELECTION 直接 `SUBJECT_TYPE_NOT_SUPPORTED`；`VotingSubjectActions.open` 抛 `IllegalArgumentException` 翻成 `PROPOSE_FORBIDDEN_FOR_TYPE` → `INSERT(status=DRAFT)`。
  - `publish` —— `findByIdForUpdate` not-found → `SUBJECT_NOT_FOUND`；非 DRAFT → `SUBJECT_NOT_DRAFT`；`updateStatus` 返回 0（乐观锁失败）→ `CONCURRENT_LIFECYCLE_MODIFICATION`。
  - `cancel` —— `byGovernment` 分流 `cancelByGovernment` / `cancelByProposer`；状态或身份不符统一翻 `CANCEL_FORBIDDEN`。
  - `openVoting` —— scheduler 入口，非 PUBLISHED → `SUBJECT_NOT_PUBLISHED`。
  - `findVisibleForOwner` —— ABAC 过滤"我的议题"。
- **`VoteSubmissionService.cast`** —— 业主投票主链路闸门：① subject 存在且 `status=VOTING`（否则 `SUBJECT_NOT_VOTING_CASTABLE`）；② tenantId 不一致 → `OPID_OUT_OF_SCOPE`；③ ELECTION → `SUBJECT_TYPE_NOT_SUPPORTED`；④ MAJOR 议题调 `AbacPolicyEngine.evaluateVoting(uid, tenantId, authLevel)`，不足 → `AUTH_LEVEL_INSUFFICIENT`；⑤ `opid` 不存在或非本人 → `OPID_NOT_OWNED`；⑥ 非投票代表 / 账户冻结 / BUILDING scope 不匹配 → `OPID_OUT_OF_SCOPE`；⑦ `INSERT t_vote_item` 捕获 `DuplicateVoteException` → `VOTE_ALREADY_CAST`；⑧ 仅返回 `vote_id`，不返回当前票数。
- **`command/`**：`ProposeSubjectCommand` / `PublishSubjectCommand` / `CancelSubjectCommand` / `CastVoteCommand`。
- **`VotingApplicationException.Reason`** 加 9 个：`PROPOSE_FORBIDDEN_FOR_TYPE` / `SUBJECT_NOT_DRAFT` / `SUBJECT_NOT_PUBLISHED` / `SUBJECT_NOT_VOTING_CASTABLE` / `OPID_NOT_OWNED` / `OPID_OUT_OF_SCOPE` / `VOTE_ALREADY_CAST` / `AUTH_LEVEL_INSUFFICIENT` / `CANCEL_FORBIDDEN` / `CONCURRENT_LIFECYCLE_MODIFICATION`。

## 5. 已识别的剩余风险

1. **C_USER 权限链路降级**：`voting:subject:cast` 对 c_user 无 sys_role 体系，endpoint 用 `isAuthenticated()`，安全边界完全压在 service 层（tenantId + uid + L3 + scope + opid 归属）。
2. **L3 face-auth 仅消费不实现**：M3-2 只读 `UserContext.authLevel`，不实现刷脸升级流程。生产业主登录默认 L1，遇 MAJOR 议题需前端先调"刷脸升 L3"接口（M4）；测试 fixture 直接构造 L3 上下文。
3. **scope=BUILDING 仅做立项 + 投票提交 + 列表**，不含 BUILDING 范围 settle 测试（沿用 COMMUNITY scope 的 `ElectionWorkflowIntegrationTest` 验 settle 链路）。
4. **Scheduler 单实例假设**：当前无分布式调度锁，沿用单实例 + 乐观锁（`UPDATE ... WHERE status=2 AND version=?`）兜底；分布式部署后续 M4 引入 Redisson scheduler 锁。
5. **`proposed_by_user_id` 不加 FK**：避免跨 tenant 的 user 删除引发删议题级联；仅审计时 join。旧记录 `cancel_*` 全为 NULL，trigger 12 仅在 INSERT/UPDATE 触发，不影响历史数据。
