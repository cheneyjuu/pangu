## 1. 目标 & 边界回顾

M2-4 把 SaaS 后台运维所需的 **「动态新建 / 删除角色 + 角色-权限授予」** 三个 endpoint 从空白补齐，覆盖 V1.4 已经定义但只有 schema 没有 endpoint 的 `admin:role:manage` 通路。M2-5 把 M2 路线图明确登记的三件清理一次性收口：

| 路线图 | 落地内容 |
|---|---|
| §5.2 SaaS 角色管理 endpoint | 3 个 REST endpoint + RoleAdmin domain port / app service / web controller / 6 个 ErrorCode |
| §6.1 反向依赖移除（infrastructure → application） | `pangu-infrastructure` 不再借道 application 拿 domain；改为显式依赖 domain |
| §6.2 prod profile mock 兜底 物理移除 + 回归保护 | application-prod.yml + ProductionEmptyContextTest 锁住 DataScopeInterceptor 的拒绝路径 |
| §6.3 voting deadline scheduler 落到 bootstrap | 已在前序提交内完成（VotingDeadlineScheduler in `pangu-bootstrap/.../scheduler/`）—— 本期无追加改动 |

**两条边界**：
1. **不暴露 update 角色** —— `RoleAdminApplicationService` 刻意不开 `updateRole`。角色的 `allowed_dept_category` / `fixed_data_scope` 一旦落库，会被历史 `effective_data_scope` 数据继承，运行期改它会导致权限链路语义飘移。需要变更走运维 Flyway 修订。
2. **PR base = `m3/dispute-pipeline`** —— `main` 尚未合入 m3 PR #6，但 schema_history 在共享数据库里已经包含 V2.8 / V2.8.1，本分支需要这两份 schema 才能跑通 Flyway validate。所以 PR 链路是 m3 → m2-4-and-cleanup（stacked PR），而不是直接挂 main。

## 2. 落地清单

### 2.1 V2.9 schema（`pangu-bootstrap/src/main/resources/db/migration/V2.9__role_admin_permission.sql`）

```sql
INSERT INTO sys_permission (permission_key, description, permission_group,
                            allowed_dept_categories, is_legal_redline) VALUES
    ('admin:role:manage', 'SaaS 管理员动态新建 / 删除角色 + 角色-权限分配',
     'ADMIN', 'G', 0);

-- 仅授予 GOV_SUPER_ADMIN（role_id=1, fixed_data_scope=ALL_COMMUNITY）；
-- 满足 trigger 6（role.allowed_dept_category='G' 出现在 permission.allowed_dept_categories='G'）
INSERT INTO sys_role_permission (role_id, permission_key) VALUES
    (1, 'admin:role:manage');
```

`is_legal_redline=0` 的取舍：V1.4 已经存在 redline=1 的 `admin:role:write`，已经强制锁定到 fixed_data_scope NOT NULL 的 G 端角色；`admin:role:manage` 仅作为 SaaS 平台管理员动态调试入口，再上 redline 等于双重套娃，没有额外收益。

### 2.2 Domain 层（`pangu-domain/.../model/role/` 与 `repository/`）

- **`SysRole` record** —— 7 字段（roleId / roleKey / roleName / allowedDeptCategory / fixedDataScope / defaultDataScope / isSystem）；`forCreate(...)` 工厂强制 isSystem=0；`withId(roleId)` 持久化后回填主键。**故意不做聚合根**：纯 CRUD，无状态机、无领域事件，端口够用。
- **`SysRoleRepository`** port —— `findById` / `findByRoleKey` / `insert` / `delete`（不暴露 update，原因见 §1）；内部静态异常 `DuplicateRoleKeyException` / `SystemRoleProtectedException`。
- **`SysRolePermissionRepository`** port —— `assign` / `revoke` / `countByRole`；内部静态异常 `DuplicateAssignmentException` / `AssignmentConsistencyException`。

### 2.3 Application 层（`pangu-application/.../admin/`）

- **`RoleAdminApplicationService`** 三条 use case，全 `@Transactional`：
  - `createRole(CreateRoleCommand)` —— 应用层入参校验（dept ∈ {G,B,S} / scope ∈ {ALL_COMMUNITY, OWNER_GROUP, ORG_ONLY} / fixed=null OR fixed=default）→ `roleRepository.insert(...)` → DuplicateRoleKeyException 翻成 `ROLE_KEY_DUPLICATE`。
  - `assignPermission(AssignPermissionCommand)` —— 先 `findById` 拦截不存在角色（明确 `ROLE_NOT_FOUND` 而非 trigger 6 的笼统报错）→ `rolePermissionRepository.assign(...)` → `DuplicateAssignmentException` → `PERMISSION_ALREADY_ASSIGNED`；`AssignmentConsistencyException` → `PERMISSION_ASSIGNMENT_INCONSISTENT`。**故意不做端归属 / redline 二次校验**——全部沉到 trigger 6 兜底，避免应用层 / DB 双校验语义飘移。
  - `deleteRole(roleId)` —— `roleRepository.delete(...)` → `SystemRoleProtectedException`（trigger 7）翻成 `ROLE_PROTECTED`；affected=0 翻成 `ROLE_NOT_FOUND`。
- **`RoleAdminApplicationException` + `Reason` 枚举 6 种**：`ROLE_PARAM_INVALID` / `ROLE_NOT_FOUND` / `ROLE_KEY_DUPLICATE` / `PERMISSION_ALREADY_ASSIGNED` / `PERMISSION_ASSIGNMENT_INCONSISTENT` / `ROLE_PROTECTED`。仿 `GovernanceLockApplicationException` 双构造器。
- **`command/{CreateRoleCommand,AssignPermissionCommand}`** records。

### 2.4 Infrastructure 层（`pangu-infrastructure/.../persistence/` 与 `repository/`）

- `SysRoleRow` / `SysRoleMapper` + XML / `SysRolePermissionMapper` + XML 标准三件套。`insert` 用 `useGeneratedKeys="true" keyProperty="roleId"` 回填主键。
- `SysRoleRepositoryImpl` / `SysRolePermissionRepositoryImpl` 适配；**关键加固**：trigger 6 / trigger 7 走 PG `RAISE EXCEPTION`（默认 SQLSTATE=P0001），Spring 把它翻成 `UncategorizedSQLException` 而非 `DataIntegrityViolationException`，所以两个 RepositoryImpl 一律 catch 父类 `DataAccessException` 后 walk-cause-chain 找 `[trigger N]` 标记识别业务语义。

### 2.5 Interfaces 层（`pangu-interfaces/.../web/`）

| Method | Path | @PreAuthorize | 用途 |
|---|---|---|---|
| POST   | `/api/v1/admin/roles` | `hasAuthority('admin:role:manage')` | 创建非系统角色 → 201 |
| POST   | `/api/v1/admin/roles/{roleId}/permissions` | `hasAuthority('admin:role:manage')` | 授予 permission → 200 |
| DELETE | `/api/v1/admin/roles/{roleId}` | `hasAuthority('admin:role:manage')` | 删除非系统角色 → 200 |

- 控制器走 `BaseController` 标准返回（`success("...", data)`）；`requireUserId()` 通过 `SecurityUtils.getUserId()` 拿当前 user，**不直接读 SecurityContextHolder**（沿用 M1 commit 142300a 的约定）。
- DTO：`CreateRoleRequest` / `AssignPermissionRequest` / `RoleResponse`，Bean Validation 兜底必填 / Pattern。
- `RoleAdminErrorCode`：六段位编排——421xx 入参 / 422xx 业务冲突 / 423xx 权限保护。
- `GlobalExceptionHandler` 增 `@ExceptionHandler(RoleAdminApplicationException.class)`，复用 `RoleAdminExceptionTranslator.translate(ex)`。

## 3. trigger 翻译加固（Spring DataAccessException 父类捕获）

> **坑**：trigger 6 / trigger 7 在 PG 端是 `RAISE EXCEPTION '[trigger N] ...'`，默认 SQLSTATE=P0001（raise_exception）。Spring 的 `SQLErrorCodeSQLExceptionTranslator` 默认规则把 P0001 归到 `UncategorizedSQLException` 而非 `DataIntegrityViolationException`。

最初 `SysRoleRepositoryImpl.delete` 只 catch `DataIntegrityViolationException`，结果 trigger 7 异常没被识别，HTTP 500 泄漏到客户端。修复：

```java
// 之前
} catch (DataIntegrityViolationException e) {
    String msg = e.getMostSpecificCause() == null ? "" : e.getMostSpecificCause().getMessage();
    if (msg != null && msg.contains("[trigger 7]")) { ... }
}

// 之后
} catch (DataAccessException e) {
    if (causeMessageContains(e, "[trigger 7]")) { ... }
}
```

`SysRolePermissionRepositoryImpl.assign` 也做了同样的父类拓宽。后续如有任何 `RAISE EXCEPTION` trigger 都按这个模式写。

## 5. 后续衔接 / 已识别风险

1. **`admin:role:manage` 与 GovernanceLockController.lock() 共享 capability** —— V2.5 当时把 `admin:role:manage` 用作 placeholder permission，本期 V2.9 才正式落库。验证：`GovernanceLockController.lock()` 在 V2.5 已经把这条 permission 写进 `@PreAuthorize`，但当时 schema 未挂；本期 V2.9 一并补齐 schema 后，那条路径自然生效，无需改 controller 代码。
2. **m3/dispute-pipeline 的 V2.8 / V2.8.1 暂寄存于本分支** —— 因为共享数据库的 schema_history 已经包含这两份 record，本分支必须包含对应文件才能 Flyway validate 通过。**rebase 到 m3/dispute-pipeline 之上之后**，这两份文件来源于上游，本分支 PR diff 自然不含它们。
3. **角色端归属变更 / 历史 effective_data_scope 一致性** —— 本期不开 update endpoint，避开了这个风险。后续 M3-x 如果需要支持 SaaS 后台变更角色 fixed_data_scope，必须配套审计 + 重算 effective_data_scope 的迁移脚本。
4. **`assignPermission` 的两步检查与并发** —— 当前先 `findById` 再 `assign`，两步之间如果其他事务 DELETE 了角色，会从 `ROLE_NOT_FOUND` 退化到 `PERMISSION_ASSIGNMENT_INCONSISTENT`（FK 不存在）。这是 admin 后台串行操作场景，且都返 4xx，不影响安全；如果未来高并发，再加 `findByIdForUpdate`。
