# 管理台后端差距分析与 M4 路线图

> 归档声明：本文基于历史页面和代码快照，不能作为当前能力或业务规范。

> 状态：路线图（research / 规划），非实现。
> 输入：`yaochi/生成页面/`（Figma 导出的 React + Vite + shadcn 管理台，22 个页面，当前 100% mock 数据）。
> 目标：把"前端已设计页面"逐一对齐"后端已有能力"，标注三态（已有 / 缺查询接口 / 缺领域），沉淀跨切面缺口，给出 M4 系列里程碑顺序。

---

## 0. 一句话结论

后端目前是一台**「写动作引擎」**——立项、投票、审查、签名、熔断、纠纷流转都齐备，但**几乎没有面向管理台的列表 / 分页 / 聚合查询接口**，且**有 9 个前端模块在后端完全没有领域模型**（业委会/任期、物业模式、资产/工单/工程、公告/推送、看板聚合）。

因此 M4 的主线不是"造新功能"，而是**两条腿**：
1. **给已有领域补查询接口**（低风险、高回报，先做）；
2. **补齐缺失领域**（业委会任期、物业模式、资产工单），按治理闭环优先级排序。

## 2. 跨切面缺口（贯穿所有模块，必须先定调）

### 2.1 缺读侧 / 列表 / 聚合接口（最高优先级）
后端几乎全是 `POST` 写动作，管理台需要的 `GET 列表 + 分页 + 筛选 + 详情聚合` 普遍缺失。即使是「已有」模块（Rbac/DualSign/FundReview），也缺列表查询。
**对策**：M4 第一阶段统一补**读侧查询接口**，配合 `DataScopeInterceptor` 做行级权限收口。

### 2.2 前端 8 角色 ↔ 后端 13 角色 映射缺口
| 前端 RoleId | 对应后端 role_key（建议映射） |
|---|---|
| street_admin | GOV_SUPER_ADMIN(1) |
| party_secretary | PARTY_SECRETARY(3) |
| committee_director | COMMITTEE_DIRECTOR(5) |
| committee_member | COMMITTEE_MEMBER(6) |
| building_rep | GRID_OPERATOR(4) + OWNER_REPRESENTATIVE(8)（前端合并了"楼栋代表/网格员"） |
| property_manager | PROPERTY_MANAGER(10) |
| property_service | PROPERTY_STAFF(11) |
| auditor | （后端**无**审计师角色）→ 需新增 AUDITOR |

未被前端覆盖的后端角色：COMMUNITY_ADMIN(2)、COMMITTEE_SECRETARY(7)、VOLUNTEER(9)、SERVICE_PROVIDER_MANAGER(12)、SERVICE_PROVIDER_STAFF(13)。
**对策**：定义双向映射表，并为 auditor 新增系统角色 + 财务只读 DataScope。

### 2.3 G 侧跨小区聚合 ↔ 后端单租户 ThreadLocal
前端 street_admin 的数据域是「辖区全部小区」（`communityId="ALL"` 跨小区汇总），而后端 `TenantContext` 是单租户 `ThreadLocal<Long>`。街道办看板 / 监管列表需要跨租户聚合，现有上下文模型不支持。
**对策**：M4 需引入「辖区聚合查询」路径（多租户 union 或 G 侧专用聚合服务），不能复用单租户拦截器。

### 2.4 物业模式（PropertyMode）建模缺口
package（包干制）/ reward（酬金制）/ trust（信托制）驱动财务行为分叉。后端仅 trust 有体现（治理锁/双签），reward 的开支单状态机、package 的固定包干，均未建模。
**对策**：PropertyMgmt + ExpenseApproval 必须先确定物业模式领域，否则财务模块前端无法落地。

### 2.5 登录身份契约：不选角色、不选小区（已定调，勿回退）

> 状态：**本期（登录 + 搭框架）已落地**。后端 `AuthService.login` 已 +1 行暴露 `role_key`；前端 yaochi 已去掉原 mock 登录页的「自选角色 / 选小区」两个下拉，改为「手机号 + 短信验证码」真实登录。本节固化此决策的业务依据，避免后续联调时把下拉加回来。

**结论：管理台（B/G 端）登录时既不选角色，也不选小区——两者都是机构事实，由后端身份绑定派生，不是用户可挑的偏好。**

1. **角色不能选**：角色是任命/选举出来的身份（周主任就是和畅雅苑业委会主任），不是登录偏好。让用户自选角色 = 谁都能勾「街道办超管」的越权黑洞。后端 `login()` 由 `手机号 → t_account → last_active_identity → role_key` 派生，前端只拿 `role_key` 映射菜单可见性（见 2.2）。

2. **小区不在登录时选**，分两类身份：
   - **B/G 端机构身份（SYS_USER）**：`tenant_id` 任命时即固定——业委会成员绑定唯一小区、物业绑定唯一项目、街道办/党组织管辖一个辖区（=「ALL 汇总」数据范围，由角色直接授予，非登录时挑选）。**yaochi 只涉及这一类。**
   - **C 端业主（C_USER）**：一人在多小区有房才是真正的多小区切换场景——由后端 `switchTenant`（**仅限 C_USER**）处理，且属于**业主端 App**，不在 yaochi 管理台范围。

3. **多身份边界情况**：一个手机号同时是「业委会主任(SYS_USER)」+「业主(C_USER)」很常见。后端处理方式：登录时**不弹身份选择**，直接用 `last_active_identity` 作默认；换身份是**登录后**的动作，走 `switch-identity` 端点——但该端点**计划在 M2 引入，控制器层目前尚未实现**（`AuthService` 类注释已写明）。即便多身份，登录这一步仍是「后端定默认、前端不选」。

4. **topbar「小区切换器」≠ 登录选择**：store 里保留的 `communityId / COMMUNITIES / setCommunityId` 是**登录后的浏览范围筛选**，主要对街道办有意义（看辖区汇总→下钻具体小区）；对单小区机构角色应**钉死在其唯一小区**，不应表现为可切换。

**待办（与 M4 衔接）**：
- M2 `switch-identity` 端点落地后，前端需在 topbar/账户菜单加「切换身份」入口（非登录页）。
- topbar 小区切换器：对非 street_admin 角色应禁用/隐藏，仅 G 侧辖区聚合（见 2.3）时可用。

## 3. M4 里程碑建议顺序

排序原则：**先低风险高回报（补查询），再按治理闭环重要性补领域**；每个里程碑沿用三段提交节奏（feat/test/docs），base=main。

### M4-0：读侧地基（跨切面，最先做）
- 统一分页 / 排序 / 筛选 DTO 约定；列表查询统一接 `DataScopeInterceptor`。
- 角色映射表（8↔13）+ auditor 角色与财务只读 scope。
- **产出**：后续所有列表接口的统一骨架。

### M4-1：已有领域补查询接口（低风险）
> 前置已就绪：**本期已交付前端 API 骨架**（`lib/api.ts` fetch 封装 + Result 信封解析、`lib/auth.ts` 会话/角色映射、vite `/pangu` dev proxy），M4-1 各列表接口前端侧直接复用，无需再造请求层。
- Voting 议题列表（`GET /voting-subjects` 分页/状态筛选）✅ **已交付**（`Page<T>`/`PageResponse<T>` 读侧地基随车落地，全平台列表接口复用 `{items,total,page,size}` 契约）
- Finance 披露列表、DualSign 锁列表、FundReview 申请列表、Rbac 角色/权限列表
- Owners 业主名册查询、Topology 组织树、Certification 认证记录查询
- **产出**：5 个 🟡 模块 + 4 个 ✅ 模块的列表页可对接真实数据。

### M4-2：业委会与任期领域（治理闭环核心）
- `t_committee` 委员实体 + 任期 term 实体 + 交接清单。
- 把「查询派生换届熔断」升级为**基于真实任期实体**的熔断（消除当前靠 ELECTION 状态推断的间接性）。
- 支撑 CommitteeRoster / TermManagement / Duties / MeetingMinutes 4 页。
- **产出**：committee 模块前端可落地；换届熔断从派生转为一等公民。

### M4-3：物业模式 + 酬金制财务
- PropertyMode 领域（package/reward/trust）+ 物业组织/合同。
- 酬金制开支单状态机（物业密码A签 → 主任密码B签）。
- 支撑 PropertyMgmt / ExpenseApproval。

### M4-4：资产 / 工单 / 工程
- 公共资产台账、维修工单状态机、工程项目。
- 支撑 Assets / WorkOrders / Engineering（前端这 3 页合计 1530 行，体量最大）。

### M4-5：沟通与看板聚合（依赖前序数据）
- 公告 / 推送通知领域。
- Overview 角色自适应看板聚合服务（跨模块 KPI + 待办）、Reports 报表导出。
- **放最后**：依赖前面各模块的数据已就位才能聚合。
