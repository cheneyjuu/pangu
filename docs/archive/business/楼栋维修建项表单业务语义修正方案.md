# 楼栋维修建项表单业务语义修正方案

> 归档声明：本文是形成当前规范前的过渡方案，包含已被后续讨论修正的责任、资金和表决结论。当前目标统一以 [维修与业主大会表决业务规范](../../business/维修与业主大会表决业务规范.md) 为准。

更新日期：2026-07-19

状态：实施中。本文记录业务事实、产品边界和后续修改依据；已实现的责任认定、授权提案冻结和最终实施锁定只覆盖本轮已验证路径，未接入的账簿、直接责任履行和动态验收能力不得被当作已完成。

适用范围：`pangu` 后端、`yaochi` 管理端以及后续业主端的维修工程建项、实施方案、报价、决定、施工、验收和结算链路。

## 0. 已确认的业务目标与权责边界

本文的业务目标不以现有代码、现有字段、当前演示项目是否已有账户或能否锁定为前提。维修工程系统的目标是：把一个维修事项从问题发现、责任认定、方案决定到施工、验收和结算形成可追溯的业务链，而不是让操作者以“楼栋维修”名称或一个资金下拉选项替代责任认定。

### 0.1 正确业务主线

1. 登记报修、巡检或计划性维修事项，只记录现象和来源，不在此时确定责任、共有范围或资金来源。
2. 勘验具体设施、位置、服务范围、故障原因和相关证据；形成维修点位与初步措施。
3. 作出责任认定：专有部分、物业服务合同内日常维保、建设单位保修、责任人损坏，或共有部分维修。
4. 对共有部分维修确定共有、决定和费用承担范围；部分共用与全体共用只回答该范围问题，不直接等同于某种资金来源。
5. 形成工程草案、工程量、预算和必要的参考询价；草案阶段不得预占资金、确认中选供应商或生成付款资格。
6. 根据责任认定和有效依据，进入相应的资金与治理路径，例如专项维修资金、合同内物业承担、建设单位保修、责任人赔偿或业主自筹。
7. 需要相关业主决定或业主大会授权的共有维修，先冻结授权提案：固定工程范围、预算、费用承担基数和施工单位选择规则；该冻结不产生中选、合同、付款或资金支取资格。
8. 有效决定、授权或法定例外依据生效后，服务端重新核验责任认定、授权依据和资金账簿，锁定最终实施方案；再按获批方式定商、签约、施工、验收、结算和支取。

### 0.2 配置、工程认定与锁定快照必须分层

| 层级 | 业务内容 | 作用与限制 |
| --- | --- | --- |
| 小区基础配置 | 设施台账及服务范围、物业服务合同责任、保修规则、资金账簿、有效授权规则 | 提供可引用的默认事实和可选路径；不得被当作某一工程已经获得资金或授权的证明。 |
| 工程认定 | 本次工程关联的设施、勘验事实、拟认定的责任路径、共有范围和证据 | 可以引用基础配置，也可以提出例外，但例外必须进入核验，不能由普通表单直接生效。 |
| 授权提案快照 | 决定范围、预算、费用承担基数、施工单位选择规则及其证据版本 | 只供相关业主决定或授权审查；不产生中选、合同、付款或资金支取资格。 |
| 最终实施锁定快照 | 已确认的承担范围、账簿或付款责任、预算上限、有效决定/授权依据及其版本 | 是定商、合同、验收和付款的业务依据；后续修改基础配置不得改写已锁定工程。 |

“设备门是否属于共用部分”优先由设施台账的权属和服务范围说明；“本次维修由谁承担”由勘验事实以及合同、保修、责任认定或共有维修规则共同确定。设备名称、物理位置或流程名称都不能单独产生结论。

### 0.3 已确认的操作权责

1. 物业可以录入问题、维护可核验的设施资料、发起勘验和工程草案，并可提出配置变更或责任认定申请。
2. 涉及资金账簿、合同责任、保修责任、费用承担范围和执行授权的基础配置，由业主委员会/财务人员或经有效授权的主体确认；系统必须保留确认人、依据、版本和生效时间。
3. 服务端只从已确认的基础配置和本工程已核验事实生成锁定快照；前端选择、附件上传或物业单方操作不得绕过确认。

### 0.4 仍待小区事实确认的内容

江湾的具体设备台账、物业服务合同责任边界、保修状态、专项维修资金及公共收益账簿、授权决议和本小区有效规约，仍须以真实来源接入。它们阻止生产项目进入相应的锁定和付款路径，但不改变本节定义的产品目标。

### 0.5 系统必须区分的业务路径

下表是业务路径，不是前端路由、项目类型或资金下拉选项。具体路径只能由工程认定和已确认的基础配置共同确定。

| 业务路径 | 认定前提 | 费用/执行依据 | 工程处理 |
| --- | --- | --- | --- |
| 物业合同内日常维保 | 设施与故障处于物业服务合同约定的维保边界 | 物业服务合同、年度维保计划或服务费用安排 | 可以由工单或小型工程执行；该路径本身不产生业主分摊、专项维修资金支取或业主决定。 |
| 建设单位保修 | 经核验仍在法定或约定保修范围，且责任属于建设单位 | 保修文件、承接查验记录、建设单位确认或委托维修文件 | 记录索赔/委托和施工结果；不得把建设单位应承担的费用转给业主资金。 |
| 责任人损坏或第三方赔偿 | 可验证的人为损坏或第三方责任 | 责任认定、赔偿协议、保险或追偿材料 | 记录先行处置和追偿关系；不得仅因设施共用就默认动用维修资金。 |
| 部分共用维修 | 非前三类责任，且设施只服务一个可确定的部分共有范围 | 相关业主的费用承担与决定依据；符合条件时可使用对应的专项维修资金账簿 | 形成单一部分共用决定范围、承担房屋/面积和资金切片；本轮只支持一个楼栋/门号范围。 |
| 全体共用维修 | 非前三类责任，且设施服务整个物业管理区域 | 全体业主的费用承担与业主大会/授权依据；符合条件时按相应账簿列支 | 形成全体共用决定范围以及按幢/按户的资金切片；不把它误称为“公共收益支付”。 |

公共收益属于独立账簿和授权路径，不是“专项维修资金不足时的替代按钮”。属于专项维修资金使用范围的项目不得以公共收益绕过该路径。

### 0.6 目标流程的验收边界

1. 建立报修或工程草案时，系统允许责任、共有范围和资金路径处于待核验状态；不能因尚未接入账户而拒绝勘验、工程量和预算工作。
2. 系统只有在资金/责任、承担范围和治理依据都已确认后，才允许锁定方案；锁定后才允许基于获批方式产生中选、合同和付款资格。
3. 工程名称、楼栋位置、报修人或设备类别不能自动推导资金来源、费用承担人群、验收角色或定商权。
4. 基础配置变更必须版本化；已锁定项目继续使用自己的快照，后续项目才引用新版本。

## 1. 本轮范围决定

本轮不考虑“将多个独立楼栋的维修事项聚合为一个工程、采购、合同或支取申请”的跨楼栋聚合场景。

本轮必须固定以下边界：

1. 一个 `RepairProject` 只对应一个共有与决定范围。
2. 同一决定范围内，可以关联多个业主报修、物业巡检或专业勘验来源，并形成多个维修点位。
3. 不同楼栋且共有/决定范围不同的事项分别建项；本轮不设计跨项目采购、合同、验收、结算或支取关系。
4. 经证明由物业管理区域全体业主共同使用的设施，本身就是一个“全体共用”决定范围，不属于跨决定范围聚合，仍在本轮范围内。
5. 只服务多个特定楼栋、但不服务全小区的共同设施，法律上可能属于跨楼栋的部分共用范围；本轮不实现该场景。遇到时必须阻止进入正式方案，不得误标为全体共用，也不得擅自拆成无关楼栋项目。

`docs/sample` 中的多楼栋委托、点位汇总和验收汇总材料仍作为原始事实保留，但不驱动本轮领域模型、数据库和界面设计。

## 3. 部分共用与全体共用

### 3.1 概念边界

“部分共用 / 全体共用”回答的是谁共同使用、谁作决定和谁承担费用，不是两种资金池，也不单独决定两条固定步骤流程。

根据[《上海市住宅物业管理规定》](https://gfdy.sh.gov.cn/zwgk/jcgk/zcfg/gz/20230215/f95d85de5cb24a4fbda5d9a24799e0a1.html)和[《上海市商品住宅专项维修资金管理办法》](https://www.shanghai.gov.cn/nw26170/20200820/0001-26170_27363.html)：

- 部分共用费用由共同拥有该部分的相关业主承担，相关事项也由该范围业主作出决定。
- 全体共用费用由物业管理区域全体业主承担。
- 专项维修资金按物业管理区域开户，但按幢立账、按户核算。全体业主共同承担仍要落到各分户账，不是一个模糊的“公共资金”。
- [《上海市住宅小区公共收益管理办法》](https://www.shanghai.gov.cn/gwk/search/content/8b8ad0526ae94db8a6c61b8aa6caf0c2)中的公共收益是独立子账户；属于专项维修资金使用范围的项目不得从公共收益列支。

所以页面不应让用户在“楼栋资金 / 公共资金”之间做一次选择，而应分别确认：

1. 共有与决定范围。
2. 资金来源：专项维修资金、责任人承担、建设单位保修、业主自筹等。
3. 实际费用承担房屋、面积和账簿/分户账切片。

### 3.2 业务如何运行

1. **登记问题来源**：来源可以是业主报修、物业巡检、专业检测、年度计划或紧急事件。业主只报告自己看到的现象，不由报修人选择法律上的共有范围。
2. **勘验并排除其他责任**：先区分专有部分、责任人损坏、建设单位保修和物业日常维保。管道、墙体或设备不能只按物理位置认定，应结合结构、走向、产权和服务范围。
3. **确定单一决定范围**：本轮只允许“一个楼栋/门号的部分共用”或“物业管理区域的全体共用”进入正式建项。无法确认时保持待核验。
4. **确定治理路径**：根据有效的既有授权、相关业主决定、业主大会决定或紧急维修依据进入对应路径。“部分共用”不等于每次都必须新建一轮投票。
5. **形成资金分摊快照**：锁定资金来源、账户/账簿、承担房屋、面积、分摊规则和金额；它与直接受影响房屋、验收参与人分别保存。
6. **形成维修点位与实施方案**：同一决定范围内可有多个点位，每个点位保存位置、现象、原因状态、拟定措施、技术要求和证据。
7. **询价、决定和定商**：先用参考询价/估算形成待决定预算和施工单位选择方式；方案依法生效后，再按已通过的选择方式定商、锁定中选报价和合同。
8. **施工、验收、结算和支取**：按锁定方案保存施工前、进场、关键工序和竣工证据；验收角色和通过条件来自有效规约、决定、授权或锁定方案，不由前端固定人员组合。

两个本轮必须跑通的例子：

- **16-403**：403 室是直接受影响和入户效果确认对象；维修点位是窗户/外墙交界；决定与费用承担范围是 16 号楼，三者不能共用一张“受影响业主”名单。
- **37 号楼大厅玻璃**：物理位置是楼栋公共区域；决定与费用范围是 37 号楼；不应为了通过校验伪造一个受影响私房。

## 4. 当前实现的语义错位

2. 关联已勘验报修时，可将勘验摘要或“业主端提交 1 张现场照片”填入工作内容；证据要求被误当成施工措施。
3. 旧工单 `planBudget` 可被当作当前点位的估算单价，但二者金额口径未确认。
4. 提交时所有项的 `roomId` 固定为 `undefined`，入户或临户共用部位只能依赖自由文本定位。
5. 楼栋流程默认将范围中全部房屋选为“受影响业主”，把决定人群、费用承担人群和验收/效果确认人群混同。
6. 创建时附件为空，施工证据由前端固定生成六个通用必填阶段，没有进入方案版本的点位级证据和材料要求。

当前项目范围只能选一个楼栋/单元或全小区，在本轮“一项目一决定范围”的边界下不是缺陷。真正问题是当前一个范围字段同时替代了物理位置、共有权利、决定人群、资金账簿和验收人群。

### 4.2 后端与数据库

1. `RepairProject.Item`、`RepairPlanDraftCommand.ItemDraft` 和 `t_repair_project_item` 是一层扁平模型，同时承担维修点位、实施措施、报价行、预算行和结算参照行。
2. `RepairProjectService.validateItemAmounts` 强制每项“估算金额 = 数量 × 估算单价”，且项合计必须等于方案预算。点位打包金额、措施费、运费、清运或暂估价无法准确表达。
3. `lockPlan` 在冻结待决定实施方案前强制要求已有中选报价，把“决定方案的预算与选择方式”和“决定生效后定商”合成一个动作。
4. 供应商报价已经有报价头统一税率、不含税金额、税额和含税总额，报价行也能保存描述、规格、品牌和采购方式；后续应复用该方向，不应把含税单价和税率搬回维修点位。
5. 竣工结算又回到“一个工程项对应一个结算项”，且税率由每行提交，与报价头统一税率及样本整表税额不一致。
6. `t_repair_building_process` 虽然是单楼栋决定流程，但没有将共有权利范围、决定基数、资金分摊和验收范围作为不同关系保存。

## 5. 目标业务模型

### 5.1 核心关系

本轮目标关系为：

`RepairProject 1 -> 1 RepairDecisionScope`

`RepairDecisionScope 1 -> N RepairWorkPoint`

`RepairDecisionScope 1 -> N RepairFundingSlice`

`RepairWorkPoint N <-> N RepairSourceReference`

`RepairQuote/Contract/Settlement -> DetailLine -> RepairAmountAllocation -> RepairFundingSlice`

- `RepairDecisionScope`：保存共有性质、共有设施/部位、权利人边界、决定或授权依据快照、计票基数和结果。
- `RepairWorkPoint`：一个可定位、可勘验、可说明现象和拟定措施的维修对象；它不是报价行。
- `RepairFundingSlice`：保存资金来源、账户/账簿、承担房屋、面积、分摊规则和各金额阶段快照。全体共用项目可以因按幢/按户账簿而形成多条切片，但仍属于一个决定范围。
- `RepairSourceReference`：保存报修、巡检、检测或计划来源与点位的关系，不改写原始记录。
- `RepairAmountAllocation`：将措施、运输、清运等不能直接落到单个点位的报价/结算金额分配到资金切片，并保证总额勾稽。

本轮不新增跨决定范围的采购包、合同覆盖、验收包或支取包模型。合同、施工、验收和付款只能覆盖当前项目的单一决定范围。

### 5.2 四种范围不能混同

每个项目必须分别表达：

1. **物理施工范围**：楼栋、单元/门号、参照房屋/公区、空间、朝向、构件和具体部位。
2. **直接受影响与效果确认范围**：遭受漏水、需要入户或需要确认修复效果的房屋/业主；公共设施可以没有私房对象。
3. **共有与决定范围**：依据共有权利、规约、决定和授权生成的决定人群与计票基数。
4. **资金承担与分摊范围**：实际资金来源、账户/账簿、承担房屋、面积、分摊规则和金额。

### 5.3 维修点位

一个 `RepairWorkPoint` 至少包含：

- 系统内部稳定 ID 和排序号，内部 ID 不暴露给录入人编辑。
- 所属项目和单一 `decisionScopeId`。
- 结构化位置：楼栋、单元/门号、参照房间或公区、空间、朝向、构件和具体部位。
- 业务可读名称，例如“16 号楼 403 室北次卧窗户与外墙交界”，不显示 `ITEM-1`。
- 问题现象，只保存可观察事实。
- 原因状态：待勘验、已确认或未能确认；只有可靠勘验/鉴定依据时才保存原因结论。
- 拟定维修措施、技术要求以及施工证据要求。
- 关联的报修/巡检/检测记录和原始附件引用。
- 可选的范围数量/单位和点位暂估金额；暂估金额必须标明来源，不强制数量乘单价。

### 5.4 金额口径

系统必须分别保存：

| 金额 | 含义 |
| --- | --- |
| 初步估算 | 建项/询价前的参考金额，可能来自点位汇总、历史价格或第三方估算 |
| 参考报价/预算依据 | 决定前的询价或估算，不自动表示供应商已中选 |
| 送审/决定预算 | 进入授权、征询或业主大会方案的含税控制金额 |
| 中选报价 | 决定生效后，按已通过选择方式锁定的报价版本 |
| 合同/审价/竣工结算/已支付 | 各自保存原始文件、有效版本、金额、税额和勾稽关系，不互相覆盖 |

报价和结算的税率、税额、不含税金额与含税总额是单据头事实。材料、人工、措施、运输、清运等作为报价/结算明细，不继续写入维修点位。

## 6. 表单和流程修正

### 6.1 建项、决定与定商分阶段

**阶段 A：创建单决定范围的工程草稿**

1. 选择一个楼栋/门号的部分共用场景，或一个全体共用场景。
2. 后端核验所选报修/巡检记录属于同一决定范围；不同范围不得合并。
3. 创建决定范围和维修点位，引用原始勘验与附件；费用承担房屋、面积和金额只能在责任认定后进入授权提案或最终实施锁定快照。
4. 录入现象、原因状态、拟定措施、技术要求、可选初步估算及来源。
5. 提交动作命名为“创建草稿并进入询价”，不宣称方案已生效或供应商已中选。

**阶段 B：冻结授权提案并完成决定/授权**

1. 收集参考询价、报价明细、技术/商务条件和签章原件，作为决定预算依据。
2. 冻结工程范围、决定预算、分摊范围/面积、施工单位选择方式、施工管理、验收和实施时间。
3. 以该不可变提案发起相关业主决定、业主大会授权或其他依法有效的授权程序；未通过时保留提案和结果，回到草案修订。
4. 资金路径在提案中已明确，但专项维修资金账簿余额、公共收益余额或业主自筹托管记录尚未接入时，不得阻断提案冻结；这些事实只在最终实施锁定和实际支取前由服务端重新核验。

**阶段 C：锁定最终实施方案并定商执行**

1. 服务端重新校验责任认定、费用承担范围、可信账簿和有效决定/授权，形成最终实施方案锁定。
2. 按已通过的施工单位选择方式完成比价、招标或其他有效定商程序，再锁定中选报价、合同范围和施工/验收标准。
3. 中选结果扩大范围、改变实质方案或超出有效预算时，生成新方案并判断是否重新冻结授权提案和重新决定，不覆盖已生效版本。

### 6.1.1 物业端可见流程与业务用语

2026 年上海现行业务口径进一步明确：超出业主大会既有授权范围的共有部分维修，应先编制并表决实施方案，再依据表决通过的实施方案确定施工单位、签订施工合同。实施方案本身应当包含预算、分摊范围和面积、施工单位选择方式、施工管理、验收及实施时间。因此，决定前收集的报价只能作为完善实施方案和确定预算的依据，不能在页面上与决定后的施工单位确定合并成一个连续的“定商流程”。

物业端按以下顺序组织办理：

1. **判断责任和费用来源**：物业根据勘验、物业服务合同、保修文件和责任材料填写初步意见，由业主委员会核对。页面询问“本次维修由谁负责”和“拟使用的费用来源”，不让物业选择抽象的“责任路径”或“资金承担路径”。
2. **完善实施方案和前期询价**：录入维修范围、预算、分摊范围、施工单位选择方式等方案内容；可向施工单位征集参考报价并进行报价对比。此时不得显示“已选定施工单位”。
3. **公示并取得有效决定**：属于既有授权范围的，按小区已生效的授权规则办理；超出授权范围的，公示实施方案、征求意见并提交相关业主表决；紧急维修按法定紧急程序另行办理。页面只告诉经办人下一步做什么，不显示内部状态推导过程。
4. **确定施工单位**：实施方案通过后，按照其中载明的选择方式和报价选择规则确定施工单位。页面使用“确定施工单位”“已选定施工单位”“选择说明”“施工单位选择记录”，不使用“定商”“中选”“定商评审”等行业缩写。
5. **签订并归档施工合同**：确定施工单位后另行签订施工承包合同，在工程档案中登记合同范围、金额、工期、质保和原件。施工单位确定页面不得以“与合同”命名，避免让经办人误以为确认报价即已签约。
6. **施工、验收、审价和支取**：按合同组织施工和验收，依规定及合同办理审价和分次支取；各金额继续保持独立口径。

不同业务分支不得被一套页面强行串成相同步骤：

| 真实情形 | 下一步办理 |
| --- | --- |
| 物业服务合同、建设单位保修或责任人承担 | 按合同、保修或责任材料交由相应责任方履行，不进入业主选择施工单位和业主资金支取流程 |
| 共有维修且在既有有效授权范围内 | 按已生效授权规则办理，保留业委会审核、公告和工程资料，不重复发起一轮业主表决 |
| 共有维修且超出既有授权范围 | 完善并公示实施方案，取得相关业主表决通过后确定施工单位并签订施工合同 |
| 符合紧急维修条件 | 进入法定紧急维修程序，保存现场、公告、审价和事后公开材料，不伪装成普通表决流程 |

物业端统一使用下列可见用语；右侧内部概念只允许存在于代码、接口和审计数据中：

| 物业端显示 | 不得直接显示的内部或行业用语 |
| --- | --- |
| 本次维修由谁负责 | 责任路径、责任路由 |
| 拟使用的费用来源 | 资金承担路径、资金路由 |
| 下一步办理 | 当前授权状态、执行状态 |
| 提交实施方案，发起相关业主表决 | 冻结授权提案、生成授权快照 |
| 确认实施方案 | 锁定最终实施方案、锁定快照 |
| 确定施工单位 | 定商、最终定商 |
| 已选定施工单位 | 中选供应商、中选结果 |
| 施工单位选择记录 | 定商评审记录 |
| 表决通过的施工单位选择方案 | 已用印定商授权快照 |

“服务端”“后端”“快照”“哈希”“资金切片”等实现词不得出现在物业办理页面、按钮、状态标签和业务错误提示中。未到后续环节时显示中性状态，例如“等待实施方案通过”“完成相关业主表决后办理”，不得显示“受阻”并暗示当前经办人操作错误。

### 6.2 第 3 步“工程清单”的直接修正

第 3 步更名为“维修点位与维修对象”。报价/工程量清单放在供应商询价和定商阶段。

| 当前字段/行为 | 目标表达 | 处理 |
| --- | --- | --- |
| `编号 ITEM-1` | 内部稳定 ID + 业务可读名称 | 内部自动生成 ID，不展示为可编辑字段 |
| `位置` 自由文本 | 楼栋、单元/门号、参照房屋/公区、空间、朝向、构件、具体部位 | 提交结构化位置；公区不得伪造 `roomId` |
| `工作内容` | 问题现象、原因状态/依据、拟维修措施 | 拆分字段；不把报修摘要或照片要求自动当成施工内容 |
| `数量/单位` 必填 | 点位范围量可选；报价行数量/单位必填 | 建项不默认 `1 项`；报价支持真实专业单位和小数 |
| `含税估算单价` | 可选点位暂估金额 + 估算来源 | 从点位移除含税单价，不强制数量乘单价 |
| `预算合计` | 初步估算或送审/决定预算（含税） | 显示金额口径、来源和当前效力 |
| `关联已勘验报修事项` | 点位与报修/巡检/勘验来源关系 | 保留在点位级；可查看原始照片和勘验依据 |
| 一个卡片一个“工程项” | 单一决定范围 -> 多个维修点位 | 页面顶部展示决定/分摊范围，下方展示点位；报价行不重复报修关系 |

### 6.3 同范围事项关联

管理端可以在“已勘验事项”中多选，但后端必须先确认属于同一共有与决定范围。

- 同一范围：允许创建一个项目并生成多个点位，保留每条来源关系。
- 范围待确认：只允许保存内部筹备草稿，不得进入决定、定商或施工。
- 范围不同：后端拒绝聚合并要求分别建项，不提供“仍然合并”绕过选项。
- 多栋部分共用设施：明确告知当前版本不支持，不自动分拆或改成全体共用。

## 7. 代码修改范围

### 7.1 数据与后端

1. 将项目与共有/决定范围建成一对一关系，由数据库和领域不变性同时保护。
2. 新增或收敛决定范围快照：保存共有性质、权利边界、决定/授权依据、人数/面积基数、结果和证据。
3. 新增资金分摊切片和承担房屋快照，与直接受影响/验收人群分开。
4. 将 `RepairProjectItem` 收敛为维修点位，增加结构化位置、问题现象、原因状态/依据、拟定措施、暂估来源和来源记录关系。
5. 去除建项阶段“每个点位必须数量乘单价”的不变性；预算总额、报价明细和结算明细各自校验。
6. 将冻结待决定方案与决定生效后的定商/执行锁定拆成两个领域动作。
7. 复用报价头统一税率和灵活明细类型；结算明细关联中选报价行或有效变更行，税率/税额收敛到结算头。
8. 实施方案锁定点位级证据要求和验收策略；验收角色及通过条件必须有真实依据快照。
9. 服务端必须拒绝将其他决定范围的点位、账簿或费用承担房屋加入当前项目。
10. 本轮不新增跨决定范围聚合表、采购包覆盖表、跨项目合同表或聚合支取表。

后续迁移只能从新的 Flyway 版本开始，不修改已执行历史迁移。具体表名和版本号由实现时依当前仓库序列确认，不在方案中预造。

### 7.2 管理端

- `RepairProjectEditor.tsx`：展示单一决定/分摊范围，将工程清单改为维修点位，支持同范围来源关联、结构化位置、现象/措施拆分、可选暂估和证据要求。
- `repair-project.ts`：替换扁平建项契约，并提供责任认定提出/确认、授权提案冻结和最终实施锁定接口；不增加跨范围聚合契约。
- `RepairProjectResponsibilityOperation.tsx`：物业只能提出基于勘验、合同、保修或责任材料的责任认定；有治理权限的主体另行确认。共有维修的专项资金、公共收益或业主自筹路径必须显式选择，不能由楼栋范围推导。
- `RepairProjectOperationPanel.tsx`：需相关业主决定的共有维修先显示“冻结授权提案”，决定生效后才显示“锁定最终实施方案”；账簿未接入不阻断提案冻结，只在最终锁定时核验，且不再把资金切片为空误报为“楼栋未接入专项维修资金”。
- 报价与结算界面：保留专业明细和单据头税率，让用户能回溯到对应点位和原始勘验证据。
- 施工与验收界面：按锁定方案显示适用证据、角色和通过条件，不由前端固定六个阶段或两组人员。

已确认的直接责任路径（物业合同、建设单位保修、责任人/第三方）不得进入业主侧供应商定商或施工合同接口；现阶段只保留其责任认定与最终锁定，并明确阻断错误入口。责任方实际履行、验收和结案凭证的专用登记链尚未实现，不能伪装成业主侧合同流程。

### 7.3 存量数据

1. 存量单楼栋/单元项目只在现有决定依据足以证明一个权利人范围时生成决定范围快照；无法证明时标记待核验。
2. 旧 `RepairProjectItem` 只能映射为历史点位/估算行，不伪造原因、结构化位置或报价行关系。
3. 多楼栋批次材料不自动归并、不自动拆分、不自动迁移到新流程，只保持原始附件和历史只读状态。
4. 新流程启用后只写新模型，不设计长期双写。

## 8. 实施顺序与验收场景

### 8.1 实施顺序

1. 先用测试固定“一项目一决定范围”、同范围多点位、四类范围隔离和资金分摊勾稽。
2. 实现决定范围、资金切片、维修点位和来源关系的数据契约。
3. 修改建项/方案 API 与管理端第 3 步，关闭新项目的旧扁平写入。
4. 拆分待决定方案冻结与决定后定商，再调整报价、结算和税额口径。
5. 最后接入施工证据、动态验收策略和业主端披露。

实现时小步提交，每一步都保持领域、数据库、接口、管理端和至少一个可验证的端到端场景一致，不先造空表或前端假流程。

### 8.2 必须通过的场景

1. **16-403**：一个 16 号楼决定范围、对应资金分摊快照和窗户/外墙交界点位；403 室是直接受影响/效果确认对象，不是唯一决定或分摊人。
2. **37 号楼玻璃**：一个楼栋公区点位，不选择或伪造私房；纸面与微信人数冲突必须阻止自动确认决定结果。
3. **同楼栋多点位**：同一楼栋、同一共有权利和决定范围内的多个已勘验点位可建立一个项目，每个点位保留来源、位置、现象、措施、照片和结算回溯。
4. **不同范围阻断**：选择两个独立楼栋的事项时后端拒绝聚合，不生成跨范围草稿、采购或合同。
5. **全体共用**：有效资料证明设施服务整个物业管理区域时，形成一个全体共用决定范围和按幢/按户的资金切片，不称为“公共收益支付”。
6. **多栋部分共用不支持**：有证据显示设施只服务多个特定楼栋时，系统明确阻止当前新流程，不误判全小区，也不返回可绕过校验的前端选项。
7. **地库排水泵**：一个项目内 7 个位置组，每组可有设备、材料、人工和运输报价行；点位与报价行不是一对一。
8. **设备门竣工**：方案保留 13 樘门的点位/规格，施工保留材料进场和过程证据，验收汇总结论与附件明细都可追溯。
9. **金额与税额**：建项不伪造 `1 项 × 0 元`；报价/结算以单据头统一税率，明细和总额可勾稽，舍入尾差可追溯。
10. **原文冲突**：导入标题与正文冲突的阅处单时保留原件并生成异常，不自动形成维修点位或项目范围。
11. **直接责任阻断**：物业合同、保修或第三方责任经确认后，项目可以形成责任与预算快照，但业主侧定商和施工合同接口必须返回“不适用”，不能以“缺少中选供应商”误导操作者。
12. **账簿后置核验**：已确认的专项维修资金路径在楼栋账簿尚未接入时仍可冻结授权提案并进入业主决定；未形成可信账簿、余额与支取依据时，最终实施方案锁定必须拒绝。

对应自动化应覆盖领域不变性、项目/方案 API、Flyway/MyBatis 映射、决定范围与资金快照、报价/结算金额勾稽、管理端类型检查和生产构建。

## 9. 实现前置事实

以下事实仍无法从 `docs/sample` 得出，不得在代码中作为江湾默认规则：

1. 江湾国际公寓当前已备案的业主大会议事规则、维修资金管理规约和有效版本。
2. 是否存在授权物业/业委会在金额、范围和有效期内直接使用维修资金的有效决定。
3. 楼栋维修和全体共用项目的验收主体、各角色权限、人数/组合、用印和通过条件。
4. 不同金额、工程类型或资金路径的审价条件、合同变更容差、质保金和付款节点。
5. 各样本项目实际使用的资金来源、账簿/分户账、分摊表、完整合同、审价、发票、支取申请和银行流水。
6. 微信昵称与有表决权业主的可靠映射，以及纸质意见与微信接龙冲突时的核验规则。

这些事实会阻塞依赖本小区规则的自动决定、分摊、验收和付款功能。系统不能用模板、前端开关或账号角色名称代替真实授权。

823888.50 元委托与 132485.20 元点位汇总的关系、以及江湾多楼栋批次合同/支取规则，因本轮不考虑跨楼栋聚合，不再作为本轮实现前置；只在未来重新启动该能力时核实。

## 10. 外部案例的证据边界

先前核对的外部案例不再作为本轮实现依据，仅保留以下研究结论，避免将不同证据拼成一条未被证明的业务链：

1. 目前没有找到一个上海一手公开案例，能同时证明“普通业主报修 -> 多楼栋同类问题合成一个项目 -> 各楼栋分别决定和分摊 -> 从对应分户账支取”的完整链路。
2. [上海荣盛名邸](https://www.songjiang.gov.cn/xwzx/001001/20260106/205ceaea-a781-4997-817c-e8b43770d16a.html)是计划性老旧电梯更新，不是普通报修；只能证明一个更新项目可分批施工和验收。
3. [上海银马苑法院判决](https://www.hshfy.sh.cn/shfy/web/flws2pdf.jsp?pa=adGFoPaOoMjAyMaOpu6YwMTE1w%2FGz9TkwNDg0usUmd3N4aD0zJndzbGI9w%2FHKwsXQvvbK6QPdcssPdcssz)只能证明同一业委会和施工方对多处防水维修仍可存在七份合同；不能证明报修来源、项目聚合、决定或资金关系。
4. [盐城馨隆花苑](https://www.yandu.gov.cn/art/2025/9/12/art_23720_4365877.html)是招标前文件，能证明三栋楼形成一个招标项目，且公告写明资金来自专项维修基金；不能证明普通报修来源、最终合同或实际分户扣账。
5. [南京和天下家园](https://fcj.nanjing.gov.cn/ztzl/wygl/zcfg/202503/P020250324612746289391.pdf)是目前最接近的公开材料：一条项目级维修资金记录中，12、14、15 幢“各自按楼幢”分摊，并有首款和尾款记录；但它仍不能证明项目来自普通报修，也不能证明只有一份合同、一份支取申请或一笔银行划转。

因此，这些案例不能用来证明“多楼栋报修应当合并”，也不能用来决定江湾的合同、分摊或支取规则。

## 11. 与总体方案的关系

本文件继续保留 `docs/archive/business/报修与业主大会表决流程重构方案.md` 中“报修事项与维修工程分离”、“实施方案版本锁定”和“决定人群、分摊人群、验收人群不混同”的方向，并作出三项优先修正：

1. 一个项目在本轮只对应一个共有与决定范围，可包含多个同范围点位；不实现跨楼栋多范围聚合。
2. `RepairProjectItem` 不再同时代表维修点位、施工措施、报价行和结算行。
3. 部分共用/全体共用、资金来源、分摊账簿、直接受影响与验收参与范围分别建模，不再用一个“楼栋/公共”开关代替。
