# 维修表决历史切换

> 关联业务：把旧维修在线票、微信接龙和外部业主大会关联切换为统一的相关业主表决，同时保留历史事实。

## 切换原则

1. 新维修工程只能通过 `t_repair_project_voting` 关联统一表决包，不再创建旧楼栋征询或旧业主大会关联。
2. 旧规则登记、旧工单发起征询、旧项目发起楼栋征询和旧项目新建业主大会关联均停止新增。
3. 已经存在的旧流程继续只读展示；为避免中断已发生的真实业务，既有旧流程的收尾接口暂时保留，但不得创建新轮次。
4. 旧汇总结果、微信截图或旧线上票若缺少冻结名册、规则快照、逐事项选票和跨渠道唯一事实，不自动转换为新表决包。
5. 历史规则和表决事实不得因当前规则换版、方案修改或脚本执行被改写。

## 上线前盘点

在目标数据库执行：

```bash
psql "$DATABASE_URL" -f scripts/report-legacy-repair-voting.sql
```

第一份结果逐项目给出以下分类：

| 分类 | 处理 |
| --- | --- |
| `UNIFIED` | 已使用统一表决，无需迁移 |
| `LEGACY_READ_ONLY` | 旧流程已经形成结果，只读保留 |
| `LEGACY_REQUIRES_MANUAL_DECISION` | 旧流程尚未结束，业务负责人逐项目决定完成旧流程或终止后重建新方案 |
| `NO_VOTING_RECORD` | 尚未发起表决，可直接使用统一表决 |

第二、三份结果列出仍属于旧维修工单的楼栋征询和业主大会关联。它们不属于维修工程项目的新表决链路，不得仅凭汇总状态自动迁移。

## 人工处理记录

对 `LEGACY_REQUIRES_MANUAL_DECISION` 的每个项目记录：项目编号、当前环节、已有原件、参与范围、所依据规则、处理决定、决定人和时间。只有全部新表决前置事实都能从原始材料无损重建时，才允许另建新方案和新表决；否则完成或终止旧流程并保留原始记录。

## 发布后验证

1. 旧发起接口返回“请使用相关业主表决”，且不新增旧表记录。
2. 历史项目仍可打开旧详情和原始材料。
3. 新项目准备表决后只新增 `t_repair_project_voting`、统一表决包、冻结名册和对应渠道台账。
4. 再次执行盘点脚本，确认没有新的 `LEGACY_BUILDING`、`LEGACY_ASSEMBLY_LINK` 或旧工单表决记录。
