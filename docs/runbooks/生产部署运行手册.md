# 生产部署运行手册

> 关联业务：Pangu 后端生产发布、Flyway 迁移校验、服务恢复与上线验收。

## 发布触发与方式

- 用户提出“部署”“上线”或“发布”时，默认通过 SSH 发布 Pangu 后端和相关 Yaochi 静态管理端，不再重复询问部署方式。
- 只有当前 SSH 认证不可用或失败时，才向用户索取新的访问方式；不得从历史会话、终端记录或代码库中提取凭据，也不得把凭据写入版本库。
- 发布必须对应明确的已提交版本，先做只读健康与版本核验，再构建、备份、切换、重启和上线验收。

## 发布原则

1. 从待发布提交的干净归档构建 JAR，不把工作区未提交修改带入生产制品。
2. 发布前比较现网 JAR 与待发布 JAR 的同版本 Flyway 文件；历史迁移不得缺失或变化。
3. 查询生产 `flyway_schema_history`，确认数据库当前版本与待执行版本连续。
4. 先备份现网 JAR，再在 `/opt/pangu` 内原子替换并重启 `pangu.service`。
5. Flyway 校验失败时恢复已验证的 JAR 并排查迁移字节，不在生产执行 `flyway repair`。

## 数据库核验约束

- systemd 环境文件中的 `SPRING_DATASOURCE_URL` 可能为空，Spring 会回退到应用配置，但 `psql` 不会。命令行核验前必须从应用启动日志或受版本管理的配置确认实际主机、端口和数据库名。
- 只从环境文件读取数据库用户名和密码，查询输出不得包含环境变量值。
- SSH、远端 shell 和 SQL 三层引号容易互相剥离。每次执行一个简单查询；必须使用字符串常量时，优先采用 PostgreSQL dollar quoting，并先用只读查询验证命令形状。

## 上线验收

1. `systemctl is-active pangu` 返回 `active`。
2. 本次启动日志包含 Flyway 校验成功、目标迁移成功和 `Started PanguApplication`。
3. 现网 JAR 的 SHA-256 与本地构建制品一致。
4. 通过生产域名调用登录接口，HTTP 状态和业务 `code` 均为 `200`。
5. 只读查询新增表、约束和权限种子，确认迁移结果可用。
6. 删除服务器与本地的临时发布制品，保留带时间戳的回滚 JAR。

## 管理端白屏防护

1. Yaochi 发布前必须依次通过 `npm run type-check`、业务渲染回归和 `npm run build`；Vite 能完成打包不代表页面中不存在未导入组件或未定义变量。
2. 构建后至少使用真实角色登录，打开本次变更涉及的菜单、标签页、弹窗和分步表单；只验证首页或静态产物 HTTP 200 不能证明业务页面可用。
3. 浏览器验收必须检查控制台。权限预期内的 `403`、业务前置条件产生的 `409` 与 JavaScript `ReferenceError` 分开记录；任何未定义组件或变量都阻止发布。
4. 新增或修改 `.ts`、`.tsx` 文件时同步维护 TypeScript 类型，不得依靠生产访问后再发现缺失导入。

### 2026-07-20 管理端白屏错误复盘

- 社区设置页面先后因 `Save` 和 `Switch` 未导入产生 `ReferenceError`，但原有 `vite build` 均成功，说明单独以打包成功作为前端发布门禁不成立。
- 泛化原因：项目此前没有 `tsconfig.json` 和类型检查脚本，Vite 的转译打包不会完成完整的未定义标识符检查；同时验收没有在目标角色下逐页打开变更入口。
- 修正：Yaochi 已增加全项目 TypeScript 检查，并修复类型契约；维修草稿渲染回归明确验证草稿阶段不得提前出现施工单位选择。本地使用业委会主任和物业经理身份实际打开社区设置、维修项目创建、议题筹备和业主大会页面，未再出现 JavaScript 运行时错误。

## 2026-07-15 错误复盘

- 第一次数据库查询直接使用空的 `SPRING_DATASOURCE_URL`，导致 `psql` 回退到本地 socket 和错误数据库名。
- 第二次把多条带字符串常量的 SQL 嵌入 SSH 命令，外层 shell 剥离了单引号并造成 SQL 语法错误。
- 小程序网络排障时连续假定服务器已安装 `jq`、`systemctl show` 支持新版 `--value` 参数且微信配置位于环境文件，实际三项均不成立，导致只读诊断命令本身反复失败。
- 泛化原因：把应用运行时配置回退规则误当成命令行环境事实，同时在未验证引号边界时组合了过多解释层。
- 后续检查：先确认有效连接参数，再拆分只读查询；远程诊断先核对工具是否存在、命令版本、环境变量名称和应用实际激活 profile；任何复合远端命令都先验证变量是否非空，并避免在同一参数中混合 shell 与 SQL 引号。

## 微信小程序干净制品上传检查

1. 小程序必须从目标提交的干净归档构建，不能直接打包含用户未提交修改的工作区。
2. `project.config.json` 可能被 Git 忽略，`git archive` 不会包含它，Taro 也不保证在新的临时构建目录中生成该文件。
3. 上传前检查构建目录 `dist/project.config.json` 存在，`appid` 非空，且放在 `dist` 时 `miniprogramRoot` 为 `./`；然后把 `dist` 作为微信开发者工具 CLI 的 `--project`。
4. 项目配置只提供微信项目元数据，不得用复制原工作区 `dist` 的方式补齐，否则会把旧包或未提交代码混入本次制品。

### 2026-07-15 微信上传错误复盘

- 第一次直接上传干净归档生成的 `dist`，因缺少 `project.config.json`，开发者工具得到空 `appid` 并返回 `41002 appid missing`。
- 第二次改传临时源码根目录，但该目录同样没有被 Git 忽略的项目配置，开发者工具按源码根目录查找 `app.json` 并失败。
- 泛化原因：把本机未纳入版本控制的微信项目元数据误认为可由代码归档或 Taro 构建自动恢复，同时没有在上传前检查最终目录的项目身份和小程序根路径。
- 后续检查：先验证干净归档、生产构建和业务校验，再单独补齐经过确认的非敏感项目元数据；确认开发者工具显示真实 `appid` 后才允许上传。

### 2026-07-15 SSH 短连接认证抖动复盘

- 现象：同一组已验证凭据在连续创建多个短 SSH 会话时，两次返回 `Permission denied`；相同命令立即重试均成功，未发现凭据变更证据。
- 边界：未读取服务器 SSH 安全日志，不能把现象武断归因于限流、并发上限或密码错误。
- 泛化处理：多步骤生产发布先建立带 `ControlMaster` / `ControlPersist` 的短时复用连接，后续上传、切换和验收共用该连接；若认证失败，先确认既有会话及服务状态，再决定是否重试，避免在制品已切换后误判凭据失效并中断验收。
