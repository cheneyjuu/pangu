# 报修与维修工程真实流程差距及重构方案

更新日期：2026-07-15

状态：业务重构基线；楼栋/全小区验收权限、工程项目、多工程项、实施方案版本、分摊快照和两类独立治理编排切片已完成；施工结算付款、前端和迁移切片按第 8 节继续推进

## 1. 结论

对照 8 个真实附件、业委会人员口述、6 篇江湾国际公寓业委会公众号材料，以及当前 `pangu` 后端、`yaochi` 管理端和 `shennong-app` 业主端后，结论是：**江湾国际公寓把“楼栋/单元维修”和“全小区公共区域维修”视为两个独立业务流程，而不是同一条维修流程的两个资金参数。** 前者由相关楼栋业主形成决定，后者由业委会组织业主大会形成决定。两条流程只复用工程项目、供应商、施工记录、结算和付款能力，不共用决策、责任和验收流程。

最严重的五个问题是：

1. 当前虽然有楼栋决定和业主大会状态，但仍以一个 `RepairWorkOrder`、一套步骤条和一套验收逻辑承载两种业务，未真正形成两个流程边界。
2. 当前以单个工单作为合同、表决、施工和验收单元，无法表达一份委托、一个合同下包含 36 个维修点位的真实批量项目。
3. 当前开工后只有状态变化，没有施工前、材料进场、隐蔽工程、施工中、完工和验收过程证据，不能形成真实工程档案。
4. 当前共有部分维修验收强制至少一户“受影响房屋”，也没有区分“楼组长与受影响业主验收”和“业委会验收”两种不同的业主侧验收责任。
5. 当前付款申请只校验合同剩余金额和一个通用证据哈希，没有预付款 30%、审价前累计 90%、完工证明、审价金额、质保期满证明等维修资金支取约束。

本文件覆盖同一路径下 2026-07-10 版本，并纠正其中四个未经真实材料验证的假设：

1. 楼栋/单元维修的业主侧验收由楼组长和受影响业主共同承担，并按锁定的最低人数及通过规则形成结论；业委会负责审价和盖章，但不作楼栋维修的业主侧验收结论。
2. 全小区公共区域项目通过业主大会流程决定，项目完工后业委会是必需的业主侧验收主体。
3. 施工单位和物业项目负责人在两条流程中都负责完工提交、工程量核对和技术/合同验收，但不能替代对应的业主侧验收结论。
4. 微信接龙是江湾楼栋维修的实际征询方式；其法律效果仍须绑定本小区备案规则、有效送达和逐户身份核验，不能配置成全平台默认规则。

两条流程的责任边界如下：

| 责任 | 楼栋/单元维修 | 全小区公共区域维修 |
| --- | --- | --- |
| 适用范围 | 某一楼栋或单元的部分共有部位 | 道路、全小区管网、监控等全体业主共同使用的区域或设施 |
| 决定范围 | 相关楼栋/单元业主 | 全体业主，通过业主大会表决 |
| 业委会角色 | 审价、盖章并依有效方案履行组织/委托和资金监督责任；不作楼栋业主侧验收签署人 | 组织前期方案、召开业主大会、公示和计票，并作为业主侧验收主体 |
| 物业角色 | 踏勘、形成方案和报价、报审、组织合规签约及施工 | 配合方案和工程执行，履行合同、现场和工程管理责任 |
| 业主侧验收 | 楼组长 + 受影响业主 + 最低人数/通过规则 | 业委会 |

### 2.3 业委会人员口述确认的两个独立流程

#### 楼栋或单元维修

`业主报修 -> 物业踏勘 -> 物业形成方案和报价 -> 相关楼栋/单元微信接龙 -> 物业生成正式报审文件并附接龙截图 -> 业委会仅审价、盖章 -> 物业组织合规签约并组织施工 -> 楼组长与受影响业主按最低人数及通过规则验收`

该流程不召开全小区业主大会，不能使用无关楼栋或单元的维修资金。业委会不负责楼栋业主决定、不因审价盖章而取得施工单位选择权，也不承担楼栋维修的业主侧验收签署；但仍应依有效实施方案履行组织/委托和资金监督责任。楼组长和受影响业主是必需验收角色，系统还必须根据实施方案中锁定的最低有效验收人数和通过规则判定结果。

#### 全小区公共区域维修（业主大会流程）

`业委会/居委会/专业公司前期准备 -> 业委会会议决定召开业主大会及会议形式 -> 方案和附件公示 -> 处理反馈并锁定表决包 -> 盖章纸质选票逐户送达和回收（或可信线上投票） -> 开箱计票 -> 结果公告 -> 签约施工 -> 业委会验收`

该流程适用于小区道路、全小区管网、监控等全体业主共同使用、由全体范围维修资金承担的项目。业委会在该流程中负责组织前期方案、决定召开业主大会、组织公示送达和计票，并在完工后作为业主侧验收主体。线上投票只有在业主、房屋和表决权能够可信一一绑定，并与纸质票去重时才可启用。

### 2.4 楼栋/单元维修证据结论

1. `16-403次卧漏水维修意见` 和 `玻璃更换意见` 确认：费用只由对应楼栋业主承担，由楼组长组织微信接龙/书面表决，通过后委托物业实施。
2. 口述确认：物业将接龙结果和方案形成正式文件报业委会，业委会仅审价、盖章同意动用该楼栋资金，物业再组织合同签署和施工。“物业签约”按组织签约理解，不据此缩减法定合同主体。
3. 批量渗水项目的委托函记录了已确认的施工单位，但未展示其询价、推荐或比选过程；不能由该个案反推“业委会负责选择楼栋维修施工单位”。
4. 楼栋验收按“楼组长 + 受影响业主 + 最低人数通过规则”执行，业委会不作为楼栋维修验收人。现有材料未给出最低人数和通过条件的具体数值，必须在实施方案中必填并锁定，不设全平台默认值。

### 2.5 全小区公共区域维修证据结论

1. 公众号材料确认了“方案公示 -> 意见处理 -> 业委会决定召开业主大会 -> 正式通知和选票送达 -> 回收计票 -> 结果公告”的完整链路。
2. 一次业主大会可以包含多个表决事项，每个事项必须绑定自己的方案版本、选项和双维度计票结果。
3. 该流程中业委会是组织主体，负责前期组织、业主大会流程和完工后业主侧验收；不能把楼栋接龙、楼组长验收规则带入该流程。
4. 全小区项目的业委会验收已确认为三个必需条件：主任或副主任至少一人在线同意；经授权的业委会委员/印章保管人完成业委会公章用印并留存用印记录；物业项目负责人或第三方专业人员至少一方共同签署。

### 2.6 共用工程证据与合同边界

竣工验收表和过程验证记录证明，两条流程都需要施工单位完工提交、物业工程核验、材料进场、施工过程、实际工程量和付款凭证。这些是共用工程能力，不能用来合并两条业务流程。

已复核 `docs/sample` 目录全部 16 个文件，并逐页检查其中 8 份维修工程 PDF。该目录包含报价单、账目/收支公示、送达表和阅处单，没有施工承包合同正文或合同签署页。`docs/sample/江湾国际公寓设备门更换.pdf` 也是报价单，不是合同。

原附件中同样没有签署完整的施工承包合同。竣工验收表中的甲乙方现场负责人和“三方确认”是验收与付款凭证语义，不足以证明合同签约主体。目标系统按上海现行规则保留业主大会、物业服务企业和施工单位三方签约校验，不把口述的“物业签约”实现成排除业主大会的两方合同。

附件、口述和公众号材料仍不能代替本小区已备案议事规则、专项维修资金管理规约、既有授权决议、完整合同、银行支取材料、完工告示和最终资金流水。缺失事实不得写成全租户统一规则。

## 3. 法规与本地规则边界

### 3.1 全国法定表决底线

《中华人民共和国民法典》第二百七十八条将使用维修资金列为业主共同决定事项。参与表决人数和专有部分面积均应达到三分之二；使用维修资金应经参与表决人数和面积均过半数同意。当前 `RepairLocalDecision.passesThreshold` 对“明确参与票”的基础计算符合这一门槛，应保留。

参考：[《中华人民共和国民法典》第二百七十八条](https://gdca.miit.gov.cn/zwgk/zcwj/flfg/art/2020/art_573d6ef5018b46b6a4e1f31ca085a710.html)

### 3.2 上海维修资金使用要求

上海市现行规定明确：

1. 业主大会可以事先决定授权物业服务企业、业主委员会直接使用维修资金的金额；授权使用情况应定期公布，业委会核对工程任务单。
2. 实施方案应包含工程名称、实施范围、预算、分摊范围及面积、施工单位选择方式、施工管理、工程验收和实施时间。
3. 业主大会成立后，应依据表决通过的实施方案确定施工单位，并由业主大会、物业服务企业和施工单位签订施工承包合同。
4. 首次支取金额不超过合同总价 30%；需审价项目在审价完成前累计进度款不超过合同总价 90%；审价后总费用不超过审价金额；完工款和质保金分别需要对应证明。
5. 工程完成后应张贴完工告示，物业向业委会提交书面维修报告。
6. 2026 年上海市业主大会、业委会规范化建设意见进一步明确：超出既有授权的共有部分维修项目，业委会应根据业主大会表决通过的实施方案确定施工单位、签订施工合同并组织竣工验收。

参考：[沪房规范〔2024〕7号《关于规范本市住宅维修资金管理的若干规定》](https://www.shanghai.gov.cn/nw12344/20240830/da309a05b18f47eeafc774cc93953489.html)

参考：[沪房物业〔2026〕36号《关于完善本市住宅小区业主大会、业主委员会规范化建设的实施意见》](https://fgj.sh.gov.cn/wyglgw/20260423/b81adff0389b4a7c9ecb99bc0b3594dc.html)

因此，江湾楼栋流程可按口述将“楼组长 + 受影响业主”设为业主侧验收签署人，但不能把业委会从法定组织和监督链路删除。系统中应表达为：业委会依据已表决实施方案履行组织/委托和资金监督责任，施工单位提交完工，物业或第三方做工程核验，楼组长与受影响业主作出楼栋业主侧验收结论。

### 3.3 “默认同意”不是全局规则

上海市 2026 年业主大会议事规则示范文本允许小区在自己的议事规则中选择：有效送达后未表示意见，可以视为参与并按多数意见、弃权或其他约定处理，也可以视为未参与。示范文本还要求选择有效送达方式，并明确同一事项不得重复投票。

因此：

1. 附件中的“未回复默认同意/按多数意见处理”不能被系统全局硬编码为合法。
2. 系统必须保存本小区实际备案规则的版本、哈希、生效时间、送达规则和未表态处理规则。
3. 只有证明某户有效送达后，才能按该规则推导其参与及意见；推导票必须与业主主动投票分开记录。
4. 普通微信群接龙若不符合本小区规则，只能作为征询或佐证材料，不能直接生成具有法律效果的表决结果。

参考：[上海市《业主大会议事规则》示范文本（沪房物业〔2026〕85号附件）](https://fgj.sh.gov.cn/cmsres/56/56fe6530a6b645a8a7076236e7ac4773/a0aa77236c2471905c674c038df75e30.pdf)、[上海市房屋管理局关于参与表决和有效送达的答复](https://www.shanghai.gov.cn/gwk/search/content/10e2f108cb094b4089e40baa5f73de0b)

示范文本不是某个小区已经生效的议事规则。实施前仍须取得江湾国际公寓实际备案文本。

### 3.4 方案公示期与会议通知期必须分开

上海市 2026 年示范文本分别设置“方案草案公示不少于 7 日”和“会议召开 15 日前发布会议通知”两个阶段。江湾 2024 年公众号材料也实际形成了先公示方案、再发布正式会议/投票公告的两步流程。

系统不得把两者合并为一个日期字段：

1. `DRAFT_PUBLICITY` 用于展示方案、附件并收集修改意见；有实质修改时重新形成版本。
2. `MEETING_NOTICE` 用于通知已锁定的会议时间、形式、议题、表决材料、送达和计票安排。
3. 两个期限均从本小区当时有效的备案议事规则生成，公众号历史日期只能作为业务样本，不能替代当前规则。

## 4. 当前系统逐项差距

| 业务环节 | 当前实现 | 与真实流程的差距 | 处理结论 |
| --- | --- | --- | --- |
| 报修受理 | 支持业主报修、物业代建、定位、派单和勘验 | 前端流程基本可用 | 保留 |
| 两类流程边界 | 同一工单状态机中包含 `LOCAL_DECISION_PENDING` 和 `ASSEMBLY_DECISION_PENDING` | 只有状态分支，没有形成楼栋/单元维修与全小区公共区域维修两个独立流程、责任和验收权限 | 拆成两个流程聚合，只共用工程执行能力 |

| 楼栋决定 | 支持在线/微信、逐户选择、人数与面积双维度计票 | 江湾实际采用微信接龙、物业正式文件、业委会审价和盖章；当前缺备案规则版本和有效送达，且未明确江湾只开放微信方式 | 保留计票核心，为江湾配置专用楼栋流程 |
| 全小区公共区域决定 | 维修工单只有待业主大会决定的状态，现有业主大会模块与维修工程关联不完整 | 缺前期准备、业委会召开决定、方案公示、正式通知、盖章选票、逐户送达、回收、开箱计票、多事项和结果公告的完整关联 | 建立独立的全小区公共区域维修流程，复用正式业主大会模块 |
| 供应商与合同 | 支持邀价、报价、推荐、四类定商方式和三方合同 | 附件没有展示施工单位的询价/比选过程和完整合同，不能推导楼栋维修由业委会选择施工单位，也不能确定最终两方/三方签约主体 | 分别锁定两条流程的定商依据，保留三方签约能力和合规校验 |
| 审价与盖章 | 当前共有部分维修主线统一进入审价、主任/副主任确认、盖章 | 口述中江湾楼栋维修的业委会只在审价、盖章节点介入；全小区公共区域维修中业委会是业主大会组织者和验收主体，两者不能共用一套权限 | 两条流程分别定义业委会操作和权限，不提供通用“业委会下一步”接口 |
| 施工过程 | `startWork` 仅改变状态；附件类型只有勘验、报价、送审等 | 缺施工前、材料进场、隐蔽工程、施工中、完工、核验人和时间 | 新增施工记录聚合 |
| 验收范围 | `setAcceptanceScope` 强制至少一户受影响房屋 | 门禁、公共门、玻璃等工程可能没有可定义的受影响房屋，当前只能伪造房屋范围 | 改为验收模板和项目/工程项范围 |
| 验收主体 | 仅支持受影响业主、楼组长、物业、业委会代表，并使用同一完成规则 | 未强制楼栋维修的楼组长和受影响业主共同参与，也没有最低人数/通过规则；全小区项目也未独立为业委会验收 | 按流程锁定业主侧验收主体和通过规则，再补充施工提交与物业工程核验 |
| 结算 | 验收记录只有结论、意见、签名和证据哈希 | 缺实际工程量、单价、金额、税费、总额及相对合同/方案的偏差 | 新增结构化竣工结算 |
| 付款 | 仅校验本次金额不超过合同剩余金额 | 缺里程碑材料和维修资金法定金额上限 | 新增付款资格策略与材料清单 |
| 完工归档 | 支持完成、评价、归档状态 | 缺楼栋/小区完工告示、物业书面报告、质保责任期和期满证明 | 补齐完工披露与质保 |
| 管理端/小程序 | 已有前半段操作和基础验收入口 | 没有工程项目台账、过程记录、材料验收、实际结算；业主验收被错误设为通用主路径 | 按新聚合重做操作界面 |

当前实现中应直接保留的能力：租户隔离、报修定位、附件私有存储、供应商主体核验、报价版本、四类施工单位选择方式、明确参与票的双维度计票、不可变送审包、三方合同签署和审计事件。

## 5. 目标业务模型

### 5.1 分离报修事项和维修工程

1. `RepairCase`：一次问题报告，记录报修人、位置、现象、初始照片、共有/专有判断和责任判断。现有 `t_repair_work_order` 收敛为这一职责。
2. `RepairProject`：一次可独立形成实施方案、决策依据、合同、施工、验收、结算、付款和归档的工程项目。
3. `RepairProjectItem`：项目内的施工点位或工作清单项，关联零到多个报修事项，保存楼栋、房屋/公共区域、维修内容、预算和最终结算。

关系必须支持：

- 一个报修事项形成一个项目；
- 多个报修事项归并为一个项目；
- 一个复杂报修拆分到多个项目；
- 没有住户报修的计划性公共设备维修直接建立项目。

附件中的 36 个渗水点位应建成“一个维修项目 + 36 个工程项”，共享治理依据、委托、合同、验收和付款，不应复制 36 套完整审批链。

### 5.2 版本化实施方案

`RepairImplementationPlan` 至少包含：

1. 工程名称、问题原因、实施范围和工程项清单；
2. 预算总额、资金来源、分摊范围、分摊面积和分摊规则；
3. 施工单位选择方式及理由；
4. 施工管理要求、过程取证模板和安全要求；
5. 验收方式、必需参与角色和实际结算方式；
6. 计划开工、完工和质保期限；
7. 适用的治理路径、审价要求和付款节点；
8. 原始报价、现场照片和正式文件附件。

方案一旦进入表决、授权核验或签约，必须锁定版本。供应商、范围、分摊、预算或验收标准发生实质变化时生成新版本，并由后端策略判断是否需要重新决策、重新签约或补充协议。

### 5.3 两个独立维修流程

#### 5.3.1 楼栋/单元维修流程

适用条件同时满足：维修对象属于一个门牌幢、楼栋或单元范围的部分共有部位，费用只由该范围相关业主承担，不使用其他楼栋或全小区范围资金。

目标状态链：

`CASE_VERIFIED -> PLAN_READY -> BUILDING_SOLICITATION -> PROPERTY_APPROVAL_DOCUMENT -> COMMITTEE_PRICE_REVIEW -> COMMITTEE_SEALED -> CONTRACT_EFFECTIVE -> IN_PROGRESS -> BUILDING_OWNER_ACCEPTANCE -> COMPLETED`

业务守卫：

1. 物业踏勘后锁定楼栋/单元、工程项、预算、分摊范围和相关业主分母。
2. 江湾租户采用微信接龙；物业逐户核验后生成正式报审文件，将原始截图作为附件，不允许只录同意总数。
3. 业委会审价并对同意动用该范围维修资金的文件盖章；这两个动作必须分别留痕。不得由审价盖章推导业委会选择施工单位或作为楼栋业主侧验收签署人；业委会依实施方案履行的组织/委托和资金监督动作应另行留痕。
4. 物业负责形成方案和报价、报审，并在楼栋业主决定及审价盖章完成后组织签约和施工。施工单位的询价、推荐或比选方式必须来自实施方案和有效规约，不从委托函反推。
5. 施工单位提交完工和结算，物业核对合同、材料、过程证据和工程量；楼组长和受影响业主按实施方案锁定的验收名单、最低有效人数和通过条件作出业主侧验收结论，业委会账号不得代为操作。
6. 入户漏水项目的受影响业主必须确认使用效果。公共门、门禁等楼栋共有设施的受影响业主范围应根据设施服务范围在方案中确定，不强制绑定某一户“受影响房屋”。

#### 5.3.2 全小区公共区域维修流程

适用条件：维修对象属于全体业主共有和共同使用的小区公共区域或设施，费用由全体范围维修资金承担，或者依法、依规必须由业主大会决定。

目标状态链：

`INTERNAL_PREPARATION -> COMMITTEE_CONVENE_DECISION -> DRAFT_PUBLICITY -> FEEDBACK_REVIEW -> BALLOT_PACKAGE_LOCKED -> MEETING_NOTICE -> BALLOT_DELIVERY -> VOTING -> COUNTING -> RESOLUTION_PUBLISHED -> CONTRACT_EFFECTIVE -> IN_PROGRESS -> COMMITTEE_ACCEPTANCE -> COMPLETED`

业务守卫：

1. 前期准备可以记录业委会、居委会、专业公司形成方案和报价的过程，但前期讨论本身不产生业主大会决定。
2. 业委会会议必须明确召开时间、会议形式、表决事项和方案版本；一个会议可以包含多个表决事项，每项分别计票和形成结论。
3. 先公示方案和附件、处理反馈，再锁定正式表决包；正式通知、盖章选票、逐户送达、回收、计票和结果公告分别留痕。
4. 江湾当前实际以纸质投票为主。每张纸质票绑定房屋、业主、方案版本、送达记录、回收状态和票面扫描件。
5. 线上投票只有在可信完成账号、自然人、业主资格、房屋和投票权一一绑定时才开放，并与纸质票执行同一表决包和禁止重复投票规则。
6. 全小区项目完工后，业委会是必需的业主侧验收主体；必须同时满足“主任或副主任在线同意 + 业委会公章用印 + 物业或第三方专业人员共同签署”。施工单位仍需提交完工和结算，不得让审批人在没有过程材料、工程核验和结算明细的情况下只点“通过”。
7. 楼栋接龙、楼组长和受影响业主验收不是该流程的替代步骤，也不得用来简化全小区业主大会决定和业委会验收。

#### 5.3.3 共用工程执行能力

两条流程只共用以下工程能力，不共用治理状态机：项目/工程项、实施方案版本、供应商和报价、合同、施工过程记录、材料进场、实际结算、整改、付款申请、完工告示、质保和审计事件。

### 5.4 资金范围与治理依据分离

资金来源只回答“钱从哪里出”，共有范围回答“哪些业主承担和决定”，治理依据回答“为什么本项目可以执行”。三者不得压缩为一个 `fundSource` 字符串。

| 路径 | 适用条件 | 必需依据 |
| --- | --- | --- |
| `PROPERTY_CONTRACT_COVERED` | 属于物业服务合同包干责任 | 物业合同条款或责任认定，不支取业主维修资金 |
| `AUTHORIZED_DIRECT_USE` | 在业主大会既有授权额度、范围和有效期内 | 既有业主大会授权决议快照、任务单、额度占用、业委会核对和后续披露 |
| `BUILDING_REPAIR_DECISION` | 楼栋/单元部分共有维修，费用只由相关业主承担 | 楼栋接龙结果、物业正式报审文件、业委会审价和盖章 |
| `COMMUNITY_ASSEMBLY_DECISION` | 全体共有项目或全小区资金项目 | 业主大会方案公示、表决包、送达、选票、计票和正式决定 |
| `EMERGENCY_REPAIR` | 符合法定紧急情形 | 危险证据、应急处置、报告和事后完整档案，不得作为普通项目快捷通道 |
| `ADMINISTRATIVE_SUBSTITUTE` | 房屋行政管理部门依法组织代为维修 | 行政踏勘、公告、实施和支取通知等材料 |

系统必须校验维修对象与资金范围一致：一单元共有部位不能分摊给二单元，小区道路等全体共有设施不能只由某一楼栋承担。

既有授权必须由业主大会依法形成。业委会会议不能自行创设授权，也不能以一份委托函替代业主大会决议；委托函只证明在有效治理依据下由谁组织执行。

公共收益另按本小区公共收益专项规约确定决策和支付规则，不能直接套用维修资金或物业包干路径。

### 5.5 表决规则快照

每一轮正式决定保存不可变的 `RepairDecisionPolicySnapshot`：

1. 议事规则文件 ID、备案版本、哈希、生效时间；
2. 表决事项、方案版本、相关业主范围、人数和面积分母快照；
3. 允许的投票和送达方式；
4. 未表态是未参与、弃权、随多数或其他规则；
5. 公告、送达、投票开始和截止时间；
6. 主动票、代理票、纸质录入票和规则推导票的来源；
7. 计票结果、计票人/复核人和异议处理记录。

`RepairDecisionCountingPolicy` 只基于该快照计票。主动投票记录不可被规则推导结果覆盖；没有有效送达证据时必须保持 `NOT_VOTED`。

### 5.6 施工过程档案

新增 `RepairExecutionRecord`，按工程项和阶段记录：

1. `PRE_CONSTRUCTION`：施工位置、面积/长度、原状和安全措施；
2. `MATERIAL_ENTRY`：材料名称、品牌、型号、规格、数量、合格证明和施工单位资质；
3. `IN_PROCESS`：关键工序、隐蔽工程覆盖前后、质量控制和安全情况；
4. `COMPLETION`：完工全景、细节、清理情况；
5. `ACCEPTANCE_PROCESS`：验收过程和整改复验。

每条记录保存原始附件 ID、拍摄时间、上传人、核验人、核验时间和工程项。照片数量由锁定的施工取证模板决定；对本次真实表单，施工前、施工中、完工阶段默认各不少于 3 张，不把这一数量误写成所有租户不可调整的法律常量。

### 5.7 验收与实际结算

验收对象从“受影响房屋集合”改为“项目 + 工程项清单”，并把工程提交、工程核验和业主侧验收分开：

1. 施工单位负责人提交完工、整改结果、工程量和结算，并对施工责任签字。
2. 物业项目/现场负责人核对实施方案、合同、材料、施工过程、实际数量和费用。
3. 楼栋/单元维修必须由楼组长和受影响业主共同作出业主侧验收结论，业委会不得替代。系统必须同时校验楼组长已验收、受影响业主已参与，以及该项目锁定的最低有效验收人数和通过条件已满足。
4. 全小区公共区域项目必须由业委会作出业主侧验收结论。验收完成条件固定为：主任或副主任至少一人在线同意；经授权的业委会委员/印章保管人完成业委会公章用印及登记；物业项目负责人或第三方专业人员至少一方共同签署。三项缺一不得通过验收。
5. 入户漏水等直接影响专有部分使用效果的项目，受影响业主或其授权代表必须确认实际效果。
6. 公共门、门禁、玻璃等项目不要求伪造受影响房屋；属于楼栋范围时，由楼组长和按设施服务范围确定的受影响业主按规则验收；属于全小区范围时由业委会验收。

验收必须形成结构化竣工结算：原工作项、实际数量、单位、单价、金额、税率、税额、总额和差异原因。超范围、换材料、换供应商或增价达到重新决策条件时，不能通过验收备注绕过变更审批。验收通过的实际结算额是最终付款上限之一。

### 5.8 付款资格与完工披露

`RepairPaymentEligibilityPolicy` 根据资金来源、合同、是否审价和里程碑校验：

1. 预付款：维修资金首次支取金额不超过合同总价 30%，且治理依据、合同及需审价项目的审价委托材料齐全。
2. 进度款：依据合同节点和过程记录；需审价项目在审价完成前累计不超过合同总价 90%。
3. 完工款：必须有完工证明、验收结论、结构化实际结算和需审价项目的审价报告；总支付不超过合同、验收结算和审价金额中的适用上限。
4. 质保金：质保期满，并有物业和业委会共同出具的责任期满证明。

`conditionEvidenceHash` 不能继续作为唯一条件，改为结构化必需材料清单与附件引用。财务模块仍负责实际支取和银行流水，维修项目只生成满足条件的付款申请。

完工后新增 `RepairCompletionDisclosure`：保存张贴范围、开始/结束时间、告示文件、现场照片和物业书面维修报告，并进入最终归档包。

## 6. 后端重构设计

### 6.1 聚合与表结构

在下一个未占用的 Flyway 版本新增，不修改历史迁移：

1. `t_repair_project`：工程项目主表，保存 `workflow_type`、共有范围和资金账簿范围，不承载两条流程的全部细分状态。
2. `t_repair_project_item`：点位/工作清单项，与现有工单建立多对多关联表。
3. `t_repair_project_item_case`：工程项与现有工单的多对多关联。
4. `t_repair_plan_version`：锁定实施方案及版本哈希。
5. `t_repair_governance_basis`：治理路径、依据文件、授权额度占用和有效期快照。
6. `t_repair_building_process`：楼栋接龙、物业正式文件、业委会审价和盖章状态；复用现有楼栋决定轮次及逐户明细。
7. `t_repair_assembly_subject_link`：维修项目与现有业主大会会议、表决包和表决事项的关联，不在维修模块复制纸质投票能力。
8. `t_repair_decision_policy_snapshot`：议事规则、送达和未表态规则快照；现有决定轮次和逐户记录迁移到该快照下。
9. `t_repair_execution_record`、`t_repair_material_inspection`：施工阶段和材料进场结构化记录。
10. `t_repair_acceptance_policy_snapshot`：锁定验收主体、必需角色、受影响业主范围、最低有效人数、通过条件，以及“物业或第三方”这类至少一方满足的组合规则。
11. `t_repair_acceptance`、`t_repair_acceptance_item`、`t_repair_acceptance_party`：验收、实际结算、参与人及其 `ENGINEERING_SUBMITTER`、`ENGINEERING_VERIFIER`、`BUILDING_LEADER_ACCEPTOR`、`AFFECTED_OWNER_ACCEPTOR`、`COMMITTEE_EXECUTIVE_APPROVER`、`COMMITTEE_SEAL_OPERATOR`、`PROPERTY_TECHNICAL_COSIGNER`、`THIRD_PARTY_TECHNICAL_COSIGNER` 权限。
12. `t_repair_completion_disclosure`：完工告示、报告和质保信息。
13. 扩展 `t_repair_payment_request`：累计金额、规则结果、必需材料引用和拒绝原因。

现有 `t_repair_contract`、报价、供应商、送审包和附件元数据应迁移为项目级外键，不重新造平行能力。

附件类型补齐施工前、材料证明、材料照片、施工中、隐蔽工程、完工、验收单、结算单、完工告示和质保证明。结构化事实存表，原始文件存私有对象存储，两者通过附件 ID 关联，禁止只存手工填写哈希。

### 6.2 应用策略接口

拆分当前过大的 `RepairWorkOrderService`，至少形成以下领域/应用策略接口：

1. `RepairWorkflowRoutingPolicy`：依据资金、共有范围、维修对象、既有授权和紧急事实选择楼栋/单元维修、全小区公共区域维修或例外路径。
2. `BuildingRepairWorkflowService`：只编排楼栋接龙、物业报审、业委会审价盖章，以及楼组长与受影响业主按锁定规则验收。
3. `CommunityAssemblyRepairWorkflowService`：只编排维修项目与业主大会会议、表决事项、结果和业委会验收的关联。
4. `RepairDecisionCountingPolicy`：依据不可变规则快照生成双维度结果；与现有业主大会模块复用同一计票内核，但楼栋接龙和正式业主大会分别保存自己的送达及证据语义。
5. `RepairChangeReviewPolicy`：判断方案、合同和实际工程偏差是否需要重新接龙、重新召开业主大会或签订补充协议。
6. `RepairAcceptanceAuthorityPolicy`：先按流程类型确定楼组长 + 受影响业主或业委会的业主侧验收权。楼栋流程按快照校验最低有效人数和通过条件；全小区流程固定校验主任/副主任在线同意、业委会用印，以及物业/第三方至少一方共同签署。
7. `RepairPaymentEligibilityPolicy`：校验每个付款里程碑的金额和材料。

策略只接收后端可信事实和不可变快照，返回“允许/拒绝 + 原因 + 所需动作”，控制器不拼业务规则，前端不自行决定流程分支。

### 6.3 状态归属

1. `RepairCase` 只保留报修、核实、责任判断、已关联项目和关闭等事项状态。
2. `RepairProject` 只承担两条流程共用的方案、采购、签约、施工、结算、付款、质保和归档主状态。
3. `BuildingRepairProcess` 独立保存楼栋接龙、物业报审、业委会审价盖章，以及楼组长/受影响业主验收状态。
4. `OwnersAssemblySession` 继续独立保存业主大会准备、公示、通知、送达、投票、计票和结果状态，维修项目通过表决事项关联。
5. 计票轮次、合同签署、施工记录、验收和付款各自保留子状态，不把两条流程重新合并成一个超长枚举。
6. 项目状态只由对应流程和子聚合结果推进，禁止通用“下一步”接口跨过必需证据。

## 7. 前端重构

### 7.1 `yaochi`

1. 从“单工单步骤条”改为“报修事项 + 工程项目台账”，并将项目入口明确分为“楼栋/单元维修”和“全小区公共区域维修”。
2. 增加 36 项这类批量工程的表格录入、导入核对和逐项照片/金额查看，但最终治理、合同和付款在项目级完成。
3. 实施方案使用结构化表单，不再用预算和备注代替。
4. 楼栋维修页按固定顺序呈现接龙、物业正式文件、业委会审价、盖章、施工，以及楼组长/受影响业主验收进度和通过规则；业委会界面不出现楼栋验收操作。
5. 全小区公共区域维修页呈现业委会内部准备、召开业主大会决定、方案公示、反馈、锁定表决包、纸质票盖章、逐户送达、回收、计票、结果公告和业委会验收；验收界面分别呈现主任/副主任在线审批、业委会用印、物业/第三方专业签署三个不可跳过的状态；一个会议支持多个事项。
6. 表决页显示议事规则版本、送达状态、主动票与规则推导票，不允许物业手工把未回复改为同意。
7. 施工页按阶段和工程项采集证据；材料进场必须录品牌、型号、规格、数量和证明。
8. 验收页先显示谁拥有业主侧验收权，再录实际工作量、费用、税额和偏差；公共设施项目不再要求受影响房屋。
9. 付款页直接显示每个法定/规约条件的通过状态和缺失材料，不能仅上传一个“条件文件”。

### 7.2 `shennong-app` 与供应商工作台

1. 业主端查看锁定实施方案、费用分摊、治理依据、正式表决和完工披露。
2. 江湾楼栋维修在 C 端展示接龙结果和验收任务，不在 C 端另开一轮与微信接龙重复的在线投票。
3. 业主大会默认支持纸质送达、回收和结果查询；线上投票在身份、房屋、表决权和电子证据能力通过专项验收后再启用。
4. 楼栋验收任务发给楼组长和实施方案锁定的受影响业主，界面实时显示最低有效人数及通过条件是否满足；全小区项目分别向主任/副主任发送在线审批任务，向授权用印人发送盖章任务，并向物业项目负责人或已指定的第三方专业人员发送共同签署任务。
5. 受影响业主在入户维修中确认实际使用效果；公共设施项目按楼栋或全小区验收权限处理。
6. 供应商工作台补充材料进场、施工过程、完工提交、整改复验和施工单位验收签署。

## 8. 实施顺序

### 8.1 上线前必须固化的租户业务资料

以下材料不再阻塞共用领域模型和第一个垂直切片的编码，但必须在租户上线前形成可校验、可版本化的规则快照，不得以平台默认值代替：

1. 江湾国际公寓当前备案的《业主大会议事规则》及历次有效版本。
2. 专项维修资金管理规约、公共收益专项规约和物业服务合同的维修责任边界。
3. 是否存在授权物业/业委会直接使用维修资金的业主大会决议、授权额度、范围和有效期。
4. 开户银行的支取材料清单，以及本小区/区级规则中触发工程审价的条件。
5. 楼栋验收已确认由楼组长和受影响业主共同参与；受影响业主的范围、最低有效验收人数，以及同意人数/比例或一票否决等通过条件，由每个项目的实施方案必填并在征询前锁定，不设全局数值。
6. 楼栋维修实际施工合同原件，用于核对历史项目签署习惯；新流程不以此为前置，直接按上海现行三方签约规则校验。
7. 公众号中纸质选票“送达到业主”的具体送达证明、无法签收时投入门口或信报箱的规则及备案依据。
8. 8 个附件对应项目实际使用的资金范围、最终结算和付款材料。

### 8.2 垂直切片

1. **切片一：楼栋漏水完整流程。** 完成物业踏勘、方案报价、微信接龙、物业正式文件、业委会审价盖章、合同、施工过程、楼组长与受影响业主验收、最低人数/通过规则、结算和付款。
2. **切片二：全小区公共区域维修。** 完成内部准备、业委会召开业主大会决定、方案公示、反馈、表决包、多事项、盖章选票、逐户送达、回收、开箱计票、结果公告、施工和业委会验收。
3. **切片三：楼栋公共设施。** 完成公共门/门禁项目，根据设施服务范围确定受影响业主，由楼组长与该范围业主按锁定规则验收，不要求业委会或伪造特定受影响房屋。
4. **切片四：批量渗水工程。** 支持一个项目 36 个工程项、逐项预算/结算/照片和一个合同、验收、付款归档包。
5. **切片五：授权直用和例外路径。** 接入既有授权额度占用、半年披露、紧急维修和行政代为维修，分别保留独立依据。
6. **切片六：可信线上业主大会投票。** 仅在身份、房屋、投票权、电子签名和纸电去重全部可验证后实施，不作为纸质流程上线的前置条件。
7. **切片七：迁移与切换。** 为存量未结束的共有部分维修建立一对一项目，并明确标注“楼栋/单元维修”或“全小区公共区域维修”；确需归并的项目由业务人员审核后合并。新流程启用后关闭旧项目级写接口，保留历史只读，不长期双写。

每个切片分别完成 `pangu` 领域/应用/基础设施/接口、`yaochi`、`shennong-app` 或供应商工作台、数据库迁移和自动化测试后提交，不先造空表或只做前端假流程。

## 9. 验收场景

重构完成至少通过以下自动化和人工联调场景：

1. 楼栋漏水项目完成接龙、物业正式文件、业委会审价盖章后施工；楼组长缺席、受影响业主未参与或最低有效人数/通过条件未满足时均不得通过验收，业委会账号不能代为提交。
2. 材料进场缺品牌、规格、数量或合格证明时，项目不能提交完工验收。
3. 一个漏水维修由相关楼栋业主分摊，系统按人数和面积双维度计算，结果能追溯到方案、规则版本和每户送达/选择。
4. 未配置有效议事规则或缺某户有效送达证据时，该户必须保持 `NOT_VOTED`，不能因截图或物业操作变成同意。
5. 本小区规则明确“有效送达后未表态随多数”时，系统分别展示主动票与推导票，并能复算最终结果。
6. 一个业主大会会议包含四个事项，每项共享会议和送达范围，但分别保存方案版本、选项、双维度计票和决定。
7. 全小区项目施工完成后，主任/副主任在线同意、授权人完成业委会用印、物业或第三方专业人员共同签署三项缺一时，不能形成验收结论或申请完工款；楼栋业主或楼组长不能替代业委会验收。
8. 楼栋公共门项目不绑定某一户受影响房屋，但必须锁定楼组长、设施服务范围内的受影响业主和通过规则，验收后的实际数量与金额可形成付款依据。
9. 一个项目包含 36 个维修点位，能共用一份委托、一份合同和一次付款，同时保留每项实际金额与照片。
10. 一单元维修不得选择二单元或全小区资金账簿；小区道路不得只选择一个楼栋分摊。
11. 在有效业主大会授权范围和剩余额度内的小额任务无需重复逐案表决；超额、超范围或授权过期时自动进入对应正式流程。
12. 不需要审价的项目不生成伪审价节点；需要审价的项目在报告完成前累计进度款不得超过合同总价 90%。
13. 维修资金首次支取超过合同总价 30%、完工款缺验收/结算、质保金缺期满证明时，后端拒绝申请并给出明确缺项。
14. 实际结算增价、扩大范围或更换材料达到重新决策条件时，系统根据流程类型要求重新楼栋接龙或重新进入业主大会。
15. 完工告示和物业书面报告未归档时，项目不能进入最终归档状态。
16. 存量历史项目能只读查看，租户隔离、房屋权限、附件私有访问和审计记录不回退。

## 10. 明确不采用的做法

1. 不在现有工单上继续堆叠几十个工程字段和状态。
2. 不为 36 个点位复制 36 套表决、合同和付款流程。
3. 不把楼栋/单元维修和全小区公共区域维修合并成一套可配置步骤条。
4. 不让业委会替楼组长和受影响业主验收楼栋项目，也不让楼组长或楼栋业主替业委会验收全小区项目。
5. 不把微信接龙、未回复默认同意或江湾业委会审价链路写成全平台规则。
6. 不用附件哈希、备注或前端勾选代替实施方案、送达、施工、验收和付款的结构化事实。
7. 不要求公共门、门禁、玻璃等项目伪造受影响房屋。
8. 不允许业委会会议或委托函代替应由业主大会/相关业主形成的决定，也不重复表决已在有效既有授权内的任务。
9. 不修改历史 Flyway 文件，不长期保留新旧项目流程双写。
