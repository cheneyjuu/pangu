## 1. 目标 & 边界回顾

M1 的 `PartyRatioWaiver` 把「业委会主任 + 街道办」双签范式跑通，但只覆盖**党员比例放宽**这一个 entity。本期把双签机制提取到一张通用表 `t_governance_lock` + 一套 `GovernanceLockApplicationService`，让后续 M2-3（财务公示）、M2-4（角色管理）等阶段直接挂入，无需再开 schema。

**两条已锁定边界**：

1. **M1 schema 完全冻结**：`t_party_ratio_waiver` / `WaiverApplicationService` 一字不动；`PartyRatioWaiver` 与 `t_governance_lock` 本期 disjoint，下一阶段再评估是否合并双签来源。
2. **M2 路线图 §2.3 那句"streetReview 通过时额外写一条 t_governance_lock"本期不实现**——避免对 M1 已合并代码的回归面。

**本期承载 entityType**：`FINANCE_DISCLOSURE` / `ELECTION_DISCLOSURE` / `FUND_LEDGER_PUBLISH` 三种，CHECK 约束兜底；新增 entityType 走 V2.x+1 schema 增量，不动 V2.5。

## 2. 落地清单

### 2.1 V2.5 schema（`pangu-bootstrap/src/main/resources/db/migration/V2.5__governance_lock.sql`）

- **`t_governance_lock`** 18 列：双签人/时间/签名各 1 组、`status` SMALLINT（1/2/3）、`lock_payload_hash` VARCHAR(64)、`version` 乐观锁、`locked_at` / `unlock_at` 业务时间戳、`create_time` / `update_time` DB 时间戳；
- **CHECK 约束**：`chk_lock_status` (1/2/3) / `chk_lock_entity_type` (3 种枚举) / `chk_unlock_signers_diff`（双签不可同人）/ `chk_lock_payload_hash_len`（必须 64 字符）；
- **唯一索引** `uidx_lock_entity (tenant_id, entity_type, entity_id)` —— 三层并发防御兜底；
- **`trg_governance_lock_atomicity`**（trigger 8）BEFORE INSERT/UPDATE：committee/street 字段成对、`unlock_at` 当且仅当双签齐备、`status` 与字段一致、`status` 不可逆；
- **新增 2 个 permission**：`lock:unlock:committee` (group=LOCK, B 端, redline=1) / `lock:unlock:street` (group=LOCK, G 端, redline=1)；
- **role-permission 挂载**：`COMMITTEE_DIRECTOR` (id=5) → committee；`GOV_SUPER_ADMIN` (id=1) / `COMMUNITY_ADMIN` (id=2) → street。

### 2.2 Domain 层（`pangu-domain/.../model/lock/`）

- **`LockEntityType`** —— 仅字符串序列化（与 schema CHECK 对齐），不带 dbValue。
- **`GovernanceLockStatus`** —— 三态 LOCKED(1) / COMMITTEE_SIGNED(2) / FULLY_UNLOCKED(3)，`dbValue` + `fromDbValue` + `isTerminal()`。
- **`GovernanceLock`** 聚合根 —— 静态工厂 `lock(...)` + `signByCommittee(...)` + `signByStreet(...)` + `transitionTo(...)` + `rehydrate(...)`；状态机 `EnumMap<Status, EnumSet<Status>>`；终签同步置 `unlockAt`；同人双签在聚合根侧抛 `IllegalStateException`。
- **`GovernanceLockRepository`** port —— `findById` / `findByIdForUpdate` / `findByEntityForUpdate` / `insert` / `update`，内部静态 `DuplicateLockException` / `OptimisticLockException`。

### 2.3 Application 层（`pangu-application/.../lock/`）

- **`GovernanceLockApplicationService`**：
  - `lock(LockCommand)`：Redis 红线锁 `lock:gov:tenant:{T}:type:{X}:entity:{E}` (TTL 30s) → `TransactionTemplate` → `findByEntityForUpdate` → `GovernanceLock.lock(...)` → `repo.insert(...)`，唯一索引兜底翻成 `LOCK_ALREADY_EXISTS`；
  - `committeeSign(CommitteeUnlockCommand)` / `streetSign(StreetUnlockCommand)`：`@Transactional` + `findByIdForUpdate` + 聚合根状态机推进 + 乐观锁更新；冲突翻成 `LOCK_CONCURRENT_MODIFICATION`；
  - `verifyLocked(tenantId, type, entityId)`：未锁定 / 已 FULLY_UNLOCKED 抛 `LOCK_NOT_HELD`，供 M2-3 disclosure publish 修改前调用；
  - 聚合根 `IllegalStateException` 由 `mapStateException` 翻成 `LOCK_SIGNER_CONFLICT` / `LOCK_INVALID_TRANSITION`。
- **`GovernanceLockApplicationException`** + `Reason` 枚举 6 种：`LOCK_NOT_FOUND` / `LOCK_ALREADY_EXISTS` / `LOCK_CONCURRENT_MODIFICATION` / `LOCK_INVALID_TRANSITION` / `LOCK_SIGNER_CONFLICT` / `LOCK_NOT_HELD`。
- **`command/{LockCommand,CommitteeUnlockCommand,StreetUnlockCommand}`** records —— 由 controller 从 `SecurityUtils` 注入用户 / 租户。

### 2.4 Infrastructure 层（`pangu-infrastructure/...`）

- **`GovernanceLockRow`** Lombok POJO，时间统一 `Instant`、status `Integer`；
- **`GovernanceLockMapper` + XML**：`SELECT ... FOR UPDATE` 双路径 / `UPDATE ... SET version = version + 1 WHERE version = #{version}` 乐观锁；
- **`GovernanceLockRepositoryImpl`**：`DuplicateKeyException → DuplicateLockException`、`update` 影响 0 行 → `OptimisticLockException`、`toRow / toAggregate` 双向翻译。

### 2.5 Web 层（`pangu-interfaces/...`）

- **`LockErrorCode`** 实现 `ErrorCode`：6 个错误码与 `Reason` 一一对应；
- **`GovernanceLockExceptionTranslator`**：`Reason → LockErrorCode` 全映射；
- **`GovernanceLockController`** 三个 endpoint：
  - `POST /api/v1/locks` `@PreAuthorize("hasAuthority('admin:role:manage')")` —— 本期由 admin 触发回归测试，M2-3 disclosure publish 落地后再细化；
  - `POST /api/v1/locks/{lockId}/committee-sign` `@PreAuthorize("hasAuthority('lock:unlock:committee')")`
  - `POST /api/v1/locks/{lockId}/street-sign` `@PreAuthorize("hasAuthority('lock:unlock:street')")`
- **DTO**：`LockRequest` (entityType / entityId / lockPayloadHash 64-hex Pattern) / `UnlockRequest` (signature ≤ 128) / `LockResponse` (16 字段)；
- **`GlobalExceptionHandler`** 增 `@ExceptionHandler(GovernanceLockApplicationException.class)`，途径 translator 映射到 HTTP 响应。

## 5. 后续风险与衔接

1. **M2-3 disclosure publish 内部委派**：财务公示发布前调用 `lockApplicationService.lock(...)`，发布后调用 `verifyLocked(...)` 校验后再做 UPDATE。当前 `POST /api/v1/locks` 挂的 `admin:role:manage` 仅作为 M2-1 回归通路，M2-3 落地后会下沉为 disclosure 内部委派。
2. **PartyRatioWaiver 与 t_governance_lock 合并**：当前两表 disjoint，下一阶段（M2-3 落地后）评估是否合并双签来源。
3. **trigger 8 与聚合根重叠校验**：聚合根状态机 + trigger 8 + CHECK 约束三层校验；任一缺失都会被另两层兜底，但**不能省**——trigger 是抗 raw SQL / DBA 直改，CHECK 是抗 trigger 维护性损坏，聚合根是抗未走 service 的入口。
