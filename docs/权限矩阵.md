**不需要，也绝对不能对所有角色开放。** “数据范围配置”在权限系统中属于最高危的 **ABAC（基于属性的访问控制）核心底层策略**。如果把这个配置权对所有角色开放，或者让普通角色可以随意修改自己的数据范围，就会直接导致多租户隔离崩塌，产生严重的越权漏洞和居民隐私泄露事件。

根据我们在《产品需求文档（PRD）》和《核心安全策略》中锁死的法理红线，数据范围配置的“配置权（谁能改）”**与**“应用权（谁受限）”必须严格遵循以下非对称控制链：

---

### 一、 谁拥有数据范围的“配置权（管理和修改）”？

在页面交互和后端微服务层，右侧的“数据范围配置”单选组（全小区、指定楼栋、仅本组织）的**写入与变更权限，必须强行实施非对称隔离**：

1. **G端高级管理员（街道办/居委会专管员）**：**拥有全量角色的配置权**。只有他们可以决定给新当选的业委会主任分拨“全小区”范围，或者给新入驻的物业公司绑定“仅本组织”范围。
2. **B端自治主官（业委会主任）**：**拥有受限的下级配置权**。业委会主任仅能为其下属的“业委会普通委员/秘书”以及“物业经理”配置数据范围（例如分派某委员去管1栋的网格责任田）。
3. **普通角色（物业员工、业委会普通委员、网格员本身）**：**完全无权查看或修改此配置**。他们登录角色管理页面时，此区域必须强制置灰（Disabled）或直接隐藏，防止“自己给自己扩权”的黑客行为。

---

### 二、 核心角色的数据范围“应用分配矩阵”（动态锁定）

不同角色在系统初始化或委任时，其“数据范围”在底层往往是**天然绑定或具有硬性逻辑限制的**。不能为了追求“完全动态”而允许管理员胡乱勾选，必须遵守以下业务防呆矩阵：

| 角色组织层级 | 核心角色 | 默认数据范围 | 为什么必须这样？（系统硬约束逻辑） |
| --- | --- | --- | --- |
| **G 端 (政务监管层)** | 街道办管理员<br>

<br>社区党组织书记<br>

<br>居委会专管员 | **全小区**<br>

<br>`ALL_COMMUNITY` | **裁判员全局视野**：必须通管全局。如果将其限制在某个楼栋，将导致 G 端无法审批全小区大额维修资金的党建前置研究案，系统流转会直接死锁。 |
| **B 端 (业主自治层)** | 业委会主任 | **全小区**<br>

<br>`ALL_COMMUNITY` | **法定代表人职责**：作为自治法人代表，其发起的换届、表决、公告必须覆盖全体业主，去污染计算总分母必须读取全量空间台账。 |
| **B 端 (业主自治层)** | 楼栋代表 / 志愿者<br>

<br>网格员 | **指定楼栋**<br>

<br>`OWNER_GROUP` | **网格化责任田**：实施“精细化物理隔离”。限制其只能在勾选的 1 幢、2 幢内核销产权证 OCR 隐私，防止跨楼栋误操作和窥探邻居隐私。 |
| **B 端 (物业服务层)** | 物业经理 / 物业员工 | **仅本组织**<br>

<br>`ORG_ONLY` | **商业与利益隔离**：物业属于受托企业。必须将其死锁在本组织 `org_id` 内部，**绝对禁止**其跨界看到业委会自荐候选人名册、G端政审意见细节，违者直接触犯合规红线。 |

---

### 三、 后端网关层的硬编码防线（AOP 切面安全校验）

在联调开发阶段，后端 `auth-service` 决不能盲目信任前端传来的配置载荷（Payload）。当有用户调用 `POST /api/v1/roles/policy/update` 尝试保存数据范围时，后端拦截器必须进行以下**法理红线对碰**：

```go
// 后端动态数据范围变更校验切面
func ValidateDataScopePolicy(ctx *gin.Context, reqDataScope string, targetRole string) {
    operatorRole := ctx.GetString("RoleCode") // 当前操作人的角色

    // 防线 1：B端用户绝对禁止调配G端角色的数据范围
    if (targetRole == "GOV_SUPER_ADMIN" || targetRole == "PARTY_LEADER") && operatorRole != "GOV_SUPER_ADMIN" {
        ctx.JSON(403, gin.H{"code": 403, "msg": "❌ 越权拦截：非政务高级专管员，无权篡改政府监管端与党组织的数据作用域！"})
        ctx.Abort()
        return
    }

    // 防线 2：法理逻辑死锁防御（防止管理员手误将业委会主任改成“仅本组织”）
    if targetRole == "COMMITTEE_DIRECTOR" && reqDataScope != "ALL_COMMUNITY" {
        ctx.JSON(400, gin.H{"code": 400, "msg": "❌ 架构阻断：业委会主任角色依法必须持有[全小区]范围，禁止降级为局部范围，否则会导致全局表决引擎因分母缺失而死锁。"})
        ctx.Abort()
        return
    }
}

```

### 总结

数据范围配置**绝不能对所有角色开放**。它是平台实施“G端全控、B端限权、物业隔离、代表网格化”的最高阶策略工具。在日常运维中，它只能由 G 端专管员或业委会主任（仅限下属）在后台动态分拨，而底层核心写接口则通过后端 AOP 切面死锁红线，从而确保多租户 SaaS 平台在超大型社区运行时的绝对数据安全与等保三级合规。


现在能给角色指派“指定楼栋 / `OWNER_GROUP`”权限的登录用户只有两类：

1. **G 端高级管理员**
   - `GOV_SUPER_ADMIN`
   - `COMMUNITY_ADMIN`
   - `STREET_ADMIN`
   - `PARTY_SECRETARY`

   他们可以给允许使用 `OWNER_GROUP` 的角色配置楼栋范围，比如 `OWNER_REPRESENTATIVE`、`GRID_OPERATOR`、`COMMITTEE_MEMBER` 等。

2. **业委会主任**
   - `COMMITTEE_DIRECTOR`

   但只能配置下属/受管角色，当前允许范围是：
   - `COMMITTEE_MEMBER`
   - `COMMITTEE_SECRETARY`
   - `OWNER_REPRESENTATIVE`
   - `GRID_OPERATOR`
   - `PROPERTY_MANAGER`
   - `PROPERTY_STAFF`

注意：虽然 `PROPERTY_MANAGER / PROPERTY_STAFF` 在“可被业委会主任管理”的名单里，但后端硬规则要求物业角色只能是 `ORG_ONLY`，所以实际上**不能给物业角色指派楼栋权限**。可实际指派 `OWNER_GROUP` 的典型目标角色是 `OWNER_REPRESENTATIVE`、`GRID_OPERATOR`、`COMMITTEE_MEMBER`。
